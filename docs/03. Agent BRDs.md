## 1. Agent ecosystem overview

The Operations Intelligence Agent Ecosystem is a modular set of specialized AI capabilities operating on a shared governed data foundation within the Operations Tower platform. Each agent is a feature area within the unified platform — not a separate freestanding bot.


| #   | Agent                        | Phase | MVP status                                                                           |
| --- | ---------------------------- | ----- | ------------------------------------------------------------------------------------ |
| 1   | Delivery Performance Agent   | 1     | **In scope**                                                                         |
| 2   | Quality Intelligence Agent   | 1     | **In scope**                                                                         |
| 3   | Workforce & Capability Agent | 2     | Out of scope (limited workforce-rebalancing recommendations ship in MVP via Agent 1) |
| 4   | Project Governance Agent     | 2     | Out of scope                                                                         |
| 5   | Client Interaction Agent     | 1     | **In scope**                                                                         |
| 6   | Operational Knowledge Agent  | 2     | Out of scope                                                                         |


**Phase‑1 trio:** Delivery Performance + Quality Intelligence + Client Interaction.

**Platform architecture (shared by all agents):**

1. **Layer 1 — Unified data foundation:** Normalise delivery dashboards, quality reports, workforce data, and governance artifacts into a single governed store with KPI semantic layer, data lineage, and auditability.
2. **Layer 2 — Intelligence and analytics layer:** Predictive analytics, trend/anomaly detection, root-cause analysis, forecasting, risk scoring, pattern recognition.
3. **Layer 3 — Conversational AI interaction layer:** Natural-language Q&A, AI-generated summaries, automated reporting, action recommendations, role-based responses, evidence citation.

**Shared design principles (all agents):**

1. **Human-centric AI** — augments operational teams; does not replace scientific reviewers, SMEs, or annotation specialists.
2. **Single source of truth** — all intelligence grounded in governed, validated data.
3. **Evidence-backed AI** — every insight links to citable source data; no hallucinated project status.
4. **Role-based intelligence** — tailored insights per role and permissions.
5. **Operational explainability** — what happened, why, impact, recommended action, confidence level.
6. **Security and governance first** — RBAC, permission-aware AI, audit logging, human-in-the-loop validation, governance approval workflows, data lineage.

**Shared platform dependencies (all Phase‑1 agents):**


| ID    | Requirement                                                                                                            | Priority |
| ----- | ---------------------------------------------------------------------------------------------------------------------- | -------- |
| PF-01 | Users authenticate via Supabase Auth (JWT access + refresh tokens).                                                    | Must     |
| PF-02 | Four roles enforced via auth claims and PostgreSQL RLS: `client`, `delivery_manager`, `bsg_leadership`, `super_admin`. | Must     |
| PF-03 | Every client-scoped table carries `org_id`; RLS policies join on `org_id` for tenant isolation.                        | Must     |
| PF-14 | Every natural-language agent query logged in `agent_queries` with question, answer, model, and latency.                | Must     |
| PF-15 | Every agent answer has ≥ 1 evidence link in `agent_query_evidence_links`, committed in the same transaction.           | Must     |
| PF-17 | Evidence links are immutable once created.                                                                             | Must     |
| PF-11 | Super Admin configures client-facing metrics via `metric_configurations` (runtime, not hardcoded).                     | Must     |

## 2. BRD — Delivery Performance Agent (Agent 1, Phase 1)

### 2.1 Executive overview — mandate

The Delivery Performance Agent is an AI-native operational intelligence system that continuously monitors delivery execution across data operations programs and provides predictive, actionable, and evidence-backed recommendations. It transforms fragmented delivery workflows into a proactive decision-support ecosystem, helping Project Managers, Delivery Leads, and Operations teams predict delivery risks, detect bottlenecks, optimize workforce utilization (limited MVP scope), and reduce escalation frequency.

### 2.2 Strategic objective

Build an AI-native delivery intelligence layer capable of monitoring operational health in near real time, predicting schedule risks, recommending mitigation actions, prioritizing interventions, and reducing reactive operational management.

### 2.3 Agent mission statement

> *"Continuously monitor delivery execution signals, identify operational risks, predict delivery outcomes, recommend mitigation strategies, and generate prioritized operational action plans for delivery leadership."*

### 2.4 Primary user personas


| Persona                          | Primary need                                                           | Platform role      |
| -------------------------------- | ---------------------------------------------------------------------- | ------------------ |
| Project Manager                  | Daily operational prioritization                                       | `delivery_manager` |
| Delivery Lead / Delivery Manager | Portfolio delivery governance                                          | `delivery_manager` |
| Operations Manager               | Workforce balancing and throughput management                          | `delivery_manager` |
| QA Lead                          | Quality-driven delivery risk visibility                                | `delivery_manager` |
| Resource Manager                 | Utilisation and SME allocation insights                                | `delivery_manager` |
| Program / BSG Leadership         | Delivery forecasting and operational transparency (limited in Phase 1) | `bsg_leadership`   |


### 2.5 Business problems, objectives, and key questions

**Problems addressed:**

- PMs spend excessive time consolidating spreadsheets for status reporting.
- Delay management is reactive rather than predictive.
- Lack of real-time visibility into throughput and milestone health.
- Difficulty identifying root causes of delivery slippage.
- Inefficient workforce utilisation across distributed teams (India, Kosovo).
- Absence of predictive delivery governance.

**Key questions the agent must answer:**

- Which projects are at delivery risk?
- Which milestones will slip?
- What changed since last week?
- Which workflow is slowing throughput?
- Are teams optimally utilised? *(limited MVP — rebalancing recommendations only)*
- What should the PM focus on today?

### 2.9 Data inputs and reasoning logic

**Structured inputs:** Delivery tracker, milestone plan, throughput logs, resource allocation, QA/rework sheets, risk register, backlog queue.

**Unstructured inputs (Phase 1 via RAG):** SOPs, project charters, escalation notes, PM notes.

**Illustrative reasoning logic:**

```
IF throughput declines
   AND rework increases
   AND absenteeism spikes
THEN raise risk score
     forecast delay probability
     generate mitigations
     trigger escalation alert
OUTPUT risk tiers: Low | Medium | High | Critical
```

| `bottlenecks`                                  | Auto-detected workflow blockers (distinct from risk alerts)                             |
| `teams` / `annotators`                         | Workforce context for utilisation and rebalancing                                       |
| `agent_queries` / `agent_query_evidence_links` | NL query audit trail                                                                    |
| `notifications`                                | Proactive alert delivery (`risk_alert`, `milestone_at_risk`)                            |
| `metric_configurations`                        | Runtime thresholds (schedule confidence, on-time delivery)                              |


**Business rules (application layer):**

1. `org_id` for all queries must come from JWT claims, never from request body.
2. Delivery-confidence threshold drives milestone `at_risk` status — read from `metric_configurations`, not hardcoded.
3. Agent query inserts require ≥ 1 evidence link in the same transaction.

## 3. BRD — Quality Intelligence Agent (Agent 2, Phase 1)

### 3.1 Executive overview — mandate

The Quality Intelligence Agent provides error trend analysis, quality drift monitoring, reviewer consistency insights, rework analysis, root-cause intelligence, and training recommendation triggers. It enables QA Leads and PMs to see quality-driven delivery risk before it becomes a client issue.

### 3.2 Strategic objective

Build an AI-native quality intelligence layer that continuously monitors annotation and review quality signals, detects drift early, surfaces structured error taxonomies with corrective actions, and links quality degradation to delivery velocity risk.

### 3.3 Agent mission statement

> *"Continuously monitor quality execution signals, detect drift and error trends, identify root causes, and recommend targeted corrective interventions before quality issues impact delivery milestones."*

*(Paraphrased from ecosystem summary — no dedicated mission statement in source BRD.)*

### 3.4 Primary user personas


| Persona            | Primary need                                                      | Platform role      |
| ------------------ | ----------------------------------------------------------------- | ------------------ |
| QA Lead            | Quality-driven delivery risk visibility before client escalation  | `delivery_manager` |
| Project Manager    | Understand whether quality trends are affecting delivery velocity | `delivery_manager` |
| Delivery Lead      | Portfolio-level quality governance                                | `delivery_manager` |
| Client Stakeholder | Quality metrics on client-facing dashboard (when enabled)         | `client`           |
| Client Leadership  | Executive confidence in quality governance                        | `client`           |


### 3.5 Business problems, objectives, and key questions

**Problems addressed:**

- Quality signals scattered across QA logs, rework sheets, and ad-hoc reports.
- Quality drift detected too late — after it has already impacted throughput.
- No structured error taxonomy linking issues to corrective actions.
- Inconsistent quality metrics across teams and sites.

**Key questions the agent must answer:**

- Is quality impacting delivery velocity?
- Which error categories are increasing?
- Are quality trends affecting delivery timelines?
- Which projects require additional QA intervention?

### 3.6 AI agent capabilities

### 3.10 Data inputs and reasoning logic

```
IF gold_set_accuracy_pct < threshold
   OR iaa_krippendorff_alpha < threshold
   OR rework_rate_pct > threshold
THEN set has_drift_alert = TRUE
     insert risk_alert (alert_type = 'quality_drift')
     populate quality_error_entries with root-cause breakdown
```

## 4. BRD — Client Interaction Agent (Agent 5, Phase 1)

### 4.1 Executive overview — mandate

The Client Interaction Agent is an AI-native client engagement and operational transparency system that transforms delivery data into executive-ready narratives, confidence-driven communication, predictive project transparency, and operational trust signals. It enables clients to understand delivery health, identify risks early, and access real-time visibility without depending entirely on PM-driven reporting cycles.

### 4.2 Strategic objective

Build an AI-native client transparency layer capable of improving client confidence, enhancing transparency, accelerating communication, reducing uncertainty, simplifying operational complexity, and enabling proactive engagement — with mandatory human-in-the-loop approval for all outbound communications.

### 4.3 Agent mission statement

> *"Continuously translate operational delivery intelligence into transparent, explainable, confidence-based client narratives and actionable engagement insights."*

### 4.4 Primary user personas


| Persona                                     | Primary need                                                         | Platform role      |
| ------------------------------------------- | -------------------------------------------------------------------- | ------------------ |
| Client Stakeholder / Client Program Manager | Delivery visibility and operational transparency                     | `client`           |
| Client Leadership                           | Executive-level confidence with auditable evidence                   | `client`           |
| Delivery Manager                            | Review and approve AI-drafted client communications                  | `delivery_manager` |
| Delivery Lead                               | Relationship governance and communication quality                    | `delivery_manager` |
| Internal PM                                 | AI-assisted communication drafting (agent generates; human approves) | `delivery_manager` |


### 4.5 Business problems, objectives, and key questions

**Problems addressed:**

- Slow client response cycles dependent on PM availability.
- Inconsistent operational narratives across reporting cycles.
- Reactive communication during delivery volatility.
- Low client confidence during periods of uncertainty.

**Key questions the agent must answer:**

- Is my project on track?
- Why is there a delay?
- What is the current project health?
- Which milestones are at risk?
- What risks exist and what mitigations are in progress?

**Business rules (application layer):**

1. 100% of outbound client communications require explicit Delivery Manager approval before `sent`.
2. Communication drafts require ≥ 1 `communication_evidence_links` row in the same transaction.
3. Only `service_role` backend account may INSERT `client_communications`; DMs may only UPDATE approval/status fields.
4. CSAT inserts must verify `submitted_by` carries `role = 'client'` (confirm DM proxy submission — see Section 8).

### 4.11 Outputs

Project health summaries, executive status reports, delivery-confidence scores, risk narratives, approved client communications.

### 4.12 Governance requirements


| Rule                           | Enforcement                                               |
| ------------------------------ | --------------------------------------------------------- |
| No hallucinated project status | Evidence links mandatory on all narratives                |
| Human-in-the-loop              | DM approval gate before `sent`                            |
| Confidence scoring mandatory   | `delivery_confidence_scores` on client-facing views       |
| Sensitive data protection      | RBAC scopes all client data to `org_id`                   |
| Audit trail                    | `client_communications` lifecycle fields + evidence links |


### 4.13 Non-functional requirements


| Category   | Requirement                                                   |
| ---------- | ------------------------------------------------------------- |
| Security   | Tenant isolation; clients see only own org data               |
| Trust      | Every narrative cites operational evidence                    |
| Latency    | Client NL queries near real-time                              |
| Compliance | Regulated-domain handling (Life Sciences, Finance, Logistics) |


### 4.14 Technical architecture


| Component                      | Role                                                                                 |
| ------------------------------ | ------------------------------------------------------------------------------------ |
| KPI semantic layer             | Operational data → client-facing metrics                                             |
| Narrative generation framework | LLM-driven summary drafting                                                          |
| RAG architecture               | Grounds client Q&A in source data                                                    |
| Confidence scoring engine      | Milestone reliability computation                                                    |
| Role-based access layer        | RLS + JWT claims                                                                     |
| FastAPI backend                | `backend/app/agents/client_interaction/`                                             |
| React frontend                 | `frontend/src/agents/client-interaction/` — client dashboard, comms review, NL query |


### 4.15 Out of scope (this agent)

- Project readiness / go-live readiness assessment engine.
- Multi-client portfolio intelligence, AI relationship sentiment analysis, predictive client-escalation detection.
- Autonomous sending of client communications (human approval is always required).
