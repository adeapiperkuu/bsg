# API Specification

> Binding API contract for the Operations Tower backend.
> This document translates the product scope, technology stack, and database schema into REST endpoints for the FastAPI service. The API must protect tenant isolation, preserve evidence-backed AI outputs, and support the Phase 1 agent set: Delivery Performance, Quality Intelligence, and Client Interaction.

---

## 1. Purpose

The API is the controlled backend layer between the React frontend, Supabase PostgreSQL, Supabase Auth, and the AI/agent services.

It must provide:

- Role-scoped access to projects, milestones, throughput, quality, risk alerts, communications, notifications, and metrics.
- Safe write paths for delivery managers and service processes.
- Read-only client views that only expose approved, tenant-scoped information.
- Evidence-backed agent query responses.
- Human review and approval workflows for AI-drafted client communications.
- Admin configuration for client-visible metrics.

The API is not responsible for replacing Supabase Auth login flows. User authentication is handled by Supabase Auth. The FastAPI backend verifies the Supabase JWT on every protected request and uses the authenticated user's role and `org_id` to enforce authorization.

---

## MVP Scope

Phase 1 includes only:

- Delivery Performance Agent.
- Quality Intelligence Agent.
- Client Interaction Agent.

The MVP UI must not expose Workforce & Capability, Project Governance, Operational Knowledge, Go-Live Readiness, Margin Analytics, ROI Analytics, Churn Prediction, autonomous multi-agent orchestration, or other portfolio intelligence features. These are Phase 2+ capabilities and may only appear in code as disabled future placeholders when explicitly needed for compatibility.

## 2. API Style

| Decision | Specification |
|---|---|
| Protocol | HTTPS only |
| Style | REST, JSON over HTTP |
| Backend framework | FastAPI |
| Version prefix | `/api/v1` |
| Auth model | Supabase JWT sent to backend |
| Primary identity source | Supabase Auth `auth.users` plus local `users` table |
| Tenant isolation | Enforced by both API filters and PostgreSQL RLS |
| Error format | Consistent JSON error object |
| Pagination | Cursor pagination for time-series/audit lists; page pagination acceptable for small admin lists |
| Date/time format | Dates use `DD-MM-YYYY`; timestamps use `DD-MM-YYYYTHH:mm:ssZ` examples |
| IDs | UUID strings |

No GraphQL, tRPC, or unversioned endpoints should be introduced unless this document is updated.

## 4. Authentication

### 4.1 Login

Login is handled by Supabase Auth from the frontend. The backend does not need a username/password login endpoint for MVP.

Frontend flow:

1. User signs in through Supabase Auth.
2. Supabase returns a JWT access token.
3. Frontend sends the token to FastAPI on every API request.
4. FastAPI validates the JWT and loads the matching row from `users`.
5. FastAPI applies role checks and `org_id` scoping before querying data.

### 4.2 Authorization Header

```http
Authorization: Bearer <supabase_access_token>
```

Every endpoint except health checks requires this header.

### 4.3 Authenticated User Context

After token verification, the backend must derive this request context:

```json
{
  "user_id": "uuid",
  "org_id": "uuid",
  "email": "person@example.com",
  "role": "client",
  "is_active": true
}
```

The backend must never trust `org_id`, `role`, or `user_id` from request bodies for normal tenant-scoped endpoints. These values must come from the verified JWT and the `users` table.

### 4.4 Inactive Users

If `users.is_active = false` or `users.deleted_at IS NOT NULL`, return:

```http
403 Forbidden
```

```json
{
  "error": {
    "code": "USER_INACTIVE",
    "message": "This user account is not active.",
    "details": {}
  }
}
```

## 5. Roles and API Permissions

The API supports four roles:

| Role | Description |
|---|---|
| `client` | External client user. Can only see their own organisation's approved/read-only project information. |
| `delivery_manager` | Internal delivery owner for one organisation/client engagement. Can manage delivery data and approve AI communications for their own `org_id`. |
| `bsg_leadership` | Internal leadership role. Can read cross-client portfolio data but should not mutate operational records in MVP. |
| `super_admin` | Platform administrator. Can manage organisations, users, and metric configurations. |

### 5.1 Permission Summary

| Resource | Client | Delivery Manager | BSG Leadership | Super Admin |
|---|---:|---:|---:|---:|
| Own profile | Read | Read | Read | Read |
| Users | Own row only | Read own org | Read all | Full access |
| Organisations | No direct access | No direct access | Read all | Full access |
| Projects | Read own org | Read/write own org | Read all | Full access |
| Milestones | Read own org | Read/write own org | Read all | Full access |
| Throughput snapshots | Read own org | Read/write own org | Read all | Full access |
| Teams | Read own org | Read/write own org | Read all | Full access |
| Annotators | No access | Read/write own org | Read all | Full access |
| Quality snapshots | Read own org | Read/write own org | Read all | Full access |
| Quality error entries | Read own org | Read/write own org | Read all | Full access |
| Risk alerts | Read own org | Read/write own org | Read all | Full access |
| Bottlenecks | Read own org | Read/write own org | Read all | Full access |
| Client communications | Read sent only | Review/approve/send own org | Read all | Full access |
| Agent queries | Own queries only | Read own org | Read all | Full access |
| Metric configurations | Read visible config | Read config | Read config | Full access |
| Notifications | Own notifications | Own notifications | Own notifications | Full access |
| CSAT scores | Create/read own submissions | Read own org | Read all | Full access |

PostgreSQL RLS must mirror these rules. API authorization checks are still required; RLS is the safety backstop, not the only enforcement layer.

## 6. Global Request Conventions

### 6.1 Content Type

```http
Content-Type: application/json
Accept: application/json
```

### 6.2 ID Format

All resource IDs are UUID strings:

```json
{
  "id": "3a0f8a6a-7d6f-41f5-9c23-74b1a5e52e4a"
}
```

### 6.3 Date and Timestamp Format

Dates use `DD-MM-YYYY`.

```json
{
  "snapshot_date": "19-06-2026"
}
```

Timestamps include the `DD-MM-YYYY` date component plus time and timezone.

```json
{
  "created_at": "19-06-2026T09:30:00Z"
}
```

### 6.4 Soft Deletes

Resources with `deleted_at` are soft deleted. Delete endpoints must set `deleted_at` rather than physically deleting rows.

Default list/get endpoints must exclude soft-deleted rows.

### 6.5 Pagination

List endpoints should support:

| Parameter | Type | Default | Notes |
|---|---|---:|---|
| `limit` | integer | `50` | Maximum `100` |
| `cursor` | string | null | Opaque cursor for next page |

Example response:

```json
{
  "data": [],
  "pagination": {
    "limit": 50,
    "next_cursor": null
  }
}
```

Small admin lists may use page pagination if cursor pagination adds unnecessary complexity, but the API should prefer cursor pagination for snapshots, query logs, notifications, and communications.

### 6.6 Filtering

Common filters:

| Filter | Applies to | Notes |
|---|---|---|
| `project_id` | Project child resources | Must be tenant-scoped |
| `status` | Projects, milestones, alerts, communications | Enum value |
| `date_from` / `date_to` | Time-series data | Inclusive bounds |
| `created_from` / `created_to` | Audit logs | Inclusive bounds |
| `team_id` | Quality and team resources | Must be same `org_id` |
| `risk_tier` | Risk alerts | `low`, `medium`, `high`, `critical` |

The backend must validate every filter. Unknown filters should return `400 Bad Request`, not be ignored silently.

### 6.7 Sorting

Common sort parameters:

```http
GET /api/v1/projects?sort=created_at:desc
```

Only allow explicit whitelisted fields per endpoint. Do not pass arbitrary sort strings into SQLAlchemy.

## 8. Error Model

All errors must use this shape:

```json
{
  "error": {
    "code": "PROJECT_NOT_FOUND",
    "message": "Project was not found.",
    "details": {
      "project_id": "uuid"
    }
  }
}
```

### 8.1 Standard HTTP Codes

| Status | Meaning |
|---:|---|
| `400` | Invalid request body, invalid filter, invalid transition |
| `401` | Missing or invalid JWT |
| `403` | Authenticated user lacks permission |
| `404` | Resource not found or not visible to this user |
| `409` | Unique constraint or state conflict |
| `422` | Pydantic validation error |
| `429` | Rate limit exceeded |
| `500` | Unexpected server error |
| `503` | External service unavailable, such as LLM provider |

Use `404` rather than `403` when revealing the existence of a cross-tenant resource would leak information.

### 8.2 Common Error Codes

| Code | Meaning |
|---|---|
| `AUTH_REQUIRED` | Missing bearer token |
| `INVALID_TOKEN` | Token cannot be verified |
| `USER_INACTIVE` | User exists but is inactive |
| `FORBIDDEN` | Role does not allow the action |
| `NOT_FOUND` | Resource not found or not visible |
| `VALIDATION_ERROR` | Request failed schema validation |
| `TENANT_SCOPE_REQUIRED` | Backend could not determine `org_id` |
| `INVALID_STATUS_TRANSITION` | Resource lifecycle transition is not allowed |
| `EVIDENCE_REQUIRED` | AI output has no evidence links |
| `COMMUNICATION_APPROVAL_REQUIRED` | Communication cannot be sent before approval |
| `LLM_PROVIDER_UNAVAILABLE` | AI provider failed or timed out |

### 10.8 `POST /projects/{project_id}/throughput`

Inserts or updates one daily throughput snapshot.

Allowed roles:

- `delivery_manager`
- `super_admin`

Request:

```json
{
  "snapshot_date": "18-06-2026",
  "units_completed": 1180,
  "units_forecast": 1225
}
```

Service behavior:

1. Verify the project is visible to the requester.
2. Derive `org_id` from the project.
3. Compute `rolling_7day_units`.
4. Upsert by `(project_id, snapshot_date)`.
5. Optionally trigger delivery confidence recomputation.
6. Optionally create or update a `risk_alerts` row if forecast risk crosses threshold.

## 12. Service Layer Rules

These rules must be implemented in FastAPI services, not only in route handlers.

### 12.1 Tenant Scoping

Every tenant-scoped query must include `org_id`.

Bad:

```python
select(Project).where(Project.id == project_id)
```

Good:

```python
select(Project).where(
    Project.id == project_id,
    Project.org_id == current_user.org_id,
    Project.deleted_at.is_(None),
)
```

Leadership and Super Admin flows still need explicit role checks before cross-org reads.

### 12.2 Evidence-Backed AI

The API must not store an AI answer without evidence.

Required transaction behavior:

1. Run agent reasoning.
2. Resolve source rows used as evidence.
3. Insert `agent_queries` or `client_communications`.
4. Insert evidence link rows.
5. Commit only if at least one evidence link exists.

### 12.3 Human Approval Gate

The Client Interaction Agent may draft communication, but it may not send it.

Only a human delivery manager or approved admin role may approve the final content.

### 12.4 Thresholds Are Configurable

The following values are draft defaults and must not be hardcoded permanently:

| Metric | Draft threshold |
|---|---:|
| Schedule confidence on track | `>= 80%` |
| Gold-set accuracy | `>= 95%` |
| Krippendorff alpha | `>= 0.85` |
| Rework rate | `<= 5%` |

Where possible, use `metric_configurations` or a future threshold configuration mechanism.

### 12.5 Client-Safe Data Shaping

Client responses must not expose:

- Other organisations.
- Annotator personal details.
- Internal-only operational commentary.
- Draft or rejected communications.
- Cross-client portfolio analytics.
- Model/provider internals beyond a safe label if necessary.

### 12.6 Immutable Audit Records

These records should be append-only:

- `agent_queries`
- `agent_query_evidence_links`
- `communication_evidence_links`
- `delivery_confidence_scores`
- `client_csat_scores`

Correction flows should create new records or use explicit admin-only correction paths where needed.

## 14. Security Requirements

- All endpoints except `/health` and `/ready` require JWT authentication.
- All tenant-scoped endpoints must derive `org_id` from authenticated context or a validated parent resource.
- Do not accept `role`, `user_id`, or `org_id` from client request bodies unless the endpoint is explicitly Super Admin-only.
- Use PostgreSQL RLS on every table named in `10. DB Schema.md`.
- Do not expose service-role Supabase credentials to the frontend.
- Do not log raw client data, full prompts, or sensitive communication bodies in application logs.
- Audit AI outputs with evidence links.
- Return `404` for cross-tenant resource access attempts where `403` would reveal the resource exists.
- Use synthetic or sanitized data in local/staging.

## 16. Final Resolved Decisions

| Decision | Resolution |
|---|---|
| Date format | All API date fields and examples use `DD-MM-YYYY`. Timestamp examples use `DD-MM-YYYYTHH:mm:ssZ`. |
| MVP agents | Phase 1 exposes only Delivery Performance, Quality Intelligence, and Client Interaction. |
| Phase 2+ visibility | Workforce & Capability, Project Governance, Operational Knowledge, Go-Live Readiness, Margin Analytics, ROI Analytics, Churn Prediction, and portfolio intelligence must not appear in the MVP UI. |
| Report generation | Weekly Client Summaries and Executive Reports are manually generated in MVP by Delivery Managers or Super Admins. |
| Scheduler | No scheduler, cron, background worker, or job queue is required for MVP report generation. Automatic generation is Phase 2+. |
| Report delivery/export | MVP supports in-app viewing only. PDF export, DOCX export, and outbound email delivery are Phase 2+. |
| Communication send behavior | In MVP, `send` publishes an approved communication in-app by setting `status = sent` and `sent_at`. |
| CSAT cadence | CSAT is collected monthly per project. |
| CSAT submitter | Only client-role users submit CSAT. Delivery Managers do not submit on behalf of clients in MVP. |
| CSAT uniqueness | One CSAT submission per client user, project, and reporting month. |
