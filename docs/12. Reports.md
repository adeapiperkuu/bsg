# Reports

> Reporting specification for the BSG Operations Tower.
> This document is based on `BSG_Ops_Intelligence_Agent_Optimized v1.0.docx` and aligns with the API, database, and AI/RAG architecture documents. It defines what reports the platform should produce, who receives them, what data they use, and what governance rules apply.

---

## 1. Purpose

The reporting layer turns operational delivery data into clear, role-scoped, evidence-backed communication.

The source Word document describes a shift from manual, PM-heavy reporting toward AI-assisted operational transparency. Clients increasingly expect real-time visibility, predictive insights, quality traceability, and consistent governance. Internally, BSG needs faster issue detection, less manual reporting effort, and more standardized operational narratives.

Reports must help users answer:

- Is the project on track?
- What changed since the last reporting cycle?
- Which milestones are at risk?
- Is quality affecting delivery?
- What risks exist, and what is being done about them?
- What should the PM or Delivery Manager focus on next?
- What can be safely communicated to the client?

The reporting layer must not create ungrounded narratives. Every AI-generated report, summary, or client-facing update must be traceable to operational evidence.

---

## MVP Scope

Phase 1 reporting includes only outputs supported by:

- Delivery Performance Agent.
- Quality Intelligence Agent.
- Client Interaction Agent.

The MVP UI must not expose Workforce & Capability, Project Governance, Operational Knowledge, Go-Live Readiness, Margin Analytics, ROI Analytics, Churn Prediction, autonomous multi-agent orchestration, or other portfolio intelligence features. These remain Phase 2+ even if the source concept document mentions them.

## 2. Reporting Principles

| Principle | Reporting Meaning |
|---|---|
| Evidence-backed | Every report must be based on source data and cite evidence where generated by AI. |
| Role-scoped | Clients, Delivery Managers, BSG Leadership, and Super Admins see different reporting depth. |
| Client-safe | Client-facing reports must not expose other tenants, internal-only notes, annotator details, or draft content. |
| Human-approved | AI-drafted outbound client communications require Delivery Manager review and approval before sending. |
| Configurable metrics | Client-facing metrics are controlled by Super Admin configuration, not hardcoded. |
| Explainable | Reports should explain what happened, why it matters, impact, confidence, and recommended action. |
| Consistent | Reports should use standard sections, metric names, statuses, and tone across projects. |
| Auditable | Report content must be traceable to source rows through evidence links. |

## 3. Report Audiences

### 3.1 Client Users

Client users need project transparency without internal operational clutter.

They should see:

- Their own projects only.
- Approved and sent summaries.
- Project health.
- Delivery confidence.
- Milestone status.
- Client-visible quality metrics.
- Client-safe risk explanations.
- Mitigation status.
- Their own CSAT submissions where applicable.

They should not see:

- Other clients.
- Internal workforce details.
- Annotator names.
- Internal PM notes.
- Draft or rejected communications.
- Cross-client portfolio analytics.

### 3.2 Delivery Managers

Delivery Managers need operational detail and actionability.

They should see:

- All projects within their own organisation/client engagement.
- Delivery and quality dashboards.
- Throughput trends.
- Open bottlenecks.
- Risk alerts.
- AI-drafted communications awaiting review.
- Evidence behind AI-generated reports.
- Internal recommendations and action plans.

They can:

- Review AI-drafted client updates.
- Edit approved report text.
- Approve or reject communications.
- Mark approved communications as sent.

### 3.3 BSG Leadership

BSG Leadership needs portfolio-level visibility.

They should see:

- Cross-client delivery trends.
- Risk concentration.
- Operational predictability.
- Client transparency outcomes.
- Quality trends.
- Reporting cycle performance.
- PM reporting effort reduction, if measured.

The project summary marks some leadership analytics, such as margin, unit economics, automation ROI, churn early warning, and broader portfolio intelligence as later-phase capabilities. They must not appear in the MVP UI.

### 3.4 Super Admin

Super Admin manages reporting configuration, especially which metrics are visible on client-facing dashboards and summaries.

Super Admin should be able to:

- Configure metric labels.
- Toggle client visibility.
- Set display order.
- Manage platform-wide reporting settings.

Super Admin access does not remove the need for careful data governance in generated reports.

## 4. Report Types

### 4.1 Project Health Summary

The Project Health Summary is the main operational report for one project.

Primary users:

- Client users.
- Delivery Managers.
- BSG Leadership.

Purpose:

- Summarize current project status.
- Explain whether the project is on track.
- Show current delivery confidence.
- Highlight quality and risk issues.
- Provide next actions or mitigations.

Recommended sections:

1. Project status.
2. Current milestone.
3. Delivery confidence.
4. Throughput trend.
5. Quality trend.
6. Open risks.
7. Mitigation actions.
8. Evidence links.

Client version should be concise and business-friendly. Internal version can include more operational detail.

### 4.2 Weekly Client Summary

The Weekly Client Summary is an AI-assisted update for client stakeholders.

Primary users:

- Delivery Managers.
- Client stakeholders.
- Client program managers.

Purpose:

- Reduce manual PM reporting effort.
- Standardize client communication.
- Give clients consistent visibility.
- Explain delivery and quality changes since the previous reporting cycle.

Recommended sections:

1. Executive headline.
2. Delivery progress this week.
3. Milestone status.
4. Delivery confidence.
5. Quality summary.
6. Risks and mitigations.
7. Decisions or support needed from the client.
8. Evidence-backed notes.

Generation rule:

- AI may draft it.
- Draft generation is manual in MVP. A Delivery Manager or Super Admin explicitly requests the draft.
- Weekly summaries are not automatically scheduled in Phase 1.
- Delivery Manager must review and approve it.
- Client can only see it after `status = sent`.

### 4.3 Executive Status Report

The Executive Status Report is a higher-level summary for client leadership or BSG leadership.

Primary users:

- Client leadership.
- BSG leadership.
- Delivery leadership.

Purpose:

- Communicate health, confidence, and risk at a leadership level.
- Avoid excessive operational detail.
- Highlight decisions, escalations, and business impact.

Generation rule:

- Executive reports are manually generated in MVP.
- They are not automatically scheduled in Phase 1.
- They are published in-app only after human approval when client-facing.

Recommended sections:

1. Overall status.
2. Confidence score.
3. Key achievements.
4. Material risks.
5. Mitigation progress.
6. Quality confidence.
7. Upcoming milestone outlook.
8. Leadership attention needed.

Tone:

- Clear.
- Calm.
- Executive-ready.
- Transparent about risk.
- Focused on mitigation and confidence.

### 4.4 Delivery Performance Report

The Delivery Performance Report supports PMs and Delivery Managers.

Primary users:

- Project Managers.
- Delivery Managers.
- Operations Managers.
- BSG Leadership.

Purpose:

- Explain project delivery performance.
- Identify schedule risk.
- Surface bottlenecks.
- Prioritize action.

Recommended sections:

1. Active milestones.
2. Throughput actual vs forecast.
3. Rolling 7-day throughput.
4. Delivery confidence trend.
5. Open bottlenecks.
6. Open risk alerts.
7. Root-cause explanation.
8. Recommended operational actions.

Source tables:

- `projects`
- `milestones`
- `throughput_snapshots`
- `delivery_confidence_scores`
- `risk_alerts`
- `bottlenecks`
- `teams`
- `quality_snapshots`

### 4.5 Quality Intelligence Report

The Quality Intelligence Report explains whether quality trends are stable, improving, or becoming a delivery risk.

Primary users:

- QA Leads.
- Project Managers.
- Delivery Managers.
- BSG Leadership.
- Client users, in a client-safe form.

Purpose:

- Track gold-set accuracy.
- Track inter-annotator agreement.
- Track rework rate.
- Explain quality drift.
- Summarize top error categories.
- Recommend corrective action.

Recommended sections:

1. Quality status.
2. Gold-set accuracy.
3. IAA.
4. Rework rate.
5. Drift alert status.
6. Error category breakdown.
7. Delivery impact.
8. Recommended QA action.

Source tables:

- `quality_snapshots`
- `quality_error_entries`
- `projects`
- `teams`
- `risk_alerts`
- `throughput_snapshots`

### 4.6 Risk and Escalation Report

The Risk and Escalation Report summarizes open risks, risk tier, contributing causes, and mitigation progress.

Primary users:

- Delivery Managers.
- Project Managers.
- BSG Leadership.
- Client users, where risk is client-visible.

Purpose:

- Make delivery and quality risk visible early.
- Explain what caused the risk.
- Track whether mitigation is underway.
- Support escalation decisions.

Recommended sections:

1. Risk overview.
2. Open high/critical risks.
3. Risk type.
4. Slippage probability, if available.
5. Contributing causes.
6. Mitigation owner.
7. Current status.
8. Next review date, if available.

Source tables:

- `risk_alerts`
- `bottlenecks`
- `milestones`
- `throughput_snapshots`
- `quality_snapshots`
- `delivery_confidence_scores`

### 4.7 Delivery Confidence Report

The Delivery Confidence Report explains the likelihood that a project or milestone will be delivered on time.

Primary users:

- Client users.
- Delivery Managers.
- BSG Leadership.

Purpose:

- Make milestone reliability transparent.
- Explain why confidence changed.
- Connect confidence to throughput, quality, and open risks.

Recommended sections:

1. Current confidence score.
2. Confidence trend.
3. Main positive drivers.
4. Main negative drivers.
5. Milestone impact.
6. Recommended mitigation.
7. Evidence.

Source table:

- `delivery_confidence_scores`

Supporting tables:

- `milestones`
- `throughput_snapshots`
- `quality_snapshots`
- `risk_alerts`
- `bottlenecks`

### 4.8 Go-Live Readiness Report

The source Word document mentions readiness reports and go-live readiness as part of the Client Interaction Agent.

Status:

- Phase 2+ feature.
- Not part of the MVP UI.
- Not represented in the current MVP database schema.

Purpose:

- Assess whether a client project is ready to move into production or the next delivery phase.

Possible sections:

1. Resource readiness.
2. Training readiness.
3. Planning readiness.
4. Tracking readiness.
5. Risk preparedness.
6. Quality preparedness.
7. Gap analysis.
8. Mitigation recommendations.

Implementation note:

- A dedicated readiness assessment data model may be needed if this becomes a later-phase requirement.

### 4.9 CSAT Report

The CSAT Report summarizes client satisfaction data.

Primary users:

- Delivery Managers.
- BSG Leadership.
- Super Admin.

Purpose:

- Track client satisfaction over time.
- Identify confidence or communication issues.
- Connect client sentiment to delivery and reporting outcomes.

Source table:

- `client_csat_scores`

Current assumptions:

- Score range is `1` to `5`.
- Source material references `4.7/5 Client CSAT / rolling`.
- CSAT is collected monthly per project.
- Only client-role users submit CSAT.
- Delivery Managers do not submit CSAT on behalf of clients in MVP.
- The reporting period is stored as `reporting_period_month`, the first day of the reporting month in `DD-MM-YYYY` date format, for example `01-06-2026`.
- A client user may submit one CSAT score per project per month.

## 5. Client Communication Lifecycle

Reports that become outbound client communications must follow a controlled lifecycle.

Database table:

- `client_communications`

Evidence table:

- `communication_evidence_links`

Allowed communication types:

- `weekly_summary`
- `executive_summary`
- `ad_hoc`

Allowed statuses:

- `draft`
- `in_review`
- `approved`
- `sent`
- `rejected`

### 5.1 Lifecycle Flow

```text
AI generates draft
  -> Draft stored as client_communications.body_draft
  -> Evidence stored in communication_evidence_links
  -> Delivery Manager reviews
  -> Delivery Manager edits body_approved if needed
  -> Delivery Manager approves or rejects
  -> Approved communication can be sent
  -> Client can see only sent communications
```

### 5.2 Governance Rules

- AI drafts must never be sent automatically.
- Drafts must include evidence links.
- Delivery Manager approval is mandatory.
- Client users can only see `sent` communications.
- Rejected drafts remain internal.
- Draft body and approved body may differ.
- The system must record who reviewed and approved the communication.

### 5.3 Required Fields

| Field | Purpose |
|---|---|
| `comm_type` | Report type: weekly, executive, or ad hoc |
| `subject` | Client-visible title/subject |
| `body_draft` | AI-generated draft text |
| `body_approved` | Human-approved final text |
| `status` | Current lifecycle state |
| `drafted_by_agent` | Usually `client_interaction_agent` |
| `reviewed_by` / `reviewed_at` | Human review metadata |
| `approved_by` / `approved_at` | Human approval metadata |
| `sent_at` | Delivery timestamp |

## 6. Dashboard Reporting

Dashboards are interactive reports. They should show current state and let users drill into supporting evidence.

### 6.1 Client Dashboard

Purpose:

- Give clients real-time visibility without waiting for PM-driven reporting cycles.

Recommended modules:

1. Project health.
2. Active milestone.
3. Delivery confidence.
4. Client-visible metrics.
5. Quality summary.
6. Open risks with mitigation status.
7. Sent weekly/executive summaries.
8. CSAT submission, if enabled.

Client dashboard rules:

- Only show projects from the client's `org_id`.
- Use `metric_configurations.is_client_visible`.
- Hide internal-only fields.
- Hide draft communications.
- Explain risk in business language.

### 6.2 Delivery Manager Dashboard

Purpose:

- Help Delivery Managers govern active delivery and reporting workflows.

Recommended modules:

1. Active projects.
2. At-risk milestones.
3. Throughput variance.
4. Quality drift alerts.
5. Open bottlenecks.
6. Pending communication approvals.
7. Risk alerts by severity.
8. Recommended actions.

### 6.3 Leadership Dashboard

Purpose:

- Give leadership a portfolio-level operating view.

Recommended modules:

1. Projects by status.
2. Delivery confidence distribution.
3. High/critical risk concentration.
4. Quality trend overview.
5. Reporting cycle status.
6. Client transparency indicators.
7. CSAT trend, if available.

Scope rule:

- Cross-client margin, unit economics, automation ROI, churn early warning, and advanced relationship analytics are Phase 2+ and must not appear in the MVP UI.

## 9. AI-Generated Report Requirements

AI may help generate:

- Weekly summaries.
- Executive updates.
- Quality narratives.
- Delivery confidence explanations.
- Risk narratives.
- Project health summaries.
- Client Q&A answers.

AI-generated reports must:

- Use retrieved evidence.
- Include or store evidence links.
- Avoid unsupported claims.
- Respect role permissions.
- Use client-safe language for client-facing reports.
- Clearly state uncertainty where evidence is incomplete.
- Avoid suggesting that a draft is approved or sent before human approval.

### 9.1 Required AI Narrative Structure

Recommended structure:

```text
Headline:
One concise sentence describing current status.

What changed:
Explain meaningful change since the last cycle.

Why it matters:
Explain delivery, quality, or risk impact.

Mitigation:
Explain what action is underway or recommended.

Confidence:
State confidence level or score where available.

Evidence:
Reference source metrics or evidence links.
```

### 9.2 No-Hallucination Rule

The source document explicitly requires no hallucinated project status. Therefore:

- Do not invent progress.
- Do not invent mitigation actions.
- Do not invent delivery confidence.
- Do not invent quality metrics.
- Do not invent client approvals.
- Do not invent dates.
- Do not summarize unavailable documents as if they were retrieved.

If evidence is unavailable, the report should say so.

## 11. Report Cadence

| Report | Suggested Cadence | Trigger |
|---|---|---|
| Project Health Summary | On demand | Dashboard load or user request |
| Weekly Client Summary | Weekly | Manual Delivery Manager or Super Admin request |
| Executive Status Report | Weekly/monthly or on demand | Manual leadership, Delivery Manager, or Super Admin request |
| Delivery Performance Report | Daily/on demand | PM or Delivery Manager dashboard |
| Quality Intelligence Report | Weekly/on demand | New quality snapshot or QA review |
| Risk and Escalation Report | On demand and event-driven | New high/critical risk alert |
| Delivery Confidence Report | On demand | Confidence recompute or milestone review |
| CSAT Report | Monthly/on demand | Client submissions |

MVP rule: report generation is manual. No automatic report scheduler exists in Phase 1.

If automatic generation is selected in Phase 2+, required infrastructure includes:

- A scheduler or cron trigger.
- A background worker or job queue.
- Idempotent job records.
- Failure/retry handling.
- Delivery Manager notification after draft creation.
- The same evidence-link and human-approval requirements used by manual generation.

Automatic jobs must create `draft` communications only. They must never publish reports directly to clients.

## 12. Export and Delivery Strategy

MVP supports in-app report viewing only.

| Delivery/export option | MVP status | Notes |
|---|---|---|
| In-app viewing | Included | Approved/sent reports are visible in the Operations Tower UI. |
| PDF export | Phase 2+ | Not implemented in MVP. |
| DOCX export | Phase 2+ | Not implemented in MVP. |
| Outbound email delivery | Phase 2+ | Not implemented in MVP. |
| Scheduled automatic delivery | Phase 2+ | Requires scheduler/job infrastructure and human approval gate. |

MVP `send` behavior means publishing an approved communication in-app by setting `status = sent` and `sent_at`. It does not send an email or create a PDF/DOCX artifact.

## 13. Report States

For client communications, use the `communication_status` enum:

| State | Meaning |
|---|---|
| `draft` | AI-generated or manually created draft, not reviewed |
| `in_review` | Delivery Manager is reviewing |
| `approved` | Approved by Delivery Manager but not yet sent |
| `sent` | Delivered or marked as delivered to client |
| `rejected` | Rejected and not client-visible |

For dashboard reports generated on demand, state may not need to be persisted unless the generated report becomes a `client_communications` record.

- View own project dashboard.

- View configured client-visible metrics.
- Submit CSAT if enabled.
- Ask client-safe project questions.

- View drafts.
- Approve communications.
- View internal workforce data.

### 13.2 Delivery Manager

Can:

- View internal project reports for own organisation.
- Generate communication drafts.
- Review, edit, approve, reject, and send communications.
- View evidence links.
- Manage operational report inputs where allowed by API.

Cannot:

- Access other organisations unless explicitly granted.

### 13.3 BSG Leadership

Can:

- View cross-client reporting.
- View portfolio summaries.
- View risk and quality trends.

Cannot:

- Mutate operational records in MVP unless another spec changes this.

### 13.4 Super Admin

Can:

- Manage metric configuration.
- Manage system-wide settings.
- Access administrative reporting.

## 15. Evidence and Audit Trail

Every AI-generated report must preserve its evidence trail.

For client communications:

- Report record: `client_communications`
- Evidence records: `communication_evidence_links`

For AI Q&A or generated answers:

- Query record: `agent_queries`
- Evidence records: `agent_query_evidence_links`

Evidence links should include:

- Source table.
- Source row ID.
- Human-readable description.
- Creation timestamp.

Reports should be reviewable after the fact. A Delivery Manager or auditor should be able to understand which operational records supported a generated statement.

## 16. Report Quality Standards

Reports should be:

- Concise enough for the audience.
- Specific about current status.
- Clear about risk and mitigation.
- Consistent in metric names.
- Free from unsupported claims.
- Written in business language for clients.
- Detailed enough for internal operators.
- Traceable to source data.

### 16.1 Bad Report Behavior

Avoid:

- "Everything is on track" without evidence.
- Hiding known high-risk items in client updates.
- Exposing internal staff issues to clients unnecessarily.
- Mixing data from multiple clients.
- Reporting stale metrics without timestamp context.
- Treating provisional thresholds as final contractual SLAs.
- Sending AI drafts without human approval.

### 16.2 Good Report Behavior

Prefer:

- "Delivery confidence is 89% based on stable throughput and no open high-risk alerts."
- "Rework increased this week, but remains below the current draft threshold."
- "One milestone is at risk; mitigation is underway through additional review capacity."
- "The system does not have enough evidence to explain the change."

## 19. Final Resolved Decisions

| Decision | Resolution |
|---|---|
| Date format | All report examples use `DD-MM-YYYY` for dates. Timestamp examples use `DD-MM-YYYYTHH:mm:ssZ`. |
| MVP agents | Phase 1 reports are limited to Delivery Performance, Quality Intelligence, and Client Interaction. |
| Phase 2+ visibility | Workforce & Capability, Project Governance, Operational Knowledge, Go-Live Readiness, Margin Analytics, ROI Analytics, Churn Prediction, and portfolio intelligence must not appear in the MVP UI. |
| Weekly summaries | Weekly Client Summaries are manually generated in MVP by Delivery Managers or Super Admins. |
| Executive reports | Executive Reports are manually generated in MVP by Delivery Managers, Super Admins, or leadership request flows. |
| Scheduler | No scheduler, cron, worker, or queue is required in MVP. Automatic report generation is Phase 2+ and must create drafts only. |
| Export/delivery | MVP supports in-app viewing only. PDF export, DOCX export, outbound email, and scheduled delivery are Phase 2+. |
| Send behavior | `send` means publishing an approved communication in-app by setting `status = sent` and `sent_at`. |
| CSAT model | CSAT is monthly per project, submitted by client-role users only. |
| CSAT scale | CSAT uses a `1` to `5` score range. |
| CSAT uniqueness | One CSAT submission per client user, project, and reporting month. |
| Go-Live Readiness | Go-Live Readiness Reporting is Phase 2+ and excluded from MVP. |
