## 1. Security Philosophy & Non-Negotiables

The Operations Tower handles sensitive delivery and quality data for clients in Life Sciences, Finance/Insurance, and Logistics — all regulated domains. The platform surfaces AI-generated insights and client-facing narratives that, if incorrect or unauthorised, could damage client relationships, violate contracts, or breach regulatory obligations.

Five rules are absolute. They cannot be relaxed by a developer, overridden by a feature flag, or deferred to a later phase. Every other control in this document supports one or more of these.

**Rule 1 — Tenant isolation is a database-layer guarantee, not an application promise.**
One client must never be able to read, infer, or access another client's data through any mechanism — direct query, guessed URL, API enumeration, or error message. PostgreSQL Row-Level Security (RLS) is the enforcement layer. A bug in application code cannot defeat RLS. This is the primary reason RLS was chosen over application-layer RBAC.

**Rule 2 — No live client data before formal governance sign-off.**
Development, staging, and CI/CD environments must run exclusively on synthetic or sanitised datasets. Connecting real client operational data to any environment requires all four gates: explicit client approval, BSG governance sign-off, security validation, and compliance review — in writing, before the connection is made.

**Rule 3 — Every AI output must be traceable to its source evidence.**
No agent insight, risk alert narrative, or client communication draft may be presented or sent unless it is linked to the specific source data rows it was derived from. An AI output without evidence links is a system error, not a degraded-mode response. This is enforced by the transactional evidence-link architecture — an agent query or communication draft that has no evidence links must not be committed to the database.

**Rule 4 — 100% of outbound AI-drafted client communications require human approval.**
The Client Interaction Agent drafts communications; a human Delivery Manager must explicitly approve them before they are sent. No code path, background job, or automation may transition a `client_communications` row to `status = 'sent'` without a verified DM approval. This is both a governance requirement and a technical constraint enforced in the service layer.

**Rule 5 — AI responses are permission-aware.**
An agent answer must never contain data from a different tenant, a different role scope, or a higher privilege level than the requesting user's JWT authorises. Agents receive the requesting user's `org_id` and `role` claims and must scope all data retrieval accordingly. The LLM is never given raw cross-tenant data — only evidence that has already passed through RLS-filtered queries.

## 2. Threat Model Summary

The following threat categories are most relevant to this platform given its architecture (multi-tenant SaaS, AI-generated content, regulated client data, distributed delivery workforce in India and Kosovo).

| Threat | Likelihood | Impact | Primary control |
|---|---|---|---|
| Cross-tenant data leak via API route bug | Medium | Critical | PostgreSQL RLS (database-layer; not defeatable by app bug) |
| Cross-tenant data leak via LLM prompt injection | Medium | Critical | Evidence retrieval scoped by `org_id` before LLM call; no cross-tenant data in prompt |
| Unauthorised AI communication sent to client | Medium | High | Communication approval gate; status machine enforced in service layer |
| AI hallucination presented as fact | Medium | High | Evidence-link mandate; every output linked to source rows |
| JWT theft enabling session hijack | Low-Medium | High | `httpOnly` cookie storage; short-lived access tokens; TLS everywhere |
| Secret key exposure via logs or error messages | Medium | Critical | Structured log format with typed fields; no secrets in `message` string |
| Enumeration of another org's project IDs via guessed URLs | Medium | Medium | RLS returns 0 rows (not 403) on cross-tenant access; 404 response hides existence |
| Privilege escalation via JWT claim manipulation | Low | Critical | JWT signature validated against Supabase public key on every request |
| Supply chain attack via third-party dependency | Low-Medium | High | Pinned exact versions in `TECH_STACK.md`; no `latest` tags |
| Data exfiltration via internal ingestion endpoint | Low | Critical | Internal routes network-restricted or mTLS-protected; `service_role` key never in frontend |
| Synthetic data contaminated with real client data | Low | High | Seed scripts reviewed before merge; CI/CD never connects to production Supabase project |

## 3. Authentication

**Provider:** Supabase Auth [CONFIRMED]
**Strategy:** JWT — short-lived access token + long-lived refresh token [CONFIRMED]
**Token storage:** `httpOnly` cookie or Supabase SDK default of `localStorage`
**SSO / external IdP:** None planned for Phase 1 [CONFIRMED]
**MFA:** recommended for `delivery_manager`, `bsg_leadership`, and `super_admin` roles given their access to cross-org or platform-config data

### 3.1 Authentication flow

```
User                     Frontend                  Supabase Auth             FastAPI
  │                          │                           │                     │
  │── enter credentials ────►│                           │                     │
  │                          │── signInWithPassword ────►│                     │
  │                          │◄── access_token (JWT)     │                     │
  │                          │    refresh_token ─────────│                     │
  │                          │                           │                     │
  │                          │── store in httpOnly cookie│                     │
  │                          │   (never localStorage)    │                     │
  │                          │                           │                     │
  │── navigate to dashboard ►│                           │                     │
  │                          │── GET /api/v1/... ────────────────────────────►│
  │                          │   Cookie: access_token=<JWT>                   │
  │                          │                           │                     │
  │                          │                           │── validate JWT sig ─│
  │                          │                           │   extract uid, role │
  │                          │                           │   extract org_id ───│
  │                          │                           │◄── current_user ────│
  │                          │                           │                     │── DB query with RLS
  │                          │◄── JSON (role-scoped) ────────────────────────│◄── filtered rows
```

### 3.2 JWT validation requirements

Every FastAPI route that reads or writes data must pass through the auth middleware. The middleware must:

- Validate the JWT signature against Supabase's public key (JWKS endpoint). Never skip this step.
- Extract `sub` (user UUID), map to `users.id` to retrieve `role` and `org_id`.
- Reject tokens that are expired, malformed, or signed with an unexpected algorithm. Return `401`.
- Never log the raw JWT value. Log only `user_id` as a typed structured field.
- Attach `CurrentUser(id, role, org_id)` to the request context for downstream use.

### 3.3 Token lifetime and refresh

| Token | Lifetime | Storage |
|---|---|---|
| Access token | Short-lived (Supabase default: 1 hour) | `httpOnly` cookie |
| Refresh token | Long-lived (Supabase default: 60 days) | `httpOnly` cookie |

The frontend must handle 401 responses by silently refreshing the access token via the Supabase Auth SDK before retrying. If refresh fails (token revoked or expired), the user is redirected to the login page.

### 3.4 Account lifecycle

- User accounts are created only by `super_admin` or the backend service account — never by self-registration in Phase 1.
- Deactivated accounts (`users.is_active = FALSE`) must be rejected at the auth middleware layer, even if the JWT is still technically valid.
- A deactivated user's JWT must be invalidated by calling Supabase Auth's `admin.deleteUser()` or by using Supabase's `signOut` revocation — not just by setting `is_active = FALSE` in the `users` table.

## 4. Authorisation & RBAC

RBAC is enforced at two layers: the FastAPI route layer (role check via dependency) and the PostgreSQL database layer (RLS policies). Both must be in place. Neither alone is sufficient.

### 4.1 Role definitions

| Role | Assigned to | Scope |
|---|---|---|
| `client` | External client stakeholders | Their organisation's data only; read-mostly |
| `delivery_manager` | BSG India/Kosovo site leads, PMs | Their organisation's data; read + write + approve communications |
| `bsg_leadership` | BSG Braunschweig HQ | All organisations; read-only |
| `super_admin` | Platform administrators | Platform config tables only; no client operational data |

Roles are stored as the `app_role` PostgreSQL enum on `users.role` and mirrored as a custom claim in the Supabase Auth JWT.

### 4.2 Permission matrix — Phase 1

| Action | `client` | `delivery_manager` | `bsg_leadership` | `super_admin` |
|---|---|---|---|---|
| Read own org's projects, milestones, throughput | ✅ | ✅ | ✅ (all orgs) | ✅ |
| Read own org's quality snapshots and error taxonomy | ✅ | ✅ | ✅ (all orgs) | ✅ |
| Read own org's risk alerts | ✅ (open only) | ✅ | ✅ (all orgs) | ✅ |
| Read own org's confidence scores | ✅ | ✅ | ✅ (all orgs) | ✅ |
| Read sent client communications | ✅ (own org, sent only) | ✅ (all statuses) | ✅ | ✅ |
| Submit natural-language agent query | ✅ (own org scope) | ✅ (own org) | ✅ (read scope) | ✅ |
| Create / update projects and milestones | ❌ | ✅ (own org) | ❌ | ✅ |
| Acknowledge / resolve risk alerts | ❌ | ✅ (own org) | ❌ | ✅ |
| Approve or reject client communications | ❌ | ✅ (own org, must be DM) | ❌ | ✅ |
| Submit CSAT score | ✅ | ❌ | ❌ | ❌ |
| Mark notifications read | ✅ (own only) | ✅ (own only) | ✅ (own only) | ✅ |
| Read `metric_configurations` | ✅ (read) | ✅ (read) | ✅ (read) | ✅ (full CRUD) |
| Write `metric_configurations` | ❌ | ❌ | ❌ | ✅ |
| Create / deactivate organisations and users | ❌ | ❌ | ❌ | ✅ |
| Insert throughput/quality snapshots (ingestion) | ❌ | ❌ | ❌ | `service_role` only |

### 4.3 FastAPI route-level enforcement

Every route handler must declare its required roles via a dependency:

```python
# backend/app/core/security.py (planned)
def require_role(*allowed_roles: AppRole):
    async def dependency(current_user: CurrentUser = Depends(get_current_user)):
        if current_user.role not in allowed_roles:
            raise HTTPException(status_code=403, detail="Insufficient permissions")
        return current_user
    return dependency

# Usage in route:
@router.patch("/communications/{id}/approve")
async def approve_communication(
    id: UUID,
    user: CurrentUser = Depends(require_role(AppRole.delivery_manager, AppRole.super_admin)),
    db: AsyncSession = Depends(get_db),
):
    ...
```

The role check happens before any database query. A wrong-role request must never reach the DB.

### 4.4 Super Admin scope boundary

Super Admin bypasses RLS **only** on the `metric_configurations` table (platform-wide config). Super Admin does **not** bypass RLS on any client operational data table (`projects`, `milestones`, `throughput_snapshots`, `quality_snapshots`, `risk_alerts`, `client_communications`, etc.). This boundary must be confirmed before RLS policies are written — see Open Questions.

## 5. Tenant Isolation

Tenant isolation is the most critical security property of this platform. It is enforced in three independent layers. All three must be present before production.

### 5.1 Layer 1 — Database (PostgreSQL RLS)

Every table that holds client-scoped data carries `org_id UUID NOT NULL` (denormalised for RLS efficiency) and has Row-Level Security enabled.

```sql
-- Example RLS policy — client role
CREATE POLICY "client_read_own_org"
  ON projects
  FOR SELECT
  TO authenticated
  USING (
    org_id = (
      SELECT org_id FROM users WHERE id = auth.uid()
    )
    AND
    (SELECT role FROM users WHERE id = auth.uid()) = 'client'
  );

-- Example RLS policy — delivery_manager
CREATE POLICY "delivery_manager_own_org"
  ON projects
  FOR ALL
  TO authenticated
  USING (
    org_id = (
      SELECT org_id FROM users WHERE id = auth.uid()
    )
    AND
    (SELECT role FROM users WHERE id = auth.uid()) = 'delivery_manager'
  );
```

RLS policy requirements:

- Every client-data table must have `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` applied in the same migration that creates it. Never deploy a table to `staging` or `prod` without its RLS policy.
- A SELECT that would return rows from a different `org_id` must return zero rows — not a 403. This prevents existence leakage (an attacker cannot distinguish "no such project" from "project exists but you can't see it").
- RLS policies must be unit-tested in the backend test suite using separate database connections for each role.

### 5.2 Layer 2 — Application middleware (FastAPI)

Every route handler that reads client data must explicitly extract `org_id` from the validated JWT claims and scope all queries to that `org_id`. This is defence-in-depth — it does not replace RLS, but makes intent explicit and allows code reviewers to catch missing scopes before they reach production.

```python
# backend/app/api/v1/projects.py (planned)
@router.get("/projects")
async def list_projects(
    user: CurrentUser = Depends(require_role(AppRole.client, AppRole.delivery_manager)),
    db: AsyncSession = Depends(get_db),
):
    # org_id comes from the JWT — never from the request body or query params
    result = await db.execute(
        select(Project)
        .where(Project.org_id == user.org_id)   # explicit application-layer filter
        .where(Project.deleted_at.is_(None))
    )
    # RLS provides the second line of defence even if the above filter were missing
    return result.scalars().all()
```

**`org_id` must never come from the request body or URL path for scoping decisions.** It must always come from the validated JWT claim via `CurrentUser.org_id`. A client cannot forge their `org_id` by passing it as a parameter.

### 5.3 Layer 3 — End-to-end tests (Playwright)

The Playwright e2e test suite must contain a dedicated tenant isolation suite that:

- Creates two separate organisations (org A and org B) with synthetic seed data.
- Authenticates as a `client` user in org A.
- Attempts to fetch the project list, quality snapshots, risk alerts, and confidence scores for org B via direct API calls (using org B's known project UUIDs).
- Asserts that every response returns either an empty array or a `404` — never org B's actual data.
- Repeats for `delivery_manager` attempting to access org B's data (should also be blocked).
- Repeats for a guessed URL with a fabricated `org_id` UUID.

This test suite is mandatory. It must run in CI on every PR and must pass before any production deployment. It cannot be skipped or marked `xfail`.

### 5.4 Tenant isolation for AI agents

Agents must not receive cross-tenant data in their prompts. Before any LLM call, the agent service must:

1. Accept only the requesting user's `org_id` and `role` — never a raw cross-org query.
2. Retrieve evidence rows through the standard RLS-filtered database connection, not through the `service_role` connection.
3. Embed only the retrieved, already-scoped evidence in the LLM prompt.
4. Never construct a prompt that concatenates evidence from multiple organisations.

The LLM has no knowledge of tenant boundaries. Tenant isolation for AI output is a pre-LLM-call responsibility of the backend service layer.

## 6. Data Security

### 6.1 Encryption at rest

Supabase (managed PostgreSQL) encrypts data at rest by default using AES-256. This covers all database tables, including append-only audit tables (`agent_queries`, `communication_evidence_links`). No additional application-level encryption is required for Phase 1, given that Supabase manages this transparently.

For any future self-hosted PostgreSQL deployment, encryption at rest must be explicitly configured and verified before client data is stored.

### 6.2 Encryption in transit

All network communication uses TLS 1.2 or higher. No plaintext HTTP connections are permitted in `staging` or `prod` for any layer:

- Browser → Vercel frontend: HTTPS enforced by Vercel (HSTS header must be configured).
- Vercel frontend → FastAPI backend: HTTPS. The backend URL must be HTTPS-only; no HTTP fallback.
- FastAPI backend → Supabase (asyncpg): TLS connection enforced by Supabase. The `DATABASE_URL` must include SSL mode parameters.
- FastAPI backend → LLM provider API: HTTPS (all major LLM providers enforce this).
- FastAPI backend → Email delivery service: HTTPS.

### 6.3 Data classification

| Classification | Examples | Controls |
|---|---|---|
| **Client operational data** | Project throughput, quality metrics, milestones, risk alerts, agent narratives, client comms | RLS on every table; `org_id` on every row; audit trail; encrypted at rest |
| **Internal platform config** | `metric_configurations`, `organisations`, `users` | `super_admin` write access only; RLS; encrypted at rest |
| **Authentication credentials** | JWT tokens, refresh tokens | `httpOnly` cookie; never logged; Supabase Auth manages storage |
| **Infrastructure secrets** | `SUPABASE_SERVICE_ROLE_KEY`, `LLM_API_KEY`, `EMAIL_API_KEY` | CI/CD secrets store only; never in code, logs, or responses |
| **Synthetic test data** | Seed data in `supabase/seed/` | Never contains real client identifiers; never pushed to production |

### 6.4 Data minimisation

The platform stores only what is needed to serve its features. Specific rules:

- The `agent_queries` table stores the full query text and answer text for audit purposes, but must not store raw LLM prompt templates containing system instructions or cross-tenant context.
- `client_communications.body_draft` stores the AI draft. If the draft is rejected by the DM, the original draft is preserved for audit but the rejected content is not sent.
- Annotator PII (full names) is stored in the `annotators` table but is not included in client-facing API responses. Clients see team-level metrics, not individual annotator names.

## 7. AI Governance & Output Controls

This section governs all AI-generated content produced by the three Phase 1 agents (Delivery Performance, Quality Intelligence, Client Interaction). The controls here exist because AI outputs reach regulated-domain clients and inform consequential business decisions.

### 7.1 Evidence-link mandate

Every AI-generated output that is stored in the database must have at least one corresponding evidence-link row created in the same database transaction:

| Output table | Evidence table | Enforcement |
|---|---|---|
| `agent_queries` | `agent_query_evidence_links` | Transaction must include ≥1 evidence link; abort if 0 |
| `client_communications` | `communication_evidence_links` | Transaction must include ≥1 evidence link; abort if 0 |

An output without evidence links is not a valid output. The FastAPI service layer must check evidence link count inside the transaction and roll back if zero.

The evidence-link tables record: `source_table` (e.g. `throughput_snapshots`), `source_row_id` (the UUID of the specific row), and an optional `description`. This means every AI insight is auditable: a human or auditor can trace exactly which data points drove any AI-generated narrative.

### 7.2 Communication approval gate — state machine

`client_communications.status` follows a strict state machine. Only the transitions listed below are permitted. The service layer must enforce this; it cannot be bypassed by direct database writes.

```
draft ──► in_review ──► approved ──► sent
   │                       │
   └──────────────────────►└──► rejected (→ draft for revision)

FORBIDDEN transitions:
  draft → sent        (skips review and approval)
  in_review → sent    (skips explicit approval)
  rejected → sent     (rejected comms must not be sent)
```

Additional constraints on the `sent` transition:

- `approved_by` must reference a user with `role = 'delivery_manager'` or `role = 'super_admin'`. The service layer must verify this; the database schema cannot enforce it by FK type alone.
- `approved_at` must be set to the current timestamp.
- `body_approved` must be set (the DM may edit the draft before approving).
- The transition to `sent` must trigger the email delivery service call. If the email call fails, the status must roll back to `approved` — not remain as `sent` with an unsent email.

### 7.3 Permission-aware LLM prompting

When constructing an LLM prompt for an agent query or communication draft, the backend must:

1. Retrieve evidence rows using the requesting user's RLS-filtered connection — not `service_role`.
2. Pass only those filtered rows as context to the LLM.
3. Include a system instruction stating the scope of the authorised data (e.g. "You have access to delivery data for organisation [name]. Do not make claims about other organisations.").
4. Never include `org_id` UUIDs, internal row IDs, or infrastructure details in the prompt — these are not useful to the LLM and would leak internal schema information.

### 7.4 Model pinning

The `LLM_MODEL` environment variable must be a pinned model identifier (e.g. `claude-sonnet-4-6`), never a floating alias like `latest` or `claude-3-sonnet`. If the LLM provider silently updates what a floating alias resolves to, agent output characteristics can change unpredictably in production. Model changes must be deliberate, tested in staging, and explicitly deployed.

### 7.5 Hallucination prevention controls

The following architectural controls reduce (but cannot eliminate) the risk of AI hallucination reaching clients:

- **Evidence-link mandate** (§7.1): every output links to real data rows; the LLM cannot fabricate data that isn't in its context window.
- **Human approval gate** (§7.2): every client-facing communication is reviewed by a human DM before sending.
- **Scoped retrieval** (§7.3): the LLM only sees evidence retrieved from the relevant org's data; it cannot fabricate cross-org information it was never given.
- **No hallucination of project status**: explicitly stated in the Client Interaction Agent BRD. If a query cannot be answered from the available evidence, the agent must return a "Insufficient evidence to answer" response (`HTTP 422`), not a fabricated answer.

## 8. Secrets Management

### 8.1 Storage by environment

| Environment | Storage location | Access |
|---|---|---|
| `dev` | `.env` file (git-ignored, local machine only) | Developer only |
| `staging` | GitHub Actions secrets (staging prefix) | CI/CD pipeline only |
| `prod` | GitHub Actions secrets (production prefix) + hosting provider secret injection | CI/CD pipeline; named break-glass access only |

`.env.example` is committed to the repository with placeholder values. It is the canonical inventory of all required secrets. **No real secret value ever appears in `.env.example`, any committed file, any log line, or any API response body.**

### 8.2 Secret inventory

| Secret | Used by | Never in |
|---|---|---|
| `SUPABASE_URL` | Frontend (VITE_ prefix), backend | N/A — public project URL |
| `SUPABASE_ANON_KEY` | Frontend (VITE_ prefix) | Backend service-role operations |
| `SUPABASE_SERVICE_ROLE_KEY` | Backend service account only | Frontend bundle, Vite build, any log, any response |
| `DATABASE_URL` | Backend (asyncpg DSN) | Frontend, logs |
| `SECRET_KEY` | Backend (internal signing) | Frontend, logs, responses |
| `LLM_API_KEY` | Backend agent services | Frontend, logs, responses |
| `LLM_BASE_URL` | Backend agent services | Frontend |
| `LLM_MODEL` | Backend agent services | No restriction — not sensitive |
| `EMBEDDING_API_KEY` | Backend RAG services | Frontend, logs |
| `EMAIL_API_KEY` | Backend notification service | Frontend, logs |
| `VERCEL_TOKEN` | GitHub Actions CI/CD | Any application code |
| `DOCKER_REGISTRY_URL` | GitHub Actions CI/CD | Any application code |

### 8.3 Operational rules

- **`SUPABASE_SERVICE_ROLE_KEY` bypasses RLS entirely.** It must be treated with the same care as a root database password. It is used only in the FastAPI service-account connection for ingestion writes and admin operations — never for serving end-user requests.
- **Rotation protocol:** any secret suspected of exposure must be rotated immediately. Rotation of `SUPABASE_SERVICE_ROLE_KEY` or `LLM_API_KEY` requires a backend redeploy. Rotation of `SUPABASE_ANON_KEY` requires a frontend rebuild and redeploy.
- **No secrets in Docker images.** The `Dockerfile` must not `COPY` any `.env` file. All secrets are injected at container runtime by the hosting platform. Verify with `docker inspect <image>` — no env vars containing keys should appear in the image metadata.
- **No secrets in CI/CD logs.** GitHub Actions masks secrets registered in the secrets store, but structured log output from the backend must also never print secret values. FastAPI exception handlers must strip sensitive fields from error output in `staging` and `prod`.
- **`LLM_MODEL` is pinned, not sensitive,** but must still not be changed silently. Any change to `LLM_MODEL` in production requires a deliberate deployment with a PR that documents the reason.

## 9. Transport Security

### 9.1 TLS requirements

| Connection | Requirement |
|---|---|
| Browser ↔ Vercel (frontend) | HTTPS; Vercel enforces TLS by default; HSTS header must be configured with `max-age` ≥ 1 year |
| Frontend ↔ FastAPI backend | HTTPS only; backend URL must be HTTPS; no HTTP fallback; redirect HTTP → HTTPS at the hosting provider level |
| FastAPI ↔ Supabase (asyncpg) | TLS enforced; `sslmode=require` in `DATABASE_URL` |
| FastAPI ↔ LLM provider | HTTPS (all major providers enforce this by default) |
| FastAPI ↔ Email provider | HTTPS |

No plaintext HTTP connections are permitted between any two components in `staging` or `prod`. In `dev`, `localhost` connections (loopback) are acceptable without TLS.

### 9.2 CORS policy

The FastAPI backend allows cross-origin requests only from explicitly listed origins in `ALLOWED_ORIGINS`. This list is environment-specific:

| Environment | Permitted origins |
|---|---|
| `dev` | `http://localhost:5173` |
| `staging` | Vercel staging preview URL (e.g. `https://ops-tower-staging.vercel.app`) |
| `prod` | Production domain only (e.g. `https://tower.boreksolutions.de`) |

Wildcard `*` origins are never permitted in `staging` or `prod`. The CORS middleware must reject preflight and actual requests from unlisted origins with `403`.

### 9.3 Security headers

The following HTTP response headers must be set on all frontend responses (configured at the Vercel/hosting level, not in application code):

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

The `Content-Security-Policy` above is a starting baseline. It must be tightened (specifically `script-src`) once the exact set of third-party scripts (Supabase, analytics if any) is confirmed.

## 10. Application Security Controls

### 10.1 Input validation

FastAPI with Pydantic v2 performs automatic request body validation. All agent query inputs, communication bodies, and ingestion payloads are validated against Pydantic schemas before reaching service logic. Validation errors return `422 Unprocessable Entity` with structured error details — never a raw Python exception traceback.

Specific rules:

- All UUID fields in path parameters must be validated as `UUID` type by FastAPI/Pydantic, not accepted as raw strings.
- Free-text fields (e.g. `query_text` in `agent_queries`, `body_draft` in `client_communications`) must have a maximum length enforced at the Pydantic schema level before the value is passed to the LLM. Unbounded text input to an LLM represents both a cost risk and a prompt-injection surface.
- The `ALLOWED_ORIGINS` list is parsed from a comma-separated environment variable. The parsing must strip whitespace and validate each entry as a valid URL before the CORS middleware uses it.

### 10.2 Rate limiting
The following endpoints are highest-priority for rate limiting before production:

- `POST /api/v1/agents/query` — LLM calls are expensive; unbounded queries by a single user could exhaust API quota and inflate cost.
- `POST /api/v1/auth/login` (Supabase Auth endpoint) — brute-force protection.
- `POST /api/v1/internal/ingest/*` — ingestion endpoints should be rate-limited and network-restricted.

### 10.3 Error response sanitisation

In `staging` and `prod`, FastAPI exception handlers must return sanitised error responses:

```python
# backend/app/core/exception_handlers.py (planned)
@app.exception_handler(Exception)
async def generic_exception_handler(request: Request, exc: Exception):
    if settings.environment == "dev":
        # Full traceback acceptable in dev
        raise exc
    # In staging/prod: log internally, return generic message
    logger.error("Unhandled exception", exc_info=exc, request_id=request.state.request_id)
    return JSONResponse(
        status_code=500,
        content={"detail": "An unexpected error occurred. Reference: " + request.state.request_id}
    )
```

Error responses must never include: stack traces, internal table names, column names, `org_id` or `user_id` values, SQL query text, or secret values of any kind.

### 10.4 SQL injection prevention

All database queries use SQLAlchemy 2.x with parameterised queries (ORM or `text()` with bound parameters). Raw string interpolation into SQL is never acceptable. This is enforced by linting (`ruff`) and code review — any `f"SELECT ... {variable}"` pattern in a database query must be treated as a critical bug.

### 10.5 Dependency security

All dependencies are pinned to exact versions in `pyproject.toml` (backend) and `package.json` (frontend). No `*`, `>=`, or `latest` version specifiers are permitted for production dependencies. The CI pipeline must include a dependency vulnerability scan (e.g. `pip audit` for Python, `pnpm audit` for Node) as a blocking check on every PR.

### 10.6 Internal ingestion endpoint protection

The internal routes (`POST /api/v1/internal/ingest/*`) are used by the backend ingestion service to insert operational data. These routes:

- Must not be publicly accessible on the internet in `staging` or `prod`.
- Must be network-restricted (firewall rule or private network) or protected by mutual TLS (mTLS) or a separate API key distinct from the user-facing JWT.
- Must use the `service_role` Supabase connection, not a user JWT.

See `08. Backend Architecture.md` §8.9 — "Internal routes are not exposed to the public internet in production — network-restricted or mTLS (confirm in `13. Security & Compliance.md`)." **This document confirms: network restriction is the minimum requirement; mTLS is preferred.**

## 11. Audit Trails & Data Lineage

Audit trails are explicitly required by the source material as a non-optional governance feature. The phrase "who did what, when — fully traceable" is taken from the regulatory-grade quality layer description. The following tables collectively constitute the platform's audit trail.

### 11.1 Audit tables

| Table | What it records | Mutability |
|---|---|---|
| `agent_queries` | Every NL query, the answer, the model used, latency, requesting user | Append-only — no UPDATE, no DELETE |
| `agent_query_evidence_links` | Which specific source rows backed each agent answer | Append-only |
| `client_communications` | Full lifecycle of every AI draft: creation, review, approval/rejection, send | All status transitions preserved; `body_draft` never deleted even after rejection |
| `communication_evidence_links` | Which source rows backed each communication draft | Append-only |
| `risk_alerts` | Every risk event, its tier, narrative, contributing causes, resolution | Soft delete only (`deleted_at`); resolution recorded with `resolved_by` and `resolved_at` |
| `throughput_snapshots` | Daily operational data — one row per project per day | Append-only — immutable historical record |
| `quality_snapshots` | Weekly quality data per project/team | Append-only — immutable historical record |

### 11.2 What every audit event must capture

For any write operation involving user-initiated actions (approving a communication, acknowledging an alert, submitting an agent query), the audit record must capture:

- `user_id` — who performed the action
- `org_id` — which tenant context
- `created_at` — exact UTC timestamp
- The full content of the action (e.g. `body_approved` on communications, `query_text` + `answer_text` on agent queries)
- For AI outputs: the `model_used` and at least one evidence link

### 11.3 Audit trail immutability

Append-only tables must be protected from accidental or malicious modification:

- No `UPDATE` or `DELETE` statements are permitted on append-only tables from application code. RLS policies must enforce `FOR INSERT` only for end-user connections on these tables.
- The `service_role` connection used for agent writes has access, but the service layer code must never issue `UPDATE` or `DELETE` on audit tables.
- Database migration files must never include `DELETE FROM` or `TRUNCATE` on audit tables for any environment other than `dev` (where `supabase db reset` is used).

### 11.4 Audit export

For regulated-domain clients (particularly Life Sciences and Finance), audit trail data must be exportable on request. The platform must provide:

- A `GET /api/v1/admin/audit/agent-queries` endpoint (accessible to `delivery_manager` and `super_admin`) that returns paginated agent query history for an org.
- A `GET /api/v1/admin/audit/communications` endpoint returning full communication lifecycle history including drafts, approvals, and rejections.

Export format: JSON (Phase 1). CSV export is a Phase 2 concern.

## 12. Compliance by Domain

The platform serves three regulated domains. This section states the specific compliance implications of each. Formal certification (SOC 2, ISO 27001, HIPAA BAA, etc.) is not in scope for Phase 1 but the architecture is designed to support it.

### 12.1 Life Sciences & Medical AI

**Regulatory context:** EU MDR/IVDR (if clients are EU-based medical device manufacturers); FDA 21 CFR Part 11 (US electronic records); HIPAA (US clients handling protected health information).

**Platform obligations:**

- Complete, immutable audit trails of all AI outputs and human review actions — provided by `agent_queries`, `communication_evidence_links`, and the `client_communications` lifecycle (§11).
- No AI output presented without evidence of which data it was derived from — provided by the evidence-link mandate (§7.1).
- Human review of all AI-drafted client communications before sending — provided by the approval gate (§7.2).
- Ability to export audit records on regulatory request — provided by audit export endpoints (§11.4).

### 12.2 Finance, Insurance & Regulated Document Workflows

**Regulatory context:** EU GDPR; AML/KYC obligations; DORA (Digital Operational Resilience Act for EU financial entities); MiFID II (if applicable); national financial regulator requirements.

**Platform obligations:**

- GDPR: personal data (annotator names, client contact details) must be processed lawfully and stored only as long as necessary.
- All AI-generated communications in the finance domain must be reviewed by a human before sending (§7.2 satisfies this).
- Audit trails must be available for regulatory inspection (§11 satisfies this).
- DORA: operational resilience requirements may mandate formal incident response procedures and RTO/RPO targets.  

### 12.3 Logistics & Supply Chain

**Regulatory context:** Lower regulatory burden than Life Sciences or Finance in Phase 1. Standard data protection obligations (GDPR for EU clients).

**Platform obligations:**

- GDPR data handling as above.
- Operational data quality and auditability are contractual rather than regulatory requirements in this domain.
- No additional platform controls beyond the standard security posture are required for Logistics in Phase 1.

### 12.4 GDPR baseline (applies to all EU clients)

| Requirement | Implementation |
|---|---|
| Lawful basis for processing | Established by client contract (Data Processing Agreement) — legal obligation, not a platform control |
| Data minimisation | Platform stores only data needed for its features (§6.4) |
| Purpose limitation | Data used only for operational intelligence for the contracting client |
| Storage limitation | Retention policy |
| Data subject access rights | No self-service access rights portal |

## 13. Data Residency

**Requirement [CONFIRMED]:** The platform's data fabric must be "EU & client-region compliant" with full tenant isolation between clients.

### 13.1 Supabase project region

The production Supabase project must be provisioned in an EU region. Available EU regions at the time of writing: Frankfurt (`eu-central-1`) and Amsterdam (`eu-west-1`).

**This decision is irreversible after project creation.** Supabase does not support project region migration. The region must be confirmed and documented before the production project is created. Confirm with the compliance officer which EU region is appropriate for the pilot client's contractual requirements.

### 13.2 Multi-region considerations

For Phase 1, a single EU-region Supabase project serves all clients. If a future client has contractual requirements for data to remain in a specific non-EU region (e.g. US-only for HIPAA compliance), a separate Supabase project in that region is required. The application layer supports multi-project configurations by having per-tenant connection strings — this is a Phase 2+ concern.

### 13.3 Backend hosting region

The FastAPI backend container must be deployed in the same or a geographically close region to the Supabase project to minimise database round-trip latency and to avoid unnecessary cross-region data transfer.

### 13.4 LLM provider data residency

LLM API calls from the backend will transmit excerpts of client operational data (evidence rows included in the prompt) to the LLM provider's servers. This has data residency implications.

- The LLM provider must have an EU data processing agreement or be confirmed to process data within the EU if clients require EU data residency for all processing.
- Data residency for LLM prompt processing must be evaluated as part of the vendor selection decision.
- Mitigation if a US-based LLM provider is used: include only aggregated, anonymised, or non-PII operational metrics in prompts (e.g. throughput numbers, accuracy percentages) rather than names or identifiable project details. This requires careful prompt engineering.

## 14. Security Testing Requirements

Security testing is not optional and must be integrated into the normal development lifecycle, not treated as a pre-launch activity.

### 14.1 Required tests in the CI pipeline (every PR)

| Test | Tool | What it verifies |
|---|---|---|
| Dependency vulnerability scan | `pip audit` (Python) + `pnpm audit` (Node) | No known CVEs in pinned dependencies |
| Static analysis | `ruff` + `mypy` | No SQL string interpolation patterns; type safety that catches missing auth checks |
| RLS unit tests | `pytest` with per-role DB connections | Each role can only access its permitted rows |
| Auth middleware tests | `pytest` | 401 on missing JWT; 403 on wrong role; 404 (not 403) on cross-tenant resource |
| Evidence-link transaction tests | `pytest` | Agent output INSERT with zero evidence links rolls back |
| Communication state machine tests | `pytest` | Forbidden status transitions return 409; `sent` requires `approved_by` DM |

### 14.2 Required e2e tests in CI (every PR)

| Test | Tool | What it verifies |
|---|---|---|
| Tenant isolation suite | Playwright | Org A user cannot read org B's data via any API route |
| Role-scope suite | Playwright | Each of the four roles sees exactly what the permission matrix allows |
| Communication approval gate | Playwright | Attempting to send an unapproved communication fails |
| Evidence-link presence | Playwright (API assertions) | Every agent query response has at least one evidence link |

### 14.3 Pre-production security review

Before the first production deployment that will handle real client data, a human security review must be completed covering:

- All RLS policies — reviewed by a developer who did not write them.
- CORS configuration — confirm `ALLOWED_ORIGINS` contains only the correct domain.
- Secrets inventory — confirm `SUPABASE_SERVICE_ROLE_KEY` does not appear in any frontend asset, log, or response.
- Authentication flow — trace a complete login-to-data-access flow and confirm `org_id` is derived from JWT, not request input.
- Communication approval gate — trace the full draft-to-sent lifecycle and confirm no shortcut path exists.

### 14.4 Periodic security activities (Phase 2+)

The following are not required for Phase 1 MVP but should be planned for once real client data is in production:

- Penetration test by an external party (annually or after major architectural changes).
- Formal OWASP Top 10 review of the API surface.
- Dependency update review (monthly at minimum — do not let pinned versions grow stale).

## 15. Incident Response

### 15.1 Security incident definition

A security incident for this platform is any event that:

- Results or may result in unauthorised access to one client's data by another client or by an unauthorised internal user.
- Involves the exposure, logging, or transmission of a secret key (`SUPABASE_SERVICE_ROLE_KEY`, `LLM_API_KEY`, etc.).
- Results in an AI-drafted client communication being sent without Delivery Manager approval.
- Involves a suspected or confirmed injection of cross-tenant data into an LLM prompt.
- Results in a data breach notifiable under GDPR or applicable regulations.

### 15.2 Immediate response steps

When a security incident is suspected or confirmed:

1. **Contain.** If a secret has been exposed, rotate it immediately. If a cross-tenant data leak is suspected, temporarily disable the affected route or user account while investigating.
2. **Investigate.** Query the audit tables (`agent_queries`, `communication_evidence_links`, `risk_alerts`) to determine the scope. Examine backend structured logs (§13 of `14. Deployment & Environments.md`).
3. **Notify.** Inform BSG leadership and the affected client(s) as soon as the scope is understood. GDPR requires notification to the relevant supervisory authority within 72 hours of becoming aware of a personal data breach.
4. **Remediate.** Fix the root cause. Deploy the fix through the standard CI/CD pipeline (do not apply hot patches directly to the production database).
5. **Post-incident review.** Document root cause, timeline, impact, and preventive actions. Update this document if a control was found to be missing or insufficient.

### 15.3 Key contacts

- Primary on-call engineer (backend access, ability to rotate secrets and roll back deployments).
- BSG leadership contact for client notification.
- Legal/compliance contact for GDPR breach notification.

## 16. Developer Security Rules

These rules apply to every developer and autonomous AI coding agent working on the Operations Tower. Violations are treated as bugs, not style preferences.

**Authentication and authorisation:**
- Every new route handler must declare a `require_role(...)` dependency before any database call. No route is unauthenticated by default except `/health`.
- `org_id` in queries must always come from `CurrentUser.org_id` (JWT claim), never from request body or path parameters.
- Never use the `service_role` connection for serving end-user requests. `service_role` is only for ingestion writes and admin operations in the backend service account.

**Database:**
- Never write a raw SQL query with string interpolation. All queries use SQLAlchemy parameterised statements.
- Never add a new table without an RLS policy in the same migration. Never deploy a table to `staging` without its RLS policy applied.
- Never add `UPDATE` or `DELETE` on append-only audit tables. If you think you need to update an audit table, re-read the requirements.

**AI and agent outputs:**
- Never commit or merge an agent service function that calls the LLM without first inserting at least one evidence link in the same transaction.
- Never construct an LLM prompt that includes data retrieved via `service_role`. Evidence context must come from RLS-filtered queries.
- Never allow a `client_communications` row to transition to `sent` without verifying `approved_by` is a `delivery_manager`.

**Secrets:**
- Never commit any secret value to the repository — not even in a comment, not even in a test file.
- Never print or log a secret value. Use typed structured log fields only.
- Never use `SUPABASE_SERVICE_ROLE_KEY` in frontend code or Vite build configurations.
- Always use `VITE_` prefix for environment variables that need to be available in the frontend bundle — and only non-sensitive values (URL, anon key) should have that prefix.

**Dependencies and versions:**
- Never use `*`, `>=`, or `latest` for production dependencies. Pin exact versions.
- Never add a new external service or library without first updating `05. Tech Stack.md`. Agents must flag new dependencies before introducing them.

**Testing:**
- Never mark a tenant isolation test `xfail` or `skip`. If it is failing, fix the isolation, not the test.
- Every new route must have at least one test for an unauthorised role receiving `403` and one test for a cross-tenant resource receiving `404`.
