# AI & RAG Architecture

> Architecture specification for the Operations Tower AI layer.
> This document is based on `BSG_Ops_Intelligence_Agent_Optimized v1.0.docx` and aligns with the API, database, and tech stack documents. It explains how the AI agents, retrieval layer, evidence trail, governance controls, and role-aware responses should work.

---

## 1. Purpose

The Operations Tower is an AI-powered operational intelligence layer for BSG's data operations business. The AI layer does not replace annotators, reviewers, SMEs, project managers, or delivery managers. Its purpose is to augment operational decision-making by turning delivery, quality, workforce, governance, and client reporting data into evidence-backed intelligence.

The AI architecture must support:

- Delivery risk prediction.
- Quality drift monitoring.
- Client-facing status summaries.
- Natural-language operational Q&A.
- Evidence-backed answers.
- Role-based and tenant-safe responses.
- Human review for client-facing AI communications.
- Auditability for every AI-generated insight.

The central rule is simple: no AI output should be treated as valid unless it can be traced back to operational evidence.

---

## 2. Source Principles

The source Word document defines these core principles:

| Principle | Meaning for the AI Architecture |
|---|---|
| Human-Centric AI | AI supports operational teams. It does not replace human annotation, SME review, or delivery ownership. |
| Single Source of Truth | AI must reason from governed operational data, not disconnected spreadsheets or isolated notes. |
| Evidence-Backed AI | Every generated insight must link to the data used to produce it. |
| Role-Based Intelligence | Different roles receive different answers based on permissions. |
| Operational Explainability | Answers must explain what happened, why it happened, impact, recommended action, and confidence. |
| Security & Governance First | RBAC, tenant isolation, audit logging, human validation, and data lineage are mandatory. |

These principles should be enforced in code, not only documented.

## 3. Scope

### 3.1 Phase 1 AI Scope

The Phase 1 MVP focuses on three agents:

| Agent | MVP Status | Purpose |
|---|---|---|
| Delivery Performance Agent | In scope | Monitors throughput, milestones, delivery confidence, bottlenecks, and schedule risk. |
| Quality Intelligence Agent | In scope | Monitors gold-set accuracy, inter-annotator agreement, rework rate, quality drift, and error categories. |
| Client Interaction Agent | In scope | Generates client-ready summaries, answers project-status questions, explains risks, and supports delivery-confidence reporting. |

### 3.2 Later-Phase Agents

The source document also defines these agents, but they are not core Phase 1 unless the project scope changes:

| Agent | Phase | Purpose |
|---|---|---|
| Workforce & Capability Agent | Phase 2 | Skill-to-project matching, SME coverage, utilization, certification tracking, and resource planning. |

The MVP UI must not expose Workforce & Capability, Project Governance, Operational Knowledge, Go-Live Readiness, Margin Analytics, ROI Analytics, Churn Prediction, autonomous multi-agent orchestration, or other portfolio intelligence features.

## 4. Important RAG Terminology

The term `RAG` can mean two different things in this project context:

| Term | Meaning |
|---|---|
| Retrieval-Augmented Generation | AI pattern where the model retrieves trusted source data before generating an answer. This is the meaning used in this document. |
| Red / Amber / Green | Governance status labels used in project management contexts. |

This document uses `RAG` only to mean Retrieval-Augmented Generation.

## 5. High-Level Architecture

The source document defines a three-layer AI architecture:

```text
Layer 3: Conversational AI Interaction Layer
  - Natural-language Q&A
  - AI summaries
  - Client narratives
  - Action recommendations
  - Evidence citation

Layer 2: Intelligence & Analytics Layer
  - Forecasting
  - Risk scoring
  - Trend detection
  - Anomaly detection
  - Root-cause analysis
  - Confidence scoring

Layer 1: Unified Data Foundation
  - Normalized delivery data
  - Quality metrics
  - Resource and team data
  - SOPs and project documents
  - Governance artifacts
  - Audit trail
```

The FastAPI backend sits between these layers and the frontend. The backend controls authentication, tenant scoping, agent orchestration, evidence persistence, and response shaping.

## 6. Layer 1: Unified Data Foundation

Layer 1 creates the operational single source of truth. It standardizes raw operational signals so agents can reason consistently.

### 6.1 Structured Data Sources

The source document names these structured inputs:

| Source | Used By | Example Usage |
|---|---|---|
| Delivery Tracker | Delivery Performance, Client Interaction | Current project status, milestone progress, schedule risk |
| Milestone Plan | Delivery Performance, Client Interaction | Planned dates, actual dates, confidence scoring |
| Throughput Logs | Delivery Performance | Forecast vs actual throughput, rolling trends |
| Resource Allocation | Delivery Performance, Workforce | Utilization and workload balance |
| QA/Rework Sheets | Quality Intelligence, Delivery Performance | Rework rate, quality impact on delivery |
| Risk Register | Delivery Performance, Client Interaction | Open risks, mitigation progress |
| Backlog Queue | Delivery Performance | Workload pressure and schedule exposure |
| SME Coverage | Client Interaction, Workforce | Readiness and capability confidence |

In the current database model, these sources map primarily to:

- `projects`
- `milestones`
- `throughput_snapshots`
- `teams`
- `annotators`
- `quality_snapshots`
- `quality_error_entries`
- `risk_alerts`
- `bottlenecks`
- `delivery_confidence_scores`
- `client_communications`
- `agent_queries`
- `agent_query_evidence_links`
- `communication_evidence_links`

### 6.2 Unstructured Data Sources

The source document also names these unstructured inputs:

| Source | Used By | Example Usage |
|---|---|---|
| SOPs | Operational Knowledge, Delivery, Client Interaction | Explain process, identify guideline ambiguity |
| Project Charters | Delivery, Governance, Client Interaction | Scope and success criteria |
| Escalation Notes | Delivery, Client Interaction | Risk history and mitigation explanation |
| PM Notes | Delivery | Context around blockers and operational changes |
| Client SOPs | Client Interaction | Client-specific process expectations |
| Training Documents | Client Interaction, Quality | Readiness and training-completion reasoning |

These sources require a document ingestion pipeline before they can be used safely in RAG. The MVP can begin with structured data RAG first and add unstructured document retrieval once governance and document storage are defined.

### 6.3 Semantic KPI Layer

The source document explicitly calls for a KPI semantic layer. This layer defines consistent business meanings for metrics so agents do not interpret raw fields differently.

Examples:

| Semantic Metric | Source Fields | Meaning |
|---|---|---|
| Delivery confidence | `delivery_confidence_scores.score_pct` | Likelihood a milestone remains on track |
| Throughput variance | `throughput_snapshots.units_completed` vs `units_forecast` | Whether delivery is above or below plan |
| Rolling throughput | `throughput_snapshots.rolling_7day_units` | Short-term delivery velocity |
| Gold-set accuracy | `quality_snapshots.gold_set_accuracy_pct` | Accuracy against trusted reference set |
| IAA | `quality_snapshots.iaa_krippendorff_alpha` | Reviewer consistency |
| Rework rate | `quality_snapshots.rework_rate_pct` | Percentage of work that needs correction |
| Quality drift | `quality_snapshots.has_drift_alert` | Whether quality trends crossed threshold |
| Risk tier | `risk_alerts.risk_tier` | Severity level of operational issue |

Agents must use semantic metric definitions rather than raw field names wherever possible.

## 7. Layer 2: Intelligence & Analytics Layer

Layer 2 converts operational data into predictions, scores, classifications, and recommendations.

### 7.1 Required Capabilities

The source document names these intelligence capabilities:

- Predictive analytics.
- Trend detection.
- Anomaly detection.
- Root-cause analysis.
- Forecasting.
- Risk scoring.
- Pattern recognition.
- Workforce optimization.

For Phase 1, the most important capabilities are delivery forecasting, quality drift detection, root-cause explanation, and client-safe narrative generation.

### 7.2 Delivery Risk Logic

The source document gives an example reasoning pattern:

```text
IF throughput declines
AND rework increases
AND absenteeism spikes
THEN raise risk score,
forecast delay probability,
generate mitigations,
trigger escalation alert.
```

In the current Phase 1 schema, absenteeism is not directly modeled as a time-series table. Until workforce utilization history is defined, delivery risk scoring should use available signals:

- Throughput decline.
- Forecast variance.
- Missed or at-risk milestones.
- Rework rate.
- Quality drift.
- Open bottlenecks.
- Existing risk alerts.
- Review turnaround if captured in future data.

### 7.3 Risk Tiers

Risk outputs use the tiers defined in the database schema:

- `low`
- `medium`
- `high`
- `critical`

Risk tier assignment should be deterministic where possible. The LLM can explain a tier, but the underlying tier should come from a service-side scoring function or explicit threshold logic.

### 7.4 Draft Thresholds

The current draft thresholds are:

| Metric | Draft Threshold |
|---|---:|
| Schedule confidence | `>= 80%` to remain on track |
| Gold-set accuracy | `>= 95%` |
| IAA | `>= 0.85` |
| Rework rate | `<= 5%` |

These values are provisional. The backend should not hardcode them permanently. Use configuration where possible.

### 7.5 Analytics vs LLM Responsibilities

The system should separate deterministic analytics from language generation.

| Responsibility | Best Owner |
|---|---|
| Calculating throughput variance | Backend analytics service |
| Calculating delivery confidence | Backend analytics service |
| Detecting threshold breaches | Backend analytics service |
| Assigning risk tier | Backend analytics service |
| Retrieving evidence rows | Backend retrieval service |
| Summarizing evidence in plain language | LLM |
| Explaining contributing causes | LLM with structured evidence |
| Drafting client narrative | LLM with strict grounding |
| Applying RBAC and tenant filtering | Backend service and database RLS |

The LLM should not be the only place where business rules are enforced.

## 8. Layer 3: Conversational AI Interaction Layer

Layer 3 allows users to interact with operational intelligence through natural language.

It supports:

- Operational Q&A.
- AI-generated summaries.
- Automated reporting drafts.
- Action recommendations.
- Role-based answers.
- Evidence citation.
- Document generation in later phases.

### 8.1 Example Questions

The source document lists example questions the AI suite should answer:

| Category | Example Questions |
|---|---|
| Delivery & Operations | Which milestones are most at risk this month? Why did throughput decline this week? Which projects require escalation attention? |
| Quality & Governance | Are quality trends affecting delivery timelines? Which error categories are increasing? Which projects require additional QA intervention? |
| Workforce & Capability | Do we have sufficient SME coverage? Which teams are nearing capacity overload? What training gaps may impact delivery? |
| Client & Leadership | Provide an executive-ready project health summary. Summarize delivery risks with mitigation plans. Generate this week's client update. |

Phase 1 should prioritize delivery, quality, and client-facing project health questions.

### 8.2 Response Requirements

Every AI response should include:

- Direct answer.
- Evidence summary.
- Reasoning explanation.
- Confidence level or confidence score where applicable.
- Recommended next action where applicable.
- Evidence links.

Example response shape:

```json
{
  "answer_text": "Batch 2 is the highest-risk milestone this month because throughput is below forecast and rework has increased.",
  "confidence": "high",
  "recommended_actions": [
    "Review rework causes with QA lead.",
    "Temporarily rebalance review capacity toward Batch 2."
  ],
  "evidence_links": [
    {
      "source_table": "throughput_snapshots",
      "source_row_id": "uuid",
      "description": "Latest throughput trend for Batch 2"
    },
    {
      "source_table": "quality_snapshots",
      "source_row_id": "uuid",
      "description": "Current week quality metrics"
    }
  ]
}
```

## 9. Agent Architecture

Each agent is a domain-specific service module, not a completely separate product.

Recommended backend structure:

```text
backend/app/agents/
  delivery_performance/
    service.py
    retrieval.py
    prompts.py
    schemas.py
  quality_intelligence/
    service.py
    retrieval.py
    prompts.py
    schemas.py
  client_interaction/
    service.py
    retrieval.py
    prompts.py
    schemas.py
  common/
    llm_client.py
    evidence.py
    guardrails.py
    prompt_context.py
    retrieval.py
```

### 9.1 Shared Agent Contract

Every agent should follow the same internal flow:

```text
1. Receive request from API route.
2. Load current user context.
3. Validate role and project access.
4. Retrieve structured evidence.
5. Retrieve unstructured evidence if available.
6. Build prompt context from approved evidence only.
7. Call LLM provider.
8. Validate generated answer.
9. Persist answer and evidence links.
10. Return role-safe response.
```

### 9.2 Shared Agent Output

Each agent should return a structured object before it is converted into API response JSON:

```json
{
  "agent_name": "delivery_performance_agent",
  "answer_text": "string",
  "confidence_label": "low | medium | high",
  "risk_tier": "low | medium | high | critical | null",
  "recommended_actions": ["string"],
  "evidence": [
    {
      "source_table": "string",
      "source_row_id": "uuid",
      "description": "string"
    }
  ],
  "model_used": "string",
  "latency_ms": 1200
}
```

The API may expose only a subset of this object depending on role.

- Project health summaries.
- Executive status reports.
- Delivery confidence narratives.
- Risk narratives.
- Client-facing Q&A answers.

Go-Live Readiness reports are Phase 2+ and are not part of the Phase 1 Client Interaction Agent.

### 12.7 Human Approval Gate

The Client Interaction Agent may draft client communications, but it must not send them automatically.

MVP report-generation behavior:

- Weekly Client Summaries and Executive Reports are manually generated by Delivery Managers or Super Admins.
- The AI layer must not schedule report generation in Phase 1.
- No scheduler, cron job, background worker, or job queue is required for MVP report generation.
- Automatic report generation is Phase 2+ and must still create drafts only, never client-published reports.
- MVP report delivery is in-app only. PDF export, DOCX export, and outbound email delivery are Phase 2+.

Required workflow:

```text
1. Agent generates draft.
2. Backend stores draft in client_communications.
3. Backend stores evidence in communication_evidence_links.
4. Delivery Manager reviews and edits the draft.
5. Delivery Manager approves the final body.
6. Communication can be sent or marked sent.
7. Client users can only see sent communications.
```

This is a hard governance rule.

### 12.8 CSAT Handling

CSAT is not generated by AI. It is client-submitted evidence that can be summarized in reports when available.

MVP CSAT model:

- Collected monthly per project.
- Submitted only by client-role users.
- Stored as one row per client user, project, and reporting month.
- Uses a `1` to `5` score scale.
- Uses `reporting_period_month` as the first day of the reporting month, for example `01-06-2026`.

AI-generated reports may reference CSAT only when a matching `client_csat_scores` row exists in evidence. The model must not infer client satisfaction from tone, risk level, or communication history.

## 13. RAG Architecture

RAG is the pattern that allows the AI system to generate grounded answers from approved source data.

### 13.1 RAG Flow

```text
User asks question
  -> API verifies JWT and loads user context
  -> Agent validates role and project access
  -> Retrieval service finds relevant structured rows
  -> Optional document retrieval finds relevant unstructured chunks
  -> Evidence filter removes inaccessible or unsafe data
  -> Prompt builder creates role-safe prompt context
  -> LLM generates answer
  -> Output validator checks grounding and policy rules
  -> Backend stores answer and evidence links
  -> API returns answer with citations
```

### 13.2 Retrieval Sources

Phase 1 retrieval should begin with structured retrieval from PostgreSQL:

| User Intent | Primary Tables |
|---|---|
| Delivery risk | `milestones`, `throughput_snapshots`, `delivery_confidence_scores`, `risk_alerts`, `bottlenecks` |
| Throughput change | `throughput_snapshots`, `projects`, `milestones` |
| Quality drift | `quality_snapshots`, `quality_error_entries`, `risk_alerts` |
| Client summary | `projects`, `milestones`, `throughput_snapshots`, `quality_snapshots`, `risk_alerts`, `delivery_confidence_scores` |
| Communication history | `client_communications`, `communication_evidence_links` |
| Past AI answers | `agent_queries`, `agent_query_evidence_links` |

Unstructured retrieval can be added when document ingestion is ready:

- SOP chunks.
- Project charter chunks.
- Escalation note chunks.
- PM note chunks.
- Training document chunks.

### 13.3 Vector Store

The vector database or embedding store is not specified in the source material.

Candidate options:

| Option | Notes |
|---|---|
| Supabase `pgvector` | Strong candidate because PostgreSQL/Supabase is already selected. Keeps data governance in one platform. |
| External vector database | Possible alternatives include Pinecone, Weaviate, or Qdrant, but none are specified in the source. |
| No vector store for Phase 1 | Acceptable if Phase 1 starts with structured retrieval only. |

Do not add a vector database dependency without project approval.

### 13.4 Embeddings

Embedding provider is not specified.

If unstructured RAG is implemented, the system needs:

- Embedding model.
- Chunking strategy.
- Document metadata schema.
- Tenant and role filters.
- Re-indexing strategy.
- Deletion and retention policy.

Until these are defined, the MVP should treat unstructured RAG as an extension point.

### 13.5 Evidence Link Requirement

Every persisted AI output must store evidence links.

For agent Q&A:

- Store answer in `agent_queries`.
- Store citations in `agent_query_evidence_links`.

For client communications:

- Store draft in `client_communications`.
- Store citations in `communication_evidence_links`.

If no evidence is available, the AI should not fabricate an answer. It should return a safe response such as:

```text
I cannot answer this from the currently available project evidence.
```

## 14. Prompt Architecture

Prompts should be built from structured sections rather than one large freeform string.

### 14.1 Prompt Sections

Recommended prompt structure:

```text
System role:
You are an Operations Tower AI agent for BSG delivery operations.

Agent mission:
<delivery, quality, or client interaction mission>

User role:
<client | delivery_manager | bsg_leadership | super_admin>

Permissions:
<what this user may and may not see>

Question/task:
<user query or draft request>

Evidence:
<retrieved evidence rows and document snippets>

Output requirements:
- Answer only from evidence.
- Cite evidence.
- Explain reasoning.
- Provide confidence.
- Include recommended actions when relevant.
- Do not expose restricted data.
```

### 14.2 Prompt Rules

All prompts must instruct the model:

- Do not invent project status.
- Do not invent milestone dates.
- Do not invent quality metrics.
- Do not expose data from other organisations.
- Do not reveal annotator personal details to client users.
- Do not treat draft communications as sent.
- Do not claim a client communication was approved unless `approved_at` and `approved_by` exist.
- If evidence is insufficient, say so clearly.

### 14.3 Client-Safe Tone

Client Interaction Agent outputs should be:

- Clear.
- Business-friendly.
- Transparent.
- Calm.
- Evidence-backed.
- Free from internal-only blame language.

They should explain risks and mitigations without exposing unnecessary internal operational detail.

## 15. Role-Aware AI Responses

The same question may produce different answers for different roles.

Question:

```text
Why did throughput decline this week?
```

### 15.1 Client Response

Client users should receive:

- Project-level explanation.
- Business impact.
- Mitigation status.
- Confidence level.
- Client-safe evidence.

They should not receive:

- Annotator names.
- Internal staffing commentary.
- Other client comparisons.
- Draft internal notes.

### 15.2 Delivery Manager Response

Delivery Managers may receive:

- Project-level explanation.
- Team-level causes.
- Bottlenecks.
- Quality/rework impact.
- Recommended internal actions.
- Evidence rows within their organisation.

### 15.3 BSG Leadership Response

BSG Leadership may receive:

- Cross-client portfolio summary.
- Trend comparison across organisations.
- Risk concentration.
- Strategic impact.

Leadership access is read-only in MVP unless another document explicitly changes that.

### 15.4 Super Admin Response

Super Admin responses should focus on configuration, system state, and administrative visibility. Super Admin access does not automatically mean client data should be exposed in every workflow. Use least-privilege response shaping where possible.

## 16. Governance and Security

The source document states that governance and security are foundational, not optional.

### 16.1 Required Controls

| Control | Requirement |
|---|---|
| RBAC | Every AI request must be role-aware. |
| Tenant isolation | Every retrieval must be scoped by `org_id` unless explicitly allowed for leadership/admin. |
| Audit logging | Store every AI Q&A in `agent_queries`. |
| Evidence links | Store source row references for every answer or draft. |
| Human-in-the-loop | Client communications require Delivery Manager approval before sending. |
| Data lineage | AI outputs must trace to source tables and rows. |
| Permission-aware responses | The model must not see or reveal unauthorized data. |
| Synthetic/sanitized build data | Local/staging must not use live client production data. |

### 16.2 Data Access Rule

The AI layer must retrieve only what the current user is allowed to see. Do not retrieve broad data and then ask the LLM to ignore sensitive parts.

Correct pattern:

```text
Apply auth -> apply tenant filter -> apply role filter -> retrieve evidence -> build prompt
```

Incorrect pattern:

```text
Retrieve broad data -> put everything in prompt -> ask model not to reveal restricted information
```

### 16.3 Audit Trail

Each AI output should record:

- User ID.
- Organisation ID.
- Project ID, if applicable.
- Agent name.
- Query or generation task.
- Final answer or draft.
- Model used.
- Latency.
- Created timestamp.
- Evidence links.

The current schema supports this through `agent_queries`, `agent_query_evidence_links`, `client_communications`, and `communication_evidence_links`.

## 17. Model Provider

The LLM provider is not specified in the source document.

The tech stack document defines placeholder environment variables:

```text
LLM_API_KEY
LLM_BASE_URL
LLM_MODEL
EMBEDDING_API_KEY
```

### 17.1 Provider Requirements

Any selected provider must support:

- Strong instruction following.
- Reliable JSON or structured outputs.
- Low enough latency for interactive Q&A.
- Enterprise-grade data handling.
- Ability to disable training on submitted data, if using a hosted provider.
- Logging controls.
- Regional/compliance review if client data is used.

### 17.2 Model Abstraction

The backend should use a provider abstraction:

```text
LLMClient
  - generate_text(prompt, options)
  - generate_structured(prompt, schema, options)
  - embed(texts, options)
```

This keeps the application from being tightly coupled to one provider before the decision is formalized.

## 18. Output Validation

The AI layer should validate model outputs before saving or returning them.

### 18.1 Validation Checks

| Check | Requirement |
|---|---|
| Evidence present | At least one evidence link for persisted outputs. |
| No unsupported claims | Claims must map back to evidence rows or provided context. |
| Role-safe content | Output must not include restricted fields for the current role. |
| Valid JSON | Structured outputs must parse successfully. |
| Valid enum values | Risk tier, status, and agent names must match allowed values. |
| No auto-send | Client communication drafts must remain drafts until approved. |

### 18.2 Failure Behavior

If validation fails:

1. Do not persist the output as trusted.
2. Return a safe fallback or retry once with stricter instructions.
3. Log the validation failure without leaking sensitive data.
4. If repeated, return an API error such as `EVIDENCE_REQUIRED` or `LLM_PROVIDER_UNAVAILABLE`.

## 19. Observability

AI observability should track reliability and governance, not only uptime.

Recommended metrics:

| Metric | Purpose |
|---|---|
| Agent query count | Usage volume by agent and role |
| Average latency | User experience and provider performance |
| Evidence coverage rate | Percentage of AI outputs with evidence links |
| No-answer rate | How often evidence is insufficient |
| Validation failure rate | Output quality and guardrail health |
| Communication approval rate | Human acceptance of drafts |
| Communication edit distance | How much humans change AI drafts |
| Risk alert precision | Whether raised alerts prove useful |
| Client query volume | Client adoption and transparency value |

Logs must not include raw sensitive client data unless explicitly approved by governance rules.

## 23. Final Resolved Decisions

| Decision | Resolution |
|---|---|
| Date format | AI examples that include dates use `DD-MM-YYYY`. Timestamp examples use `DD-MM-YYYYTHH:mm:ssZ`. |
| MVP agents | Phase 1 includes only Delivery Performance Agent, Quality Intelligence Agent, and Client Interaction Agent. |
| Phase 2+ visibility | Workforce & Capability, Project Governance, Operational Knowledge, Go-Live Readiness, Margin Analytics, ROI Analytics, Churn Prediction, and portfolio intelligence must not appear in the MVP UI. |
| Go-Live Readiness | Go-Live Readiness reports and readiness assessment are Phase 2+. |
| Report generation | Weekly Client Summaries and Executive Reports are manually generated in MVP. The AI layer must not schedule them automatically. |
| Scheduler | No scheduler, cron, worker, or queue is required for MVP report generation. Automatic generation is Phase 2+. |
| Report export/delivery | MVP report delivery is in-app only. PDF export, DOCX export, and outbound email delivery are Phase 2+. |
| CSAT model | CSAT is monthly per project, submitted by client-role users only, and can be summarized by AI only when evidence exists. |
| CSAT scale | CSAT uses a `1` to `5` score range. |
