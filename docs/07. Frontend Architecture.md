## 1. Overview

The Operations Tower frontend is a **single-page application (SPA)** built with React 18, TypeScript, Vite, and Tailwind CSS. It provides role-scoped dashboards and workflows for four user types: **Client**, **Delivery Manager**, **BSG Leadership**, and **Super Admin**.

Design principles:

1. **Role-first routing** — each role lands on a distinct view with navigation limited to permitted actions.
2. **Server state via TanStack Query** — dashboards poll live operational data; Zustand holds UI-only state.
3. **Evidence visibility** — every AI-generated insight renders linked source citations.
4. **Tenant safety** — the frontend never constructs cross-org queries; it relies on backend + RLS, with Playwright e2e tests verifying isolation (`05. Tech Stack.md` §7).

## 5. Routing and Navigation

Routing library (React Router) is implied by the SPA structure; exact version to be pinned at scaffold. Planned route map by role:

### 5.1 Public routes

| Path | Page | Description |
|---|---|---|
| `/login` | LoginPage | Supabase Auth email/password (or SSO if added later) |
| `/auth/callback` | AuthCallback | OAuth redirect handler if enabled |

### 5.2 Client routes (`app_role = client`)

| Path | Page | Data sources |
|---|---|---|
| `/client` | ClientDashboard | Projects, `metric_configurations` (visible), confidence scores |
| `/client/projects/:id` | ClientProjectDetail | Milestones, throughput chart, quality summary |
| `/client/projects/:id/ask` | ClientAgentQuery | NL query to Client Interaction Agent |
| `/client/communications` | ClientCommunications | `client_communications` where `status = sent` |

### 5.3 Delivery Manager routes (`app_role = delivery_manager`)

| Path | Page | Data sources |
|---|---|---|
| `/dm` | DMDashboard | Portfolio projects, open alerts, bottlenecks |
| `/dm/projects/:id` | DMProjectDetail | Full delivery + quality tabs |
| `/dm/alerts` | AlertInbox | `risk_alerts`, `bottlenecks` with resolve actions |
| `/dm/communications` | CommunicationQueue | Pending drafts for approval |
| `/dm/communications/:id` | CommunicationReview | Draft body + evidence links + approve/reject |
| `/dm/ask` | DMAgentQuery | NL query (Delivery or Quality agents) |

### 5.4 BSG Leadership routes (`app_role = bsg_leadership`)

| Path | Page | Notes |
|---|---|---|
| `/leadership` | LeadershipPortfolio | Cross-org read-only project list |
| `/leadership/projects/:id` | LeadershipProjectDetail | Read-only detail |

Full cross-client analytics (margin, unit economics) depend on Phase 2+ agents — not guaranteed in Phase 1 UI (`01. Project Summary.md` §8 item 7).

### 5.5 Super Admin routes (`app_role = super_admin`)

| Path | Page | Data sources |
|---|---|---|
| `/admin/metrics` | MetricConfiguration | CRUD on `metric_configurations` |

Super Admin does **not** receive a client-data dashboard — platform configuration only (`01. Project Summary.md` §2).

### 5.6 Route guard pattern

```
Request path
     │
     ▼
AuthGuard — session valid?
     │ no → /login
     ▼
RoleGuard — user.role permitted for route prefix?
     │ no → /unauthorized
     ▼
OrgScopeProvider — inject org_id into query keys
     │
     ▼
Render page
```

Role is read from the authenticated user's profile (`users.role` via backend `/me` or Supabase custom claim — confirm at implementation).

## 6. State Management

### 6.1 Zustand (client/UI state)

| Store | Responsibility | Persists? |
|---|---|---|
| `useAuthStore` | User id, role, org_id, token metadata | Session only (prefer httpOnly cookie — see Open Questions) |
| `useUIStore` | Sidebar collapse, active tab, modal open state | Optional localStorage for prefs |
| `useNotificationStore` | Unread count badge | Refreshed from API |

Zustand must **not** duplicate server-fetched project lists, alerts, or snapshots — those belong in TanStack Query.

### 6.2 TanStack Query (server state)

| Query key pattern | Endpoint (see API spec) | Refetch strategy |
|---|---|---|
| `['projects', orgId]` | `GET /projects` | On mount + window focus |
| `['throughput', projectId]` | `GET /projects/:id/throughput` | `refetchInterval: 60_000` (live dashboard) |
| `['quality', projectId, week]` | `GET /projects/:id/quality` | On mount; manual refetch on week change |
| `['alerts', orgId, status]` | `GET /alerts` | Poll 30–60s for DM inbox |
| `['communications', orgId, status]` | `GET /communications` | On mount + after mutation |
| `['metrics', 'config']` | `GET /admin/metrics` | Stale-time 5m (changes infrequently) |

Mutations (approve communication, resolve alert, submit CSAT) use `useMutation` with optimistic updates disabled for governance actions — always wait for server confirmation.

### 6.3 Type alignment

TypeScript interfaces in `src/types/` mirror:

- Pydantic schemas in `backend/app/schemas/`
- PostgreSQL enums from `10. DB Schema.md` §4 (`AppRole`, `ProjectStatus`, `CommunicationStatus`, …)

Shared types may eventually be generated from OpenAPI (`09. API Specification.md`) once published.

## 7. Data Fetching Strategy

### 7.1 API client (`lib/api.ts`)

- Base URL from `VITE_API_BASE_URL`
- Attaches `Authorization: Bearer <access_token>` on every request
- Centralised error handling: 401 → logout redirect; 403 → toast + no retry
- JSON only; no GraphQL

### 7.2 Supabase client (`lib/supabase.ts`)

Used for:

- Authentication (`signInWithPassword`, `signOut`, session listener)
- **Not** for direct PostgREST queries against tenant tables in production UI — all operational reads go through FastAPI so business rules and evidence assembly stay server-side

Direct Supabase reads may be limited to the authenticated user's own `users` row if needed for bootstrap.

### 7.3 Polling vs push

Phase 1 uses **polling** via TanStack Query `refetchInterval` for dashboards. WebSockets or Supabase Realtime are not specified in `05. Tech Stack.md` — do not add without flagging.

## 9. Frontend Workflows

Cross-reference: `06. Technical Workflow.md` for full sequence diagrams.

### 9.1 Login and session bootstrap

1. User submits credentials on `/login`
2. Supabase Auth returns session
3. Frontend stores session (httpOnly cookie preferred — `05. Tech Stack.md` §10)
4. Fetch user profile (`role`, `org_id`)
5. Redirect to role default route (`/client`, `/dm`, `/leadership`, `/admin/metrics`)

### 9.2 Live delivery dashboard (Delivery Manager)

1. `DMDashboard` mounts → parallel queries for projects, open alerts
2. `ThroughputChart` renders last N days from `throughput_snapshots`
3. `ConfidenceScoreBadge` shows latest `delivery_confidence_scores` per milestone
4. Poll continues while tab focused
5. User clicks alert → navigates to detail with `EvidencePanel`

### 9.3 Communication approval

1. `CommunicationQueue` lists drafts (`status IN ('draft', 'in_review')`)
2. User opens `CommunicationReview` → loads draft + `communication_evidence_links`
3. User approves → `PATCH /communications/:id/approve` mutation
4. Optional send step → `PATCH /communications/:id/send`
5. Invalidate communications query; show success toast

Reject path sets `status = rejected` — UI must not offer send until a new draft exists.

### 9.4 Client NL query

1. User types question in `AgentQueryPanel`
2. `POST /agents/query` with `{ agent_name, project_id, query_text }`
3. Loading state; disable submit until response
4. Render `answer_text` with `EvidencePanel` citations
5. No edit of answer client-side — immutable audit log on server

### 9.5 Super Admin metric configuration

1. Load all `metric_configurations`
2. Toggle `is_client_visible`, reorder `display_order`
3. Save mutations → client dashboards pick up changes on next load

## 10. UI Interaction Patterns

| Pattern | Usage |
|---|---|
| **Role-scoped navigation** | Sidebar items hidden, not merely disabled, for unauthorised roles |
| **Threshold badges** | Green/amber/red against draft WIP thresholds (`01. Project Summary.md` §5) |
| **Evidence drill-down** | Click citation → modal with snapshot detail (read-only) |
| **Governance confirmations** | Approve/send actions require explicit confirm dialog |
| **Empty states** | Distinguish "no data yet" vs "no permission" |
| **Error boundaries** | Per-route boundary; agent query failures show retry without losing form input |
| **Responsive layout** | Tailwind breakpoints; dashboards prioritise tablet+ (primary user device TBD) |

Visual styling follows `15. Branding.md` once populated.

## 11. Authentication Integration

| Concern | Planned approach |
|---|---|
| Provider | Supabase Auth (`05. Tech Stack.md` §4) |
| Token | JWT access + refresh |
| Storage | httpOnly cookie (preferred) — confirm vs SDK default localStorage |
| RBAC | Role from `users.role`; routes guarded client-side **and** enforced server-side |
| Tenant scope | All query keys include `orgId`; never accept org_id from URL alone without server validation |

Client-side guards are UX only — security depends on API + RLS (`05. Tech Stack.md` §10 rationale).

## 12. Environment Configuration

Frontend reads Vite-prefixed public variables only:

```
VITE_SUPABASE_URL=
VITE_SUPABASE_ANON_KEY=
VITE_API_BASE_URL=
```

Secrets (`SUPABASE_SERVICE_ROLE_KEY`, `LLM_API_KEY`) never appear in the frontend bundle (`05. Tech Stack.md` §9).

## 15. Phase 1 Frontend Scope Boundaries

**In scope:**

- Role-scoped dashboards for Client and Delivery Manager
- Three agent UI modules (delivery, quality, client-interaction)
- Communication approval queue
- Agent NL query with evidence display
- Super Admin metric configuration
- CSAT submission form

**Out of scope (no UI until Phase 2+):**

- Workforce rebalancing recommendations UI
- SOP / knowledge search
- Cross-client leadership analytics beyond read-only lists
- Automation engineer tooling
