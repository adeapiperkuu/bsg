## 1. Overview & Guiding Principles

The Operations Tower serves regulated-domain clients (Life Sciences, Finance/Insurance, Logistics) in Europe and the USA, with delivery operations running from India and Kosovo. The deployment model must satisfy five non-negotiable constraints that are stated explicitly in the source material and carry hard governance weight — they are not preferences:

**Principle 1 — Data never crosses tenant boundaries.** Every client's operational data is walled off at the database layer using PostgreSQL Row-Level Security (RLS). A misconfigured application route cannot leak cross-tenant data because the database itself will reject the query. This is architecturally enforced, not just policy.

**Principle 2 — No live client data before governance sign-off.** All environments below production run exclusively on synthetic or sanitised datasets. Connecting any environment to real client production data requires explicit client approval, a governance sign-off, a security validation, and a compliance review — in that order. This is not a preference; it is a hard deployment gate.

**Principle 3 — Every AI output must be traceable to its evidence.** The platform's "Evidence-Backed AI" principle means that agent queries, client communications, and risk alerts must each be linked to the specific source rows they were derived from. This has deployment implications: the evidence-link tables in the database must be live and populated before any AI-facing feature is enabled in any environment.

**Principle 4 — Human approval before client communication is sent.** 100% of AI-drafted outbound client communications must be reviewed and explicitly approved by a human Delivery Manager before the message leaves the system. No automated sending pipeline may bypass this gate.

**Principle 5 — EU & client-region data residency.** The Supabase project region and any additional cloud infrastructure must be selected to satisfy EU data residency requirements and, where applicable, client-specific regional constraints. Region selection is a compliance decision, not a performance one.

## 2. Environment Definitions

Three environments are defined. Each has a distinct purpose, data policy, and access control profile.

### `dev` — Local Development

| Property | Value |
|---|---|
| Purpose | Individual developer iteration; feature development and unit testing |
| Trigger | Runs on each developer's local machine on demand |
| Data policy | Synthetic data only — generated from `supabase/seed/` scripts; **never** real client data |
| Supabase instance | Local Supabase (Docker-based via `supabase start`) or a dedicated shared dev Supabase project |
| Backend | FastAPI started locally via `uvicorn` or `docker-compose` |
| Frontend | Vite dev server (`pnpm dev`) with hot-module replacement |
| Auth | Supabase Auth running locally; test users seeded from seed scripts with all four roles (`client`, `delivery_manager`, `bsg_leadership`, `super_admin`) |
| External services | All external API calls (LLM, email) should be mocked or pointed at sandbox/test endpoints |
| Access | Local machine only; no public URL |

### `staging` — Integration & QA

| Property | Value |
|---|---|
| Purpose | Integration testing, end-to-end Playwright test runs, pre-release review, and stakeholder preview |
| Trigger | Automatically deployed on every merge to `main` (via GitHub Actions) |
| Data policy | Synthetic/sanitised data only — same seed scripts as `dev`; no real client data under any circumstances |
| Supabase instance | Dedicated Supabase project (separate from production); migrations run automatically on deploy |
| Backend | Containerised FastAPI image deployed to hosting provider |
| Frontend | Deployed to Vercel (assumed) on the `staging` branch preview URL |
| Auth | Supabase Auth; test user accounts seeded; no real user accounts |
| External services | LLM and email providers pointed at sandbox/test credentials; never production API keys |
| Access | Internal team only; URL not publicly listed; basic auth or IP allowlist on top of app auth |
| Gate to prod | All Playwright e2e tests — including tenant isolation suite — must pass before any production deployment is considered |

### `prod` — Production

| Property | Value |
|---|---|
| Purpose | Live client-facing Operations Tower |
| Trigger | Manual promotion from `staging` after explicit approval gate (see Section 11) |
| Data policy | Live client data permitted only after: (1) explicit client approval, (2) governance sign-off, (3) security validation, (4) compliance review — all four, in writing |
| Supabase instance | Dedicated production Supabase project; EU-region (or client-region) selected at project creation; cannot be changed post-creation |
| Backend | Containerised FastAPI image deployed to hosting provider |
| Frontend | Deployed to Vercel production domain (`tower.boreksolutions.de` or equivalent) |
| Auth | Supabase Auth; real user accounts; RLS enforced for all tenants |
| External services | Production LLM and email API keys; stored in CI/CD secrets store and injected at runtime |
| Access | Public-facing; role-scoped by Supabase Auth + RLS |

## 3. Architecture Topology

The platform has three independently deployed layers plus one managed external service. Understanding how they connect is essential for correctly scoping network rules, CORS configuration, and secrets injection.

```
┌─────────────────────────────────────────────────────────────┐
│  CLIENT BROWSER / MOBILE                                     │
│  React 18 SPA (TypeScript, Vite, Tailwind, shadcn/ui)        │
│  Auth: Supabase Auth JS SDK — JWT in httpOnly cookie         │
└────────────────────┬────────────────────────────────────────┘
                     │ HTTPS — REST JSON
                     ▼
┌─────────────────────────────────────────────────────────────┐
│  BACKEND API                                                 │
│  Python 3.12 / FastAPI 0.111 / Uvicorn 0.30                 │
│  Containerised: Docker                                       │
│  Auth middleware: validates Supabase JWT on every request    │
│  Passes authenticated DB connection with user role claim     │
└────────────────────┬────────────────────────────────────────┘
                     │ asyncpg (TLS)
                     ▼
┌─────────────────────────────────────────────────────────────┐
│  SUPABASE (managed)                                         │
│  PostgreSQL 16 + RLS                                        │
│  Supabase Auth (JWT issuance + refresh)                     │
│  pgvector extension (candidate for RAG embeddings)          │
│  Region: EU (Frankfurt or Amsterdam )                       │
└─────────────────────────────────────────────────────────────┘

External calls from backend:
  ├── LLM provider API 
  ├── Embedding API
  └── Email delivery service

**CORS policy:** The backend `ALLOWED_ORIGINS` environment variable must list only the Vercel domain(s) for the given environment. Wildcard `*` origins are never acceptable in `staging` or `prod`.

**No direct database access from the frontend.** The Supabase `anon` key is embedded in the frontend bundle — this is safe only because RLS policies restrict what the anon key can read. The `service_role` key (which bypasses RLS) must never appear in any frontend code, build artifact, or client-side environment variable.

## 5. Containerisation

### Backend Dockerfile location

```
backend/Dockerfile
```

The `Dockerfile` must produce a minimal, reproducible image. Agents generating this file must follow these constraints:

- **Base image:** `python:3.12-slim` — no `-alpine` (musl/glibc compatibility issues with asyncpg and some LLM SDK wheels).
- **No secrets in the image.** All environment variables are injected at runtime by the hosting platform or CI/CD pipeline. The `Dockerfile` must not `COPY` any `.env` file.
- **No dev dependencies in the production image.** Use a multi-stage build: `builder` stage installs all deps (including dev tools for compilation), `runtime` stage copies only the installed packages and app code.
- **Non-root user.** The container must run as a non-root user. Add `RUN useradd -m appuser` and `USER appuser` before the `CMD`.
- **Health check.** Include a `HEALTHCHECK` that hits `GET /health` on the FastAPI app. The `/health` route must return `200 OK` with `{"status": "ok"}` and must not require authentication.
- **Port:** The container exposes port `8000`. The hosting platform maps this to its own external port.

### docker-compose.yml (local dev only)

```
docker-compose.yml  # root of repo
```

The `docker-compose.yml` orchestrates the local development stack and must define:

- `backend` service — builds from `backend/Dockerfile`, mounts `backend/` as a volume for hot-reload (via `uvicorn --reload`), exposes port `8000`
- `supabase` — not run via docker-compose directly; use `supabase start` (Supabase CLI manages its own containers internally); the compose file should document this as a prerequisite comment rather than defining a service
- Environment variables loaded from `.env` (via `env_file: .env` directive); `.env` is git-ignored and must be created locally from `.env.example`

The `docker-compose.yml` is for local development only. It is not used in staging or production.

## 6. CI/CD Pipeline

**Platform:** GitHub Actions.

**Workflow files location:** `.github/workflows/`

### 6.1 Workflow: `ci.yml` — Runs on every pull request

This workflow must complete without error before any PR can be merged to `main`. It has no deployment steps.

```
Trigger: pull_request → any branch targeting main

Jobs (run in parallel where possible):

  backend-checks:
    - Checkout repo
    - Set up Python 3.12
    - Install dependencies (pip install --break-system-packages or uv)
    - ruff check backend/           # linting
    - ruff format --check backend/  # formatting check
    - mypy backend/                 # type checking
    - pytest backend/tests/ -v --asyncio-mode=auto
      (uses a test Supabase project or mock DB — see Section 7)

  frontend-checks:
    - Checkout repo
    - Set up Node / pnpm 9
    - pnpm install
    - pnpm run lint           # ESLint 9
    - pnpm run format:check   # Prettier 3 check
    - pnpm run tsc            # tsc --noEmit
    - pnpm run test           # Vitest 2

  e2e-checks:
    - Checkout repo
    - Set up Node / pnpm 9 + Python 3.12
    - Start local Supabase (supabase start)
    - Seed test data (supabase db reset --local)
    - Start FastAPI backend (uvicorn in background)
    - Start Vite preview build
    - pnpm playwright install --with-deps
    - pnpm playwright test e2e/
      CRITICAL: tenant isolation tests must be in this suite
      and must not be skipped or marked xfail.
      A client-role user must provably be unable to read
      another org's data through any route.
```

### 6.2 Workflow: `deploy-staging.yml` — Runs on merge to `main`

```
Trigger: push → main

Jobs (sequential):

  run-ci:
    - Reuse / re-run all checks from ci.yml
    - If any check fails: abort; do not proceed to deploy

  deploy-backend-staging:
    - docker build -t ops-tower-backend:${{ github.sha }} backend/
    - docker push $DOCKER_REGISTRY_URL/ops-tower-backend:${{ github.sha }}
    - Deploy to hosting provider (staging)
      Inject env vars from GitHub Actions secrets (staging set)

  deploy-frontend-staging:
    - vercel deploy --env staging --token $VERCEL_TOKEN

  run-migrations-staging:
    - supabase db push --linked   (targets staging Supabase project)
    - If migration fails: alert team; stop; do not proceed

  smoke-test-staging:
    - curl GET /health → assert 200 {"status": "ok"}
    - Authenticated request per role → assert role-scoped response
    - Cross-tenant isolation assertion → assert no cross-org data returned
```

### 6.3 Workflow: `deploy-production.yml` — Manual promotion only

```
Trigger: workflow_dispatch (manual trigger from GitHub Actions UI)
Required reviewers: minimum 1 named approver
(configure via GitHub Environments > production > required reviewers)

Jobs (sequential):

  validate-staging:
    - Confirm staging smoke tests are green for the SHA being promoted
    - Print current prod SHA and incoming SHA to deployment log
    - Confirm no unapplied migrations exist on staging

  deploy-backend-production:
    - Pull the exact same Docker image SHA deployed to staging
      (never rebuild the image for production — identical artifact)
    - Deploy to hosting provider (production)
      Inject env vars from GitHub Actions secrets (production set)

  deploy-frontend-production:
    - vercel promote <staging-deployment-url> --token $VERCEL_TOKEN
      (promotes the exact staging build — no rebuild)

  run-migrations-production:
    - supabase db push --linked   (targets production Supabase project)
    - Migrations run AFTER backend deploy — old backend must be
      forward-compatible with new schema (additive-only rule)
    - If migration fails: trigger rollback procedure (Section 14)

  post-deploy-smoke-test:
    - GET /health → assert 200
    - Role-scoped authenticated requests for all four roles
    - Tenant isolation assertion
    - Notify team on success
```

### 6.4 Branch strategy

| Branch | Purpose | Deploys to |
|---|---|---|
| `feature/*` | Feature development; open PRs against `main` | Nowhere (CI checks only) |
| `main` | Integration branch; always deployable | Staging (automatic on merge) |
| Manual trigger on `main` SHA | Production promotion | Production (manual gate, approved) |

No long-lived `develop` branch. All feature work goes directly to `main` via PR. Hotfixes follow the same path.

## 7. Database & Migrations

**Tool:** Supabase CLI. Migration files live in `supabase/migrations/`.

**File naming:** `{unix_timestamp}_{snake_case_description}.sql` — generated automatically by `supabase migration new <description>`.

### Per-environment migration behaviour

| Environment | How migrations run | When |
|---|---|---|
| `dev` | `supabase db reset --local` — wipes local DB and re-runs all migrations + seed | On demand by developer |
| `staging` | `supabase db push --linked` — applies pending migrations to the staging project | Automatically, as part of `deploy-staging.yml`, after backend deploy |
| `prod` | `supabase db push --linked` — applies pending migrations to the production project | Manually, as part of `deploy-production.yml`, after backend deploy, with human approval |

### Seed data policy

- `supabase/seed/` contains only synthetic, fabricated, or anonymised operational data.
- Seed scripts are applied in `dev` and `staging` only. They are **never** run against the production Supabase project.
- Seed data must include test users for all four roles (`client`, `delivery_manager`, `bsg_leadership`, `super_admin`) across at least two distinct organisations so that tenant isolation tests can be exercised end-to-end.
- Seed data must never contain: real client names, real project data, real annotator names, or real quality metrics from live BSG operations.

### RLS policy deployment

RLS policies are defined in migration files — in the same migration as the table they protect, or a dedicated subsequent migration. A table must never be deployed to `staging` or `prod` without its RLS policy already applied. Never deploy a new table and leave it unprotected, even temporarily.

## 8. Environment Variables & Secrets Management

### Storage locations by environment

| Environment | Where vars live | Who can access |
|---|---|---|
| `dev` | `.env` file at repo root (git-ignored) | Local developer only |
| `staging` | GitHub Actions secrets (staging-prefixed set) | CI/CD pipeline only |
| `prod` | GitHub Actions secrets (production set) + hosting provider secret injection | CI/CD pipeline; named on-call team member for break-glass |

`.env.example` is committed to the repository with placeholder values and a one-line comment per key. It is the canonical list of all required variables. No real secret values ever appear in `.env.example` or in any committed file.

### Complete variable reference

**Supabase**
```
SUPABASE_URL=             # Project URL e.g. https://xxxxxxxxxxxx.supabase.co
SUPABASE_ANON_KEY=        # Public anon key — safe to embed in frontend bundle
                          # RLS policies are the security layer for this key;
                          # this key alone cannot access cross-tenant data.
SUPABASE_SERVICE_ROLE_KEY= # Full-access key — bypasses RLS entirely.
                           # BACKEND ONLY. Never in frontend code or Vite build.
                           # Used only by FastAPI service account for admin ops.
```

**Backend (FastAPI)**
```
DATABASE_URL=      # asyncpg DSN: postgresql+asyncpg://user:pass@host:port/db
                   # Derived from Supabase connection pooler string
SECRET_KEY=        # Random 32+ byte hex string; signs internal session data
ENVIRONMENT=       # Literal: dev | staging | prod
                   # Gates debug routes, log verbosity, error detail in responses
ALLOWED_ORIGINS=   # Comma-separated CORS-allowed origins
                   # dev:     http://localhost:5173
                   # staging: https://staging.tower.boreksolutions.de
                   # prod:    https://tower.boreksolutions.de
```

**LLM / AI layer**
```
LLM_API_KEY=       # API key for chosen LLM provider
LLM_BASE_URL=      # Base URL for LLM REST API endpoint
LLM_MODEL=         # Pinned model identifier — never a floating alias like "latest"
                   # Must be updated deliberately and tested before deploying
EMBEDDING_API_KEY= # If embedding provider differs from LLM provider
                   # Leave blank or omit if the same provider handles both
```

**Email / notifications**
```
EMAIL_API_KEY=        # API key for email delivery service
EMAIL_FROM_ADDRESS=   # Verified sender e.g. tower@boreksolutions.de
```

**CI/CD infrastructure**
```
VERCEL_TOKEN=          # Vercel deployment token (GitHub Actions secret)
DOCKER_REGISTRY_URL=   # Container registry base URL
                       # e.g. ghcr.io/boreksolutions/ops-tower
```

### Security rules for secrets

- No secret value ever appears in a log line, exception message, or API response body. FastAPI exception handlers must sanitise output in `staging` and `prod`.
- `SUPABASE_SERVICE_ROLE_KEY` is never injected into the Vite build process. The frontend receives only `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`.
- All production secrets are rotated immediately if there is any suspicion of exposure. Rotation requires a backend redeploy to inject the new values.
- `LLM_MODEL` must be a pinned identifier. If the provider silently updates what a floating alias resolves to, the impact must be evaluated before the model variable is updated.

## 9. Data Residency & Compliance Constraints

These are governance requirements, not suggestions. They flow directly from the project's client base (regulated Life Sciences, Finance, Logistics in Europe/USA) and BSG's explicit commitment to "EU & client-region compliant" data handling.

### Supabase project region

The production Supabase project must be created in an EU region (Frankfurt `eu-central-1` or Amsterdam `eu-west-1`). This decision is **irreversible after project creation** — Supabase does not support region migration. Confirm the region before provisioning the production project.

For clients with contractual data-residency requirements outside the EU (e.g. US Life Sciences clients with HIPAA obligations), a separate Supabase project in the appropriate region may be required per client. This is a post-Phase 1 concern to resolve once the pilot client and vertical are confirmed.

### Tenant isolation — three enforcement layers

Tenant isolation is enforced at three layers. All three must be in place before production:

**Layer 1 — Database (RLS):** Every client-data table has `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` and a policy filtering on `org_id`. RLS is the primary security boundary. A buggy API route cannot leak cross-tenant data because the database itself returns zero rows for an unauthorised query.

**Layer 2 — Application (FastAPI middleware):** Every route that reads client data extracts `org_id` from the validated JWT claim and passes it as an explicit filter to the query. This is defence-in-depth — it does not replace RLS but makes intent explicit and catches bugs at code review time.

**Layer 3 — Test suite (Playwright e2e):** The e2e suite must include a test that authenticates as a `client` user in org A, attempts to fetch data belonging to org B via the API, and asserts either an empty result set or a 403 response — never org B's actual data. This test is mandatory and must not be skipped.

### Regulated domain considerations

Source data spans three regulated domains, each with different traceability requirements:

- **Life Sciences / Medical AI:** Audit trails (who did what, when) must be complete and exportable for regulatory review. The `agent_queries` and `communication_evidence_links` tables serve this function. May be subject to HIPAA (US clients) or MDR/IVDR (EU).
- **Finance / Insurance:** All client-facing narrative content must be human-approved before sending (the `client_communications` approval gate). Data lineage must be traceable end-to-end.
- **Logistics:** Lower regulatory burden in Phase 1, but operational data quality and basic auditability are still required.

No vertical-specific deployment topology is required in Phase 1 — a single platform with tenant isolation and audit trails serves all three.

## 10. Security Posture

### Authentication flow

1. User navigates to the Operations Tower frontend.
2. Frontend calls Supabase Auth (email/password in Phase 1; SSO is not planned).
3. Supabase Auth issues a short-lived JWT access token and a long-lived refresh token.
4. Token stored in an `httpOnly` cookie. `httpOnly` prevents JavaScript access, mitigating XSS token theft.
5. Frontend includes the cookie on every API request to the FastAPI backend.
6. FastAPI auth middleware validates the JWT signature against Supabase's public key. Extracts `user_id`, `org_id`, and `role` from the claims. Attaches them to the request context.
7. The backend runs database queries in the context of the authenticated user's role. RLS policies evaluate using `auth.uid()` and the role claim.
8. On JWT expiry the frontend requests a new access token via the Supabase Auth SDK refresh flow.

### Role model

| Role | Visibility | Permitted actions |
|---|---|---|
| `client` | Their organisation's projects, quality metrics, confidence scores, sent communications, their own agent query history | Submit NL queries, read dashboards, submit CSAT scores |
| `delivery_manager` | All projects within their organisation; full agent output | Everything `client` can do, plus: approve/reject AI-drafted communications, create/update projects and milestones, acknowledge and resolve risk alerts |
| `bsg_leadership` | All organisations, all projects — read-only cross-org view | No write operations; read-only portfolio analytics |
| `super_admin` | Platform configuration tables only; no access to client operational data | Manage `metric_configurations`; create and deactivate organisations and users |

### CORS

The FastAPI backend rejects requests from any origin not listed in `ALLOWED_ORIGINS`. In production, only the Vercel production domain is listed. Wildcard `*` origins are never permitted in `staging` or `prod`.

### Key boundaries

- `SUPABASE_SERVICE_ROLE_KEY` is used only by the FastAPI backend service account for admin-scoped operations. It must never appear in the Vite build, any frontend bundle, or any client-facing response.
- All inter-service communication uses HTTPS/TLS. No plaintext HTTP in `staging` or `prod`.
- All LLM API keys are stored only in the CI/CD secrets store and injected as runtime environment variables. They are never logged.

## 11. Production Promotion Gate

Before any deployment to production is permitted, all of the following must be satisfied. This gate exists because the platform handles sensitive regulated-domain client data and because AI-generated content reaches external clients directly.

### Technical gates (automated — enforced by CI/CD)

- All CI checks (lint, type-check, unit tests, integration tests) pass on the SHA being promoted.
- All Playwright e2e tests pass on staging, including the tenant isolation suite.
- Staging `GET /health` returns `200 OK`.
- All four role-scoped authenticated routes return correctly filtered data on staging (automated smoke test).
- No database migration is pending on staging that has not been applied and verified.

### Governance gates (human sign-off — required before connecting real client data)

These apply to the first production deployment that will handle a real client's operational data. They must be obtained in writing and stored:

1. **Client approval:** The client has explicitly approved connecting their operational data to the platform.
2. **Governance sign-off:** BSG leadership has reviewed and signed off that RBAC, tenant isolation, and audit trails are in place.
3. **Security validation:** A review of RLS policies, CORS configuration, secrets management, and authentication flow has been completed.
4. **Compliance review:** A domain-appropriate compliance review has been completed (Life Sciences / Finance / Logistics — each has different requirements).

### Operational gates (human sign-off — required for every production deployment)

- At least one named team member has reviewed the database migration diff.
- If the deployment includes schema changes: a rollback plan is written and reviewed before deployment begins.
- The previous production image SHA is recorded in the deployment log.

## 12. Local Development Setup

This is the canonical onboarding reference for new developers. Assumes macOS or Linux. Windows is supported via WSL2.

### Prerequisites

```bash
node >= 20         # for pnpm and Vite
pnpm 9             # install: npm install -g pnpm@9
python 3.12        # for the FastAPI backend
docker             # Docker Desktop or Docker Engine
docker compose     # included with Docker Desktop
supabase           # Supabase CLI
                   # macOS: brew install supabase/tap/supabase
                   # Other: see https://supabase.com/docs/guides/cli
```

### First-time setup

```bash
# 1. Clone the repo
git clone <repo-url> && cd ops-tower

# 2. Copy env template and fill values
cp .env.example .env
# Edit .env:
#   SUPABASE_URL          → from `supabase start` output
#   SUPABASE_ANON_KEY     → from `supabase start` output
#   SUPABASE_SERVICE_ROLE_KEY → from `supabase start` output (backend only)
#   DATABASE_URL          → asyncpg DSN from `supabase start` output
#   SECRET_KEY            → python -c "import secrets; print(secrets.token_hex(32))"
#   ENVIRONMENT           → dev
#   ALLOWED_ORIGINS       → http://localhost:5173
#   LLM_API_KEY           → sandbox/test key
#   LLM_BASE_URL          → provider URL
#   LLM_MODEL             → pinned model string

# 3. Start local Supabase
supabase start
# First run pulls Docker images — may take several minutes.
# Outputs SUPABASE_URL, SUPABASE_ANON_KEY, and DATABASE_URL for your .env.

# 4. Apply migrations and seed synthetic data
supabase db reset --local
# Wipes local DB, runs all migrations in order, then applies seed scripts.

# 5. Install frontend dependencies
cd frontend && pnpm install && cd ..

# 6. Install backend dependencies
cd backend
pip install --break-system-packages -e ".[dev]"
# Alternative: uv pip install -e ".[dev]"
cd ..

# 7. Start the dev stack
# Option A — individual processes (recommended for active development)
cd frontend && pnpm dev &                            # Vite on :5173
cd backend && uvicorn app.main:app --reload --port 8000 &

# Option B — docker-compose (closer to staging behaviour)
docker compose up

# 8. Verify
open http://localhost:5173              # Frontend
curl http://localhost:8000/health       # Should return {"status":"ok"}
open http://localhost:54323            # Supabase Studio (local DB browser)
```

### Running tests locally

```bash
# Backend — unit + integration
cd backend && pytest tests/ -v --asyncio-mode=auto

# Frontend — unit
cd frontend && pnpm test

# Type checking
cd frontend && pnpm run tsc      # TypeScript — tsc --noEmit
cd backend && mypy app/          # Python

# Linting and formatting
cd frontend && pnpm run lint && pnpm run format:check

## 13. Monitoring, Observability & Alerting

> **Note:** Specific observability tooling is not confirmed in the source material. The requirements below are the minimum necessary before production client data is connected, derived from the platform's audit-trail and traceability requirements. Tool selection is an open question — see Section 15.

### What must be observable

| Signal | Why it matters | Source |
|---|---|---|
| Backend error rate and 5xx responses | Any backend error may mean an insight failed to cite evidence or a communication was not gated correctly | FastAPI exception handlers; structured JSON logs |
| `client_communications` approval gate failures | If the approval step errors or is bypassed, AI drafts may reach clients unreviewed | Application-level log on every status transition in the `client_communications` table |
| Cross-tenant query attempts | Any query touching a different `org_id` is a critical security event — even one blocked by RLS | PostgreSQL statement log + FastAPI middleware |
| LLM API errors and latency | Agent outputs fail or degrade if LLM calls error; the NL query response-time SLA is < 2 hours | FastAPI service layer; log every LLM call with latency and error code |
| Database migration failures | A failed migration in production can leave the schema in a partially applied state | Supabase CLI exit code in CI/CD; alert on non-zero |
| JWT validation failures | Indicate expired keys, misconfigured Supabase project, or a potential attack | FastAPI auth middleware; log every failure |

### Structured log format

All FastAPI log output must be structured JSON. Every log entry must include:

```json
{
  "timestamp":   "ISO 8601 — always UTC",
  "level":       "INFO | WARNING | ERROR | CRITICAL",
  "request_id":  "UUID — generated per request for tracing",
  "user_id":     "UUID | null for unauthenticated requests",
  "org_id":      "UUID | null",
  "route":       "/api/v1/...",
  "method":      "GET | POST | PATCH | ...",
  "status_code": 200,
  "latency_ms":  142,
  "message":     "human-readable description"
}
```

`user_id` and `org_id` must appear only as typed fields — never embedded inside the `message` string, to avoid leaking PII into freeform log text.

## 14. Rollback Procedure

### Application rollback (frontend + backend)

Because production always deploys the exact same Docker image SHA and Vercel build that were validated on staging, rollback is a re-deploy of the previous known-good SHA. The previous SHA must be recorded in the deployment log at the start of every production deploy run.

```bash
# Backend: re-deploy previous image SHA
docker pull $DOCKER_REGISTRY_URL/ops-tower-backend:<previous-sha>
# redeploy <previous-sha> using hosting provider CLI or dashboard

# Frontend: Vercel instant rollback
vercel rollback <previous-deployment-url> --token $VERCEL_TOKEN
```

### Database migration rollback

Supabase does not have a native `migrate down` command. Database rollback requires a new forward migration (a `.sql` file that reverses the changes).

**Additive-only rule:** Every migration that modifies or removes existing schema (dropping a column, renaming a column, changing a type, dropping a table) must be accompanied by a pre-written reverse migration that is reviewed and stored *before* the forward migration is deployed to production.

Purely additive migrations (adding a nullable column, adding a new table with no FK constraints on existing tables) can be deployed without a pre-written reverse migration, because the previous application code remains compatible with the new schema.

**Procedure on migration failure in production:**

1. CI/CD alerts the team that `supabase db push` exited non-zero.
2. Do not manually patch the production database.
3. The backend remains on the previous image (backend deploys before migrations — old code is forward-compatible with a partially applied additive schema).
4. Apply the pre-written reverse migration via `supabase db push --linked` targeting production.
5. Confirm `GET /health` returns 200 and run all four role-scoped smoke tests.
6. File a post-incident review documenting root cause and preventive action.
