## 1. Agent Summary

The Operational Knowledge Agent is a context-aware assistance agent for operational knowledge, SOP retrieval, historical issue referencing, process guidance, and best-practice recommendations.

Source-provided description:

- Provides SOP retrieval.
- Provides process guidance.
- Provides historical issue referencing.
- Provides best-practice recommendations.
- Provides context-aware operational assistance.
- Answers: How is quality trending this month?

The Word document defines this agent at the ecosystem level only. It does not include a full BRD section for Operational Knowledge Agent in version 1.0. The details below extend the source definition using the platform architecture, governance model, and existing AI/RAG documentation.

## 3. Mandate

The Operational Knowledge Agent turns operational process knowledge into searchable, contextual guidance.

It should help users answer:

- What SOP applies to this project or issue?
- What process should the team follow?
- Have we seen a similar issue before?
- What best-practice mitigation was used previously?
- Which historical lessons are relevant to the current project?
- How is quality trending this month?

The agent should connect current operational questions to governed knowledge sources such as SOPs, project charters, escalation notes, historical issue records, and lessons learned.

---

## 4. Strategic Objective

Build an operational knowledge layer that:

- Reduces time spent searching SOP repositories.
- Helps PMs and Delivery Managers apply consistent process guidance.
- Makes historical lessons reusable.
- Improves operational consistency across teams.
- Supports quality and governance decisions with past evidence.
- Helps teams avoid repeating known issues.

The agent should make BSG's operational knowledge easier to use during live delivery, not merely easier to store.

## 6. Problems Solved

The source document describes operational data and knowledge as fragmented across dashboards, SOP repositories, notes, and governance artifacts.

The Operational Knowledge Agent addresses:

- SOPs being difficult to locate during active delivery.
- Process guidance being inconsistently applied.
- Historical issue knowledge being trapped in PM notes or escalation records.
- Similar issues being solved repeatedly from scratch.
- Quality corrections not being linked back to guidelines or best practices.
- Teams lacking context-aware guidance at the moment they need it.

## 7. Questions the Agent Answers

Source-provided example:

- How is quality trending this month?

Additional expected examples:

- Which SOP applies to this quality issue?
- Have we seen this error pattern before?
- What corrective action worked last time?
- What process should the team follow for escalation?
- Which guideline explains this labeling rule?
- What changed in the SOP since the last project phase?
- What best practice applies to reducing rework?
- Which historical projects had similar drift?

Clarification:

- "How is quality trending this month?" overlaps with the Quality Intelligence Agent.
- Operational Knowledge should answer that question by combining quality trend evidence with SOP/process/historical context.
- Quality Intelligence remains the owner of quality metrics and drift detection.

## 8. Core Capabilities

### 8.1 SOP Retrieval

Finds relevant SOPs for a project, task, quality issue, or user question.

Expected behavior:

- Retrieve only approved SOPs.
- Respect tenant and role permissions.
- Show relevant excerpts or summaries.
- Link back to source documents where available.
- Avoid presenting outdated SOP guidance as current.

### 8.2 Process Guidance

Provides step-by-step operational guidance based on approved process documents.

Examples:

- How to escalate a milestone risk.
- How to handle recurring quality drift.
- How to prepare a client update.
- How to perform reviewer calibration.

### 8.3 Historical Issue Referencing

Finds similar past issues and explains what happened.

Examples:

- Similar quality drift cases.
- Similar throughput bottlenecks.
- Similar escalation patterns.
- Prior mitigation actions and outcomes.

### 8.4 Best-Practice Recommendations

Recommends best practices based on approved SOPs and historical outcomes.

Examples:

- Recommend a reviewer calibration when class confusion increases.
- Recommend an SOP review when guideline ambiguity appears.
- Recommend escalation if a similar historical issue led to milestone slip.

### 8.5 Context-Aware Operational Assistance

Tailors guidance to the current project, role, and available evidence.

Expected behavior:

- Use project context.
- Use quality context.
- Use delivery context.
- Use role permissions.
- Provide practical next steps.

## 10. Required Knowledge Store

The source document references SOP repositories and governance artifacts. To implement the Operational Knowledge Agent properly, the platform needs a knowledge store.

Required metadata for every knowledge item:

| Field | Purpose |
|---|---|
| `id` | Unique document or chunk identifier |
| `org_id` | Tenant isolation |
| `project_id` | Optional project scope |
| `source_type` | SOP, charter, escalation note, PM note, training doc, lesson learned |
| `title` | Human-readable document title |
| `version` | Version or revision identifier |
| `status` | Draft, approved, archived |
| `effective_date` | Date the guidance became valid |
| `owner` | Business owner or approver |
| `visibility` | Internal-only, client-safe, leadership-only |
| `content_chunk` | Searchable text chunk |
| `embedding` | Vector representation if vector search is used |

No current MVP table fully models this. A Phase 2 schema extension is required before production implementation.

## 11. RAG Behavior

Operational Knowledge Agent is heavily dependent on Retrieval-Augmented Generation.

Recommended flow:

1. Validate authenticated user and role.
2. Determine project and organisation context.
3. Retrieve relevant structured operational data.
4. Retrieve relevant approved knowledge documents.
5. Filter by tenant, role, status, and visibility.
6. Rank results by relevance and recency.
7. Build prompt with source excerpts and metadata.
8. Generate answer with citations.
9. Validate that recommendations are grounded in retrieved sources.
10. Persist answer and evidence links.

The agent must not answer process questions from model memory alone.

---

## 12. Evidence and Citation Rules

Every answer must cite evidence from:

- Structured operational rows.
- Approved knowledge documents.
- Historical issue references.
- SOP excerpts.

If no approved source exists, the agent must say that guidance is unavailable rather than inventing process steps.

Example safe response:

```text
I cannot find an approved SOP or historical issue record for this question in the current project context.
```

- Providing approved client-safe SOP context.
- Explaining historical context where approved.

## 14. Role-Based Response Rules

### Internal Users

Internal users may receive:

- SOP references.
- Process guidance.
- Historical issue examples.
- Internal lessons learned.
- Recommended operational actions.

Access is still restricted by organisation, project, and role.

### Client Users

Client users should not receive Operational Knowledge Agent access by default.

If client-safe access is approved later, clients may receive:

- Approved client-facing process summaries.
- Approved SOP excerpts.
- General explanation of mitigation process.

Clients must not receive:

- Internal PM notes.
- Internal lessons learned.
- Other client examples.
- Staff-specific details.
- Draft SOPs or unapproved process documents.

## 15. Example Answer Pattern

Question:

```text
How is quality trending this month?
```

Expected behavior:

1. Retrieve this month's quality snapshots.
2. Retrieve prior comparable period if available.
3. Retrieve relevant error entries.
4. Retrieve SOP or guideline context if quality issues are linked to process ambiguity.
5. Explain the trend.
6. Reference evidence.
7. Recommend a process action if evidence supports it.

Example answer structure:

```text
Quality is stable overall, with rework remaining below the current threshold. The main issue category this month is boundary precision. The relevant SOP section suggests reviewer calibration when boundary precision errors increase, so the recommended action is to run a short calibration review before the next batch.
```

The final implementation must include evidence links rather than only prose.

## 16. Governance and Security

Required controls:

- RBAC.
- Tenant isolation.
- Approved-source retrieval only.
- Version-aware SOP retrieval.
- Audit logging.
- Evidence links.
- Client-safe filtering.
- No model-only process advice.
- No cross-client historical examples unless explicitly anonymized and approved.

The agent must treat SOPs and historical issue references as governed knowledge, not casual notes.

## 19. Success Criteria

The Operational Knowledge Agent is successful when:

- Users can find the right SOP quickly.
- Process answers cite approved documents.
- Historical issue references are relevant and permission-safe.
- Recommendations are grounded in evidence.
- The agent reduces repeated manual searching.
- The agent improves consistency of operational decisions.
- Client users never see internal-only knowledge.
