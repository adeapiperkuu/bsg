## 1. Agent Summary

The Client Interaction Agent is an AI-native client engagement and operational transparency agent. It translates operational delivery intelligence into client-ready narratives, confidence-based updates, real-time project answers, executive summaries, and risk explanations.

It helps BSG reduce manual client reporting effort while giving clients clearer visibility into project health, milestone confidence, quality trends, and mitigation progress.

Source-provided description:

- Provides AI-assisted status updates.
- Provides real-time client Q&A.
- Provides executive-ready summaries.
- Provides transparency dashboards.
- Provides delivery confidence reporting.
- Answers: What is the current project health? Which milestones are at risk?

| Item | Status |

| Product area | Client transparency and communication |
| Primary interface | Operations Tower web UI |
| Primary backend module | `client_interaction_agent` |
| Primary users | Client stakeholders, client program managers, client leadership, internal PMs, delivery leadership |

The Client Interaction Agent is part of the Phase 1 trio together with:

- Delivery Performance Agent.
- Quality Intelligence Agent.

The agent must not expose Phase 2+ functionality in the MVP UI, including:

- Go-Live Readiness reports.
- Margin analytics.
- ROI analytics.
- Churn prediction.
- Multi-client portfolio intelligence.
- AI relationship sentiment analysis.
- Predictive client escalation detection.

The source document mentions readiness assessment and go-live readiness as a future or extended Client Interaction capability. Based on the current resolved project decisions, those features are Phase 2+ and must not appear in the MVP UI.

## 9. Core Capabilities

### 9.1 Client Narrative Generation

Converts operational complexity into simple, executive-ready client summaries.

Expected behavior:

- Summarize project health.
- Explain meaningful changes.
- Highlight progress.
- Explain risk in business language.
- Avoid unnecessary internal operational detail.
- Reference evidence.

Source example:

> Projects remain on track with minor fluctuations that have already been mitigated.

### 9.2 Delivery Confidence Intelligence

Explains delivery confidence scores and milestone reliability.

Expected behavior:

- Retrieve the current delivery confidence score.
- Explain the main drivers behind the score.
- Connect confidence to throughput, quality, open risks, and bottlenecks.
- Avoid inventing a score if no score exists.

Source example:

> Milestone confidence is 89% due to stable throughput and proactive QA interventions.

### 9.3 Risk Transparency Engine

Translates operational issues into client-safe risk narratives.

Expected behavior:

- Explain risks clearly.
- Show whether mitigation is underway.
- Avoid internal blame language.
- Avoid exposing staff-level details.
- Show client action needed, if any.

Source example:

> Resource constraints were identified and mitigated through additional trained resource deployment.

### 9.4 Executive Summary Generation

Creates concise status summaries for client leadership and BSG delivery leadership.

Expected behavior:

- Keep summaries short.
- Emphasize overall health, confidence, material risks, mitigation, and outlook.
- Include only client-safe evidence when the summary is client-facing.

### 9.5 Client Conversational Interface

Allows client users and internal users to ask natural-language questions about project status.

Expected behavior:

- Enforce role and tenant access before retrieval.
- Answer only from available evidence.
- Include evidence links or evidence descriptions.
- Return a safe no-answer response when evidence is insufficient.

### 9.6 Recommendation and Guidance Engine

Provides suggested actions where evidence supports them.

MVP examples:

- Recommend monitoring a milestone because confidence dropped.
- Recommend explaining a quality drift mitigation in the next weekly summary.
- Recommend Delivery Manager review before publishing a client update.

Phase 2+ examples:

- Pre-go-live readiness actions.
- Resource strategies tied to broader Workforce & Capability Agent logic.
- Pilot validation recommendations.

## 12. Governance Rules

The source document states:

- No hallucinated project status.
- All narratives must be evidence-backed.
- Confidence scoring is mandatory where confidence is discussed.
- Sensitive data protection is required.

Implementation rules:

- The agent must not invent project status, dates, risks, metrics, or mitigation actions.
- Every generated answer or draft must cite source evidence.
- Client users can only see their own organisation's project data.
- Draft communications must not be visible to clients.
- Client-facing communications require Delivery Manager approval before publishing.
- The agent must never send or publish automatically.
- MVP delivery is in-app only. PDF, DOCX, and outbound email delivery are Phase 2+.

---

## 13. Human Approval Workflow

The Client Interaction Agent may generate communication drafts, but a human Delivery Manager must approve them.

Workflow:

1. Delivery Manager requests a draft.
2. Agent retrieves project evidence.
3. Agent generates `body_draft`.
4. Backend stores the draft in `client_communications`.
5. Backend stores evidence in `communication_evidence_links`.
6. Delivery Manager reviews and edits the approved body.
7. Delivery Manager approves or rejects the communication.
8. Approved communication can be published in-app.
9. Client users can only see communications with `status = sent`.

## 14. RAG and Evidence Behavior

Core architecture from the source:

- KPI semantic layer.
- Narrative generation framework.
- RAG architecture.
- Confidence scoring engine.
- Role-based access layer.

Recommended RAG flow:

1. Validate authenticated user and role.
2. Validate project access.
3. Retrieve structured project evidence.
4. Retrieve document evidence if unstructured RAG is available.
5. Filter evidence by role and tenant.
6. Build prompt context.
7. Generate answer or draft.
8. Validate output for unsupported claims.
9. Persist output and evidence links.
10. Return role-safe response.

Evidence is mandatory for persisted outputs.

## 15. Role-Based Response Rules

### Client User

Client users receive:

- Client-safe project health.
- Delivery confidence.
- Milestone risk.
- Quality summary if configured as client-visible.
- Mitigation status.
- Sent communications only.

Client users must not receive:

- Draft communications.
- Other client data.
- Annotator names.
- Internal staffing details.
- Internal PM notes.
- Cross-client portfolio data.

### Delivery Manager

Delivery Managers receive:

- Full project evidence for their organisation.
- Draft generation tools.
- Communication review and approval tools.
- Risk and mitigation detail.
- Evidence links.

### BSG Leadership

BSG Leadership receives:

- Read-only cross-client reporting where implemented.
- Portfolio-level summaries only within confirmed scope.

Margin analytics, ROI analytics, churn prediction, and broader portfolio intelligence are Phase 2+.

Primary endpoints:

| Endpoint | Usage |
|---|---|

## 17. Success Criteria

The Client Interaction Agent is successful when:

- Client users can ask project-health questions and receive evidence-backed answers.
- Delivery Managers can generate weekly and executive drafts.
- AI-drafted communications require human approval before publishing.
- Clients only see approved/sent communication.
- Delivery confidence is explained using source data.
- Client-facing risk narratives are transparent but safe.
- No answer includes unsupported or cross-tenant information.
