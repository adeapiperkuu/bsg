## 1. Agent Summary

The Project Governance Agent is an AI-enabled program governance agent for charter management, scope tracking, dependency visibility, escalation summaries, and action tracking across BSG delivery programs.

Source-provided description:

- Provides charter generation.
- Provides governance tracking.
- Provides scope change analysis.
- Provides dependency visibility.
- Provides escalation summaries.
- Provides action tracking.
- Answers: What are the top governance risks? Which dependencies threaten timelines?

The Word document defines this agent at the ecosystem level only. It does not include a dedicated full BRD section (unlike Delivery Performance §7.1 and Client Interaction §7.5). The details below extend the source definition using the shared platform architecture, RBAC model, and Phase 2 roadmap.

**Terminology note:** Project Governance uses **Red / Amber / Green (RAG)** status for program health — distinct from **Retrieval-Augmented Generation**.

## 3. Mandate

The source positions BSG's market shift toward predictive project governance and away from spreadsheet-driven governance.

The Project Governance Agent turns fragmented governance artifacts — charters, scope decisions, dependencies, escalation notes, action registers — into a single governed operational layer inside the Operations Tower.

It should help users answer:

- What are the top governance risks across my portfolio?
- Which dependencies are blocking delivery timelines?
- Which scope changes are pending approval?
- What escalations require leadership or client input?
- What actions are due before the next governance call?
- Which projects have open governance blockers vs. clean scope posture?

The agent connects program-level questions to governed sources: project metadata, scope state, dependency register, escalation register, governance actions, and read-only delivery/quality risk signals.

---

## 4. Strategic Objective

Build a program governance layer that:

- Reduces spreadsheet-driven governance and PM administrative overhead.
- Makes cross-project dependencies visible before they become delivery crises.
- Standardizes escalation handling and action tracking.
- Improves governance confidence for clients and leadership.
- Supports evidence-backed governance summaries with human approval.
- Shifts governance from reactive escalation to proactive dependency and scope management.

The agent should make BSG's governance posture traceable and operational, not buried in email threads and PM notes.

## 6. Problems Solved

The source document describes operational intelligence goals including improved governance confidence, reduced PM administrative overhead, and delivery governance becoming predictive rather than reactive.

The Project Governance Agent addresses:

- Governance tracked in spreadsheets and disconnected PM tools.
- Cross-project dependencies invisible until milestones slip.
- Scope changes lacking a single approved source of truth.
- Escalations scattered across email, meetings, and risk registers.
- No unified register linking scope, dependencies, actions, and escalations per project.
- Governance call preparation requiring manual consolidation.
- Difficulty answering which dependencies threaten timelines at portfolio level.

## 8. Core Capabilities

### 8.1 Charter Generation

Draft and maintain project charters from governed project context.

Expected behavior:

- Generate charter drafts from project metadata, milestones, scope notes, and approved Knowledge documents (`project_charter` source type).
- Store approved charters in the Knowledge library or a dedicated governance store.
- Version charters when scope materially changes.
- Require human review before treating a charter as approved.

### 8.2 Governance Tracking

Maintain a per-project governance register.

Expected behavior (matches `/governance` **Governance Register** UI):

- Show scope posture per project (approved, pending revision, locked).
- Count open dependencies, actions, and escalations per project.
- Roll up portfolio KPIs: open actions, at-risk items, open escalations, SLA adherence.

### 8.3 Scope Change Analysis

Track scope state and change requests.

Expected behavior:

- Record scope status and version label (e.g. `v2`, `Pending v2`).
- Flag projects with pending scope revisions.
- Link scope changes to dependencies and escalations when relevant.
- Do not auto-approve scope changes without DM action.

### 8.4 Dependency Visibility

Cross-project dependency tracker.

Expected behavior (matches **Dependency Tracker** UI):

- Track dependency title, type (`client_action`, `internal`, `external`), blocking project, owner, due date, overdue days, status.
- Support resolve workflow ("Mark Resolved").
- Highlight blocking dependencies on the critical path.

### 8.5 Escalation Summaries

Formal escalation register and narrative summaries.

Expected behavior (matches **Escalation Register** UI):

- Track title, project, severity, raised by, date, status, assignee, notes.
- Distinguish governance escalations (program/human) from automated delivery risk alerts (`risk_alerts`).
- Allow promotion from Delivery/Quality alerts when appropriate.
- Generate escalation summaries for governance calls.

### 8.6 Action Tracking

Governance action items and weekly cadence.

Expected behavior (matches **Governance This Week** UI):

- Track actions with owner and due date.
- Show items due before the next governance call.
- Support checklist completion.
- Compute SLA adherence (e.g. percentage of actions closed on time).

### 8.7 Weekly Governance Summary (AI-assisted)

Auto-generated portfolio governance narrative with human approval.

Expected behavior:

- Aggregate open dependencies, escalations, actions, and material delivery risks.
- Generate draft summary text with evidence links.
- Require Delivery Manager approval before treating as official (mirror Client Interaction approval discipline).

### 8.8 Governance Agent Q&A

Natural-language interface after structured data is stable.

Expected behavior:

- `agent_name`: `project_governance_agent`
- Ground answers in governance tables plus read-only shared signals.
- Persist queries in `agent_queries` with `agent_query_evidence_links`.
- Reject insufficient-evidence answers.
- Block internal governance Q&A for `client` role by default.

## 12. Reasoning and Cross-Agent Logic

Governance intelligence should combine deterministic registers with optional LLM narration.

Example governance risk logic:

```text
IF open_blocking_dependencies > 0
   AND dependency_overdue_days > threshold
THEN elevate governance risk tier (Red/Amber/Green)
     AND include in weekly summary critical path
     AND optionally suggest escalation creation
```

Example scope logic:

```text
IF scope_status = pending_revision
   AND blocking_dependencies exist
THEN flag project in Governance Register as at-risk
```

Do not duplicate:

- Delivery confidence scoring → Delivery Performance Agent
- Quality drift detection → Quality Intelligence Agent
- Workforce overload → Workforce & Capability Agent

## 14. Governance Rules (Platform)

From BSG_OPS §12 AI Governance & Security Model and shared agent principles:

- **Evidence-backed AI:** every generated summary or agent answer must cite source rows or documents.
- **No hallucinated project status:** do not invent dependencies, escalations, or scope states.
- **RBAC:** role-scoped reads and writes; tenant isolation via `org_id` plus RLS.
- **Human-in-the-loop:** weekly governance summaries and charter approvals require DM review.
- **Audit logging:** resolve, create, approve, and escalate actions are auditable.
- **Synthetic data in non-prod:** dev/staging use seed data until governance sign-off for production.
- **Client visibility:** clients see only approved, client-safe escalation context for their projects.

## 16. Example Answer Pattern

Question:

```text
Which dependencies threaten timelines this week?
```

Expected behavior:

1. Retrieve open `project_dependencies` with `status = blocking` for visible projects.
2. Compute overdue days from `due_date`.
3. Optionally read open `risk_alerts` for same projects.
4. Rank by overdue severity and blocking impact.
5. Return answer with evidence links to dependency rows.
6. Recommend resolve/escalate actions only when supported by data.

Example answer structure:

```text
Three dependencies threaten timelines this week. The critical blocker is "Client schema v2 approval"
on Helios Docs (4 days overdue, client action). GPU quota uplift on Nimbus NLP is 2 days overdue
(external). Legal review on Vertex Finance is on track. Recommended action: escalate Helios schema
approval before Thursday's governance call.
```

## 19. Data Model Gaps

Governance requires new Phase 2 tables. Suggested minimum:

| Table | Purpose |
|---|---|
| `project_scope_states` | Scope status and version per project |
| `project_dependencies` | Dependency tracker |
| `governance_escalations` | Escalation register |
| `governance_actions` | Action tracking |
| `governance_weekly_summaries` | AI summary drafts + approvals |
| `governance_evidence_links` | Optional evidence for summaries/Q&A |

All tables must include:

- `org_id` for tenant isolation
- RLS policies
- `created_at` / `updated_at`
- Soft delete where appropriate
- `created_by` / `updated_by` for audit

Open design choice: reuse `mitigation_recommendations` for actions vs. separate `governance_actions` table.

## 21. Success Criteria

The Project Governance Agent is successful when:

- Delivery Managers can manage dependencies, escalations, and actions without spreadsheets.
- Governance Register reflects live per-project posture.
- Weekly governance summaries are evidence-backed and DM-approved.
- Agent answers cite governance rows, not invented projects.
- Cross-agent boundaries are respected (no duplicate delivery scoring).
- Clients never see internal governance detail beyond approved escalation context.
- Governance questions from the BSG_OPS appendix are answerable from the platform.

Source-aligned success metrics:

- Reduced PM administrative overhead on governance tracking
- Improved governance confidence (client and internal)
- Faster identification of timeline-threatening dependencies
