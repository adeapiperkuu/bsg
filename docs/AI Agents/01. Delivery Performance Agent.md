# Delivery Performance Agent - 

> This document defines the simplified, production-ready architecture for the Delivery Performance Agent inside the BSG AI-Native Operations Suite. It is a documentation-only architecture refactor. No backend code is included here.
>
> The RBAC model, security model, DB ownership rules, business requirements, and agent independence requirements remain unchanged.

---

## 0. Purpose and Scope

The Delivery Performance Agent monitors delivery execution signals, identifies delivery risks, predicts confidence against project milestones, detects operational bottlenecks, and produces a daily delivery summary for project dashboards.

It remains a feature module inside the shared platform, not a separate product or separate service.

| Shared platform logic | Delivery Agent-owned logic |
|---|---|
| Authentication | Delivery dashboard aggregation |
| Users and organisations | Delivery confidence scoring |
| RBAC and permissions | Delivery risk scoring |
| Project visibility and tenant isolation | Milestone delivery status logic |
| Notifications foundation | Bottleneck detection |
| Global DB/session setup | Daily summary payload and optional narrative |

The Delivery Agent must not import from the Quality Intelligence Agent or Client Interaction Agent. Cross-agent context may be read only through governed shared tables, never through direct calls into another agent module.

---

## 1. Product Surfaces

The refactored architecture supports the same eight Delivery Performance surfaces:

| # | Surface | Refactored approach |
|---|---|---|
| 1 | Project Dashboard | One dashboard endpoint aggregates all delivery data |
| 2 | Milestone Tracking | Milestone status is calculated and updated by delivery services |
| 3 | Delivery Confidence Score | Deterministic scoring over throughput, targets, trends, and milestones |
| 4 | Risk Score | Deterministic risk calculation over confidence, throughput trend, milestone timing, and bottlenecks |
| 5 | Resource Utilisation | Lightweight team/headcount aggregation only |
| 6 | Traffic Light Status | Derived by the scoring service, not stored and not a separate module |
| 7 | Bottleneck Detection | Deterministic signal detection over throughput and team patterns |
| 8 | Daily Operational Summary | Dashboard aggregation plus optional AI summary |

## 3. Dependency Flow

Use the simple application flow:

```text
Routes -> Services -> SQLAlchemy / DB Session
```

Allowed dependencies:

```text
delivery/routes
  -> delivery/services
  -> delivery/schemas

delivery/services
  -> delivery/analytics
  -> SQLAlchemy models + AsyncSession

delivery/services/summary_service.py
  -> delivery/ai/summary_service.py only when AI summary is enabled

delivery/ai
  -> delivery/prompts
  -> platform LLM client/config
```

Disallowed dependencies:

- Routes must not contain scoring or aggregation rules.
- Analytics modules must not import FastAPI, SQLAlchemy sessions, ORM models, or LLM clients.
- AI modules must not calculate confidence, risk, bottlenecks, or traffic-light status.
- Delivery Agent code must not call code inside other agent modules.
- Repository classes should not be added by default.

If a query becomes reused or complex, start with a small private service/helper function. Add a repository class only if it clearly removes meaningful complexity.

## 4. Data Model and Ownership

No new Delivery Agent tables are required for this refactor.

### 4.1 Delivery Agent Writes

| Table | Purpose |
|---|---|
| `throughput_snapshots` | Daily delivery throughput by project |
| `delivery_confidence_scores` | Historical confidence snapshots |
| `risk_alerts` | Delivery, milestone, and workforce imbalance risks |
| `bottlenecks` | Operational blockers |
| `milestones` | Milestone status updates where permitted by business rules |

### 4.2 Delivery Agent Reads

| Table | Purpose |
|---|---|
| `projects` | Project metadata, targets, dates, organisation scope |
| `project_assignments` | User-to-project visibility and assignment |
| `users` | Assignment and role context through platform services |
| `teams` | Team grouping for resource utilisation |
| `annotators` | Active headcount denominator where available |
| `metric_configurations` | Thresholds and configurable scoring cut-points |
| `quality_snapshots` | Optional read-only contribution to root-cause context |

The Delivery Agent does not own `users`, `organisations`, authentication, RBAC, or global project visibility rules.

## 5. Project Assignments

Do not assume one user maps to one project.

The relationship is:

```text
users -> project_assignments -> projects
```

This supports:

- One user assigned to many projects.
- One project assigned to many users.
- Delivery managers viewing multiple active projects.
- Clients viewing only projects they are allowed to access.
- Leadership and Super Admin access according to existing RBAC rules.

Routes must never trust a request body or path parameter as proof of project access. The service should query project delivery data only after platform scoping confirms the current user can access the requested project.

## 6. Core Business Logic

Thresholds should come from `metric_configurations` where supported. Values below are defaults, not hardcoded constants.

### 6.1 Confidence

Confidence is deterministic and computed from:

- Latest throughput snapshot.
- Rolling 7-day throughput.
- Project daily target.
- Milestone planned date.
- Recent throughput trend.

The scoring service owns:

```text
calculate_confidence()
```

Output:

- `score_pct`
- `forecast_completion_date`
- confidence status
- optional persisted `delivery_confidence_scores` row during scoring runs

### 6.2 Risk

Risk is deterministic and based on:

- Delivery confidence trend.
- Throughput decline.
- Days remaining until milestone.
- Open high-severity bottlenecks.
- Optional quality drift signal read from shared governed data.

The scoring service owns:

```text
calculate_risk()
```

The output maps to `risk_alerts.slippage_probability`, `risk_alerts.risk_tier`, and structured contributing causes.

### 6.3 Traffic Light Status

Traffic light status is not stored and does not live in a separate `traffic_light.py` module.

The scoring service owns:

```text
calculate_status()
```

Default mapping:

```text
GREEN:
  confidence >= on_track_threshold
  and no open high/critical delivery risks
  and current milestone is not missed

AMBER:
  confidence < on_track_threshold
  or any open medium/high delivery risk
  or one or more open bottlenecks

RED:
  any open critical delivery risk
  or current milestone is missed
  or confidence is below the configured critical threshold
```

Precedence is red first, then amber, then green.

### 6.4 Milestone Status

Milestone status remains a delivery business rule:

```text
pending -> on_track
  when planned date is upcoming and confidence is acceptable

pending/on_track -> at_risk
  when confidence falls below the configured threshold

any -> completed
  when actual_date is set

on_track/at_risk -> missed
  when planned_date passes without completion, or actual_date is after planned_date
```

Status transitions should happen in the scoring service as part of confidence/risk calculation, not inside routes.

### 6.5 Bottlenecks

Bottleneck detection remains deterministic:

```text
Candidate signal:
  A team's contribution to project throughput drops below its historical share
  for a configured number of days without a matching headcount drop.

Output:
  Open bottleneck row with project, optional team, title, detail, and status.
```

Bottlenecks remain distinct from risk alerts:

- A bottleneck is an observed operational blocker.
- A risk alert is a probabilistic delivery concern.

### 6.6 Resource Utilisation

Resource utilisation stays lightweight for Phase 1.

The Delivery Agent may report:

- Active headcount by team.
- Relative team allocation across a project.
- Workforce imbalance signals where the existing schema supports them.

It must not expand into skill matching, certification tracking, SME coverage, or full workforce planning. Those belong to the future Workforce & Capability Agent.

## 7. Dashboard API

Use one primary dashboard endpoint:

```text
GET /delivery/dashboard/{project_id}
```

The endpoint returns one aggregated payload:

```text
{
  "overview": { ... },
  "milestones": [ ... ],
  "confidence": { ... },
  "risks": [ ... ],
  "bottlenecks": [ ... ],
  "traffic_light": { ... },
  "daily_summary": { ... }
}
```

The response should include enough structured data for the frontend to render the dashboard without issuing multiple delivery-specific reads.

### 7.1 Dashboard Request Flow

```text
Frontend
  -> GET /delivery/dashboard/{project_id}
  -> delivery/routes/dashboard.py
  -> validate current user and project access
  -> delivery/services/dashboard_service.py
  -> SQLAlchemy queries using AsyncSession
  -> delivery/services/scoring_service.py
       calculate_confidence()
       calculate_risk()
       calculate_status()
  -> delivery/services/summary_service.py
  -> optional delivery/ai/summary_service.py
  -> DashboardResponse schema
```

The dashboard service should aggregate:

- Project overview.
- Current and upcoming milestones.
- Latest confidence and confidence history needed by the UI.
- Open risks grouped by tier.
- Open bottlenecks.
- Traffic-light status.
- Daily structured summary.
- Optional AI-generated daily summary narrative.

### 7.2 Supporting Endpoints

The single dashboard endpoint is the main UI read path. Supporting endpoints may still exist for focused workflows:

- Throughput ingestion.
- Risk alert acknowledgement/resolution.
- Bottleneck acknowledgement/resolution.
- Milestone create/update where RBAC permits.

These supporting endpoints should not duplicate dashboard aggregation logic.

## 8. Daily Summary

Daily Summary is simplified to:

```text
Dashboard request -> Aggregate data -> Optional AI summary
```

No extra orchestration layer is needed.

The structured daily summary includes:

- Latest throughput and delta versus previous snapshot.
- Throughput versus forecast/target.
- Risks created or escalated today.
- Bottlenecks opened or resolved today.
- Milestones due soon.
- Current traffic-light status.

The structured payload is authoritative. The AI summary is optional presentation text and must not introduce facts that are not present in the structured payload.

If the LLM provider is not configured, the dashboard still returns the structured daily summary.

## 9. Analytics and AI Separation

### 9.1 `analytics/`

`analytics/` is deterministic only.

It may contain:

- Throughput trend calculations.
- Confidence formula helpers.
- Risk formula helpers.
- Bottleneck signal helpers.
- Date/window utilities for delivery metrics.

It must not contain:

- LLM calls.
- Prompt text.
- Natural-language generation.
- FastAPI routes.
- SQLAlchemy sessions.
- ORM persistence.

Given the same inputs, `analytics/` must return the same outputs every time.

### 9.2 `ai/`

`ai/` is for summarization only. Prompt templates live in `prompts/`.

It may contain:

- Prompt loading.
- Prompt assembly from already-scoped structured data.
- LLM calls through the platform LLM client.
- LLM response validation.

It must not contain:

- Confidence scoring formulas.
- Risk scoring formulas.
- Bottleneck detection rules.
- Database queries.
- Permission checks.

The AI layer receives already-authorized, already-aggregated data from services. It never performs cross-tenant retrieval and never bypasses platform access controls.

## 10. Event Flow

The event flow is conceptual only. Do not implement an event bus for Phase 1.

```text
ThroughputUpdated
  -> ConfidenceCalculated
  -> RiskCalculated
  -> DashboardUpdated
```

This describes the lifecycle of delivery data:

1. A throughput snapshot is created or updated.
2. Confidence is recalculated from the latest delivery data.
3. Risk is recalculated from confidence, throughput trend, milestone timing, and bottlenecks.
4. Dashboard reads reflect the updated state.

Implementation can be synchronous during ingestion, scheduled through a simple job, or triggered manually by an internal admin action depending on the platform scheduler decision. The architecture does not require event infrastructure.

## 11. Security and RBAC

This refactor does not change RBAC, security model, tenant isolation, or DB ownership rules.

Delivery routes must continue to use:

- Existing authentication dependencies.
- Existing role checks.
- Existing project visibility rules.
- Existing organisation scoping.
- Existing database session and RLS guarantees.
- Existing rules for service-role usage.

Client-safe response shaping remains required:

- Client users may see project delivery status, confidence, risks, bottlenecks, and summary fields that are safe for their organisation.
- Client users must not see internal workforce detail, annotator-level data, or internal staffing narratives.
- Delivery managers may see operational delivery detail for projects in their organisation.
- Leadership and Super Admin retain their existing access patterns.

No new roles are introduced.

---

## 12. Transactions and Persistence

Services own transaction boundaries.

Write operations that must stay consistent should run in one transaction:

- Throughput upsert plus dependent scoring when executed synchronously.
- Confidence score insert plus milestone status update.
- Risk alert creation plus notification creation.
- Bottleneck creation plus notification creation.

Read-only dashboard aggregation should not create database rows except when an explicitly enabled AI summary audit trail requires it under existing platform evidence rules.

Traffic-light status and daily summary are derived outputs and should not be persisted as source-of-truth rows.

## 13. Testing Strategy

Keep testing focused and practical for a four-developer team.

Priority tests:

- Unit tests for `analytics/` formula helpers.
- Unit tests for `calculate_confidence()`, `calculate_risk()`, and `calculate_status()`.
- Service tests for dashboard aggregation with a seeded DB session.
- Permission tests for cross-organisation dashboard access.
- Client-safe shaping tests for dashboard and daily summary responses.
- Regression tests for milestone status transitions.

Avoid heavyweight test infrastructure unless the feature grows beyond the Phase 1 scope.

## 15. Key Improvements

This refactor improves the architecture by:

- Removing the repository layer and using the simpler `Routes -> Services -> SQLAlchemy / DB Session` flow.
- Moving Delivery Agent code into a self-contained `backend/app/agents/delivery/` module.
- Keeping platform concerns outside the agent instead of duplicating auth, users, RBAC, or project visibility logic.
- Replacing multiple dashboard read paths with one `GET /delivery/dashboard/{project_id}` endpoint.
- Moving traffic-light logic into the scoring service through `calculate_status()`.
- Simplifying Daily Summary to dashboard aggregation plus optional AI narrative.
- Separating deterministic analytics from LLM-based summarization.
- Supporting many-to-many user/project relationships through `project_assignments`.
- Keeping event flow conceptual without introducing an event bus.
- Preserving production readiness while reducing implementation overhead for a small team.
