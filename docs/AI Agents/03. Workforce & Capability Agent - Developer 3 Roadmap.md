# Workforce & Capability Agent - Developer 3 Roadmap

> Owner: Developer 3  
> Product: BSG Operations Tower / AI-Native Operations Suite  
> Agent: Workforce & Capability Agent  
> Status: Implementation planning document  
> Scope: Independent workstream for skills tracking, competency matrix, resource allocation, training, capability gaps, and utilization.

---

## 1. Executive Summary

The Workforce & Capability Agent helps BSG understand whether the right people, skills, certifications, and training capacity are available for each delivery project.

Delivery Performance answers: "Are projects on track?"  
Quality Intelligence answers: "Is quality stable?"  
Workforce & Capability answers: "Do we have the right workforce capability to deliver the work?"

This agent is independent from the other agents because it can be implemented around its own workforce data model and dashboards. It will integrate later with Delivery Performance and Quality Intelligence, but it does not need them to be finished first.

Primary capabilities:

- Employee / annotator profiles
- Skills taxonomy
- Competency matrix
- Certification tracking
- Training records
- Capability gap detection
- Resource allocation recommendations
- Skill coverage dashboard
- Utilization dashboard

---

## 2. Product Purpose

BSG delivery teams operate across India and Kosovo and work on projects that require different domain capabilities: Life Sciences, Finance, Insurance, Logistics, computer vision, NLP, medical review, QA review, and other specialized data operations.

Without a Workforce & Capability Agent, resource planning is likely to remain spreadsheet-driven:

- Which team has available capacity?
- Who is certified for a specific project type?
- Which skills are missing for upcoming work?
- Which people need training?
- Which projects depend on too few SMEs?
- Which teams are overloaded or underused?

The agent turns this into a governed operational layer inside the Operations Tower.

## 5. Independence Boundary

This agent is fully independent for implementation.

It owns:

- Workforce data
- Employee/annotator profiles
- Skills and competencies
- Certifications
- Training tracking
- Utilization tracking
- Workforce dashboards
- Capability gap logic
- Resource allocation recommendations

It can reuse:

- Existing auth, RBAC, and RLS foundation
- Existing `teams` and `annotators` tables
- Existing project/org model
- Existing dashboard UI patterns
- Existing `risk_alerts` enum value: `workforce_imbalance`
- Existing `mitigation_recommendations` pattern for recommended actions

It should not own:

- Delivery milestone confidence scoring
- Quality metric calculations
- Client-facing reporting
- Knowledge document ingestion

Integration points later:

- Delivery Agent can consume workforce imbalance alerts.
- Quality Agent can feed training recommendations from recurring quality error categories.
- Client Interaction Agent may use high-level workforce confidence, but never individual employee details.

## 6. Workflow Alignment With Other Agents

This section compares the Workforce workflow with the other agent workflows and confirms the correct project flow.

### 6.1 Correct Agent Flow

The Workforce & Capability Agent should produce workforce and capability signals that other agents can consume through shared database tables.

Recommended flow:

```text
Projects, Teams, Annotators
        |
        v
Workforce data:
  - utilization
  - skills
  - certifications
  - training records
  - project skill requirements
        |
        v
Workforce Agent computes:
  - skill coverage
  - SME coverage
  - utilization risk
  - training gaps
  - capability gaps
  - resource recommendations
        |
        v
Shared operational outputs:
  - capability_gaps
  - risk_alerts where alert_type = workforce_imbalance
  - mitigation_recommendations
  - dashboard summaries
        |
        v
Other agents read these outputs as scoped evidence
```

The agent should not call code inside another agent module directly. Cross-agent communication should happen through governed shared tables and APIs.

### 6.2 Comparison With Delivery Performance Agent

Delivery owns:

- Milestones
- Throughput
- Delivery confidence
- Delivery risk
- Bottlenecks
- Delivery dashboard summary

Workforce owns:

- Workforce profiles
- Utilization history
- Skill coverage
- SME coverage
- Certifications
- Training records
- Capability gaps
- Resource allocation recommendations

Boundary:

- Delivery may show that delivery risk is affected by capacity or workforce imbalance.
- Workforce owns the detailed calculation of why the workforce is overloaded, underutilized, under-skilled, or missing certifications.
- Delivery should not implement skill matching, certification tracking, or training tracking.
- Workforce should not calculate milestone confidence or delivery slippage probability.

Correct integration:

```text
Workforce detects overloaded team
  -> writes risk_alerts.alert_type = workforce_imbalance
  -> optionally writes mitigation_recommendations
  -> Delivery dashboard may show that risk as part of project risk context
```

### 6.3 Comparison With Quality Intelligence Agent

Quality owns:

- Gold-set accuracy
- IAA
- Rework rate
- Quality drift
- Error taxonomy
- Quality corrective action recommendations

Workforce owns:

- Whether the right people have the skills/training to fix those quality problems
- Which employees or teams need training
- Whether certification or SME coverage is missing

Boundary:

- Quality can say "boundary precision errors are increasing; calibration is recommended."
- Workforce should say "these teams/employees are missing the required calibration/training coverage."
- Quality should not manage employee training records.
- Workforce should not calculate quality drift.

Correct integration:

```text
Quality detects recurring error category
  -> writes quality_error_entries.recommended_action
  -> Workforce maps that recommendation to training_programs and training_records
  -> Workforce surfaces training gaps and coverage risk
```

### 6.4 Comparison With Client Interaction Agent

Client Interaction owns:

- Client-safe status summaries
- Weekly/executive communication drafts
- Client Q&A
- Delivery confidence explanation
- Client-safe risk narratives
- Human approval workflow

Workforce owns:

- Internal workforce evidence
- Employee/annotator detail
- Skill and utilization calculations

Boundary:

- Client Interaction must not expose employee names, staffing details, individual utilization, or internal training gaps to client users.
- Client Interaction may consume only aggregated workforce confidence, such as "SME coverage is sufficient" or "capacity risk has been mitigated", when approved as client-safe.
- Workforce should not generate client-facing narratives.

Correct integration:

```text
Workforce produces aggregated workforce status
  -> Client Interaction reads only client-safe summary fields
  -> Delivery Manager reviews any client-facing language before publish
```

### 6.5 Comparison With Operational Knowledge Agent

Knowledge owns:

- SOPs
- Guides
- Training documents
- Lessons learned
- Project charters
- Document retrieval and citations

Workforce owns:

- Training assignments
- Training completion
- Required training per skill
- Employee capability status

Boundary:

- Knowledge stores or retrieves training material.
- Workforce tracks who needs the training and who completed it.
- Workforce should not create a separate document storage system for training files if Knowledge document storage is available.

Correct integration:

```text
Knowledge stores approved training document
  -> Workforce training_programs can reference it
  -> Workforce tracks completion in training_records
```

### 6.6 Confirmed Structure

The Workforce structure is correct with one important adjustment: implementation must be data-first, not AI-first.

Correct sequence:

1. Workforce foundation: teams and annotators CRUD.
2. Utilization tracking.
3. Live workforce dashboard.
4. Skills and competency matrix.
5. Certifications and training records.
6. Capability gaps.
7. Workforce recommendations.
8. Workforce Agent Q&A with evidence links.

This keeps the agent independent while still aligned with the overall project flow.

## 9. Backend Features

### 9.1 Workforce CRUD

Implement endpoints for:

- List/create/update/delete teams
- List/create/update/delete annotators
- Get employee/annotator profile
- Attach skills to employee
- Attach certifications to employee
- Attach training records to employee

The existing API spec already includes teams and annotators:

- `GET /projects/{project_id}/teams`
- `POST /projects/{project_id}/teams`
- `PATCH /teams/{team_id}`
- `DELETE /teams/{team_id}`
- `GET /teams/{team_id}/annotators`
- `POST /teams/{team_id}/annotators`
- `PATCH /annotators/{annotator_id}`
- `DELETE /annotators/{annotator_id}`

### 9.2 Skills Matrix API

Needed endpoints:

- `GET /workforce/skills`
- `POST /workforce/skills`
- `PATCH /workforce/skills/{skill_id}`
- `DELETE /workforce/skills/{skill_id}`
- `GET /workforce/skills-matrix`

The skills matrix should return skills by site, team, and proficiency.

Example response shape:

```json
{
  "skills": [
    {
      "skill_id": "uuid",
      "skill_name": "Radiology QA",
      "category": "Life Sciences",
      "india_count": 8,
      "kosovo_count": 4,
      "expert_count": 5,
      "coverage_status": "covered"
    }
  ]
}
```

### 9.3 Utilization API

Needed endpoints:

- `GET /workforce/utilization`
- `POST /teams/{team_id}/utilization`
- `GET /workforce/utilization/summary`

Dashboard metrics:

- Teams over 85% allocation
- Teams below 60% allocation
- Average utilization by site
- Average utilization by project
- Capacity risk by team

### 9.4 Capability Gap Engine

The gap engine should be deterministic first.

Inputs:

- Project skill requirements
- Annotator skills
- Certifications
- Training records
- Utilization snapshots

Logic:

1. For each active project, load required skills.
2. Count available employees with matching skill and required proficiency.
3. Count available SMEs for critical skills.
4. Check certification validity.
5. Check required training completion.
6. Check utilization threshold.
7. Create or update capability gap records.

Example:

```text
Project requires 5 advanced Radiology QA annotators and 2 SMEs.
Available: 3 advanced annotators and 1 SME.
Create:
- skill_shortage, required 5, available 3, severity high
- sme_shortage, required 2, available 1, severity high
```

### 9.5 Recommendations

The agent should recommend actions such as:

- Move available certified annotators from underutilized team to overloaded team.
- Schedule missing training for specific employees.
- Add SME coverage before milestone start.
- Reduce allocation on overloaded team.
- Escalate critical skill shortage to delivery manager.

MVP can store these as `mitigation_recommendations` or a new workforce-specific table. Reusing `mitigation_recommendations` is faster and consistent with the current platform.

### 9.6 Agent Q&A

Later, add `workforce_capability_agent` to `/agent-queries`.

Questions:

- "Which teams are overloaded this week?"
- "Do we have enough SMEs for Project Alpha?"
- "Who is certified for clinical NLP?"
- "What training gaps exist for Kosovo teams?"
- "Which capability gaps are critical?"

Rules:

- Delivery managers can ask about their own org.
- Leadership can ask portfolio-level questions if approved by product/security.
- Clients cannot access workforce Q&A.
- Every answer must have evidence links.

Access rules:

| Role | Workforce access |
|---|---|
| `client` | No access to employee/annotator details |
| `delivery_manager` | Read/write workforce data in own org |
| `bsg_leadership` | Read-only portfolio workforce view |

- Every new table must include `org_id`.
- Every new table must have RLS policies.
- Do not expose annotator names to clients.

- Soft-delete mutable workforce entities.
- Add tests proving client users cannot access workforce endpoints.
