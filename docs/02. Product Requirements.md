## 1. Product overview

### 1.1 Vision

Build a unified AI-powered operational intelligence layer on top of BSG's existing managed data-labeling / data-operations delivery business. The platform transforms fragmented operational data (spreadsheets, QA logs, dashboards, SOP repos) into a governed single source of truth and uses specialized AI agents to predict delivery risk, monitor quality drift, and generate transparent, evidence-backed client reporting.

**Core objective:** Enable BSG teams and clients to interact with operational intelligence through AI-driven, evidence-backed, role-based experiences.

### 1.2 Strategic positioning

| Today | Future state |
|---|---|
| High-quality data operations and labeling services provider | AI-native operations partner powered by intelligent operational agents |
| Manual reporting, spreadsheet-driven governance | Real-time operational intelligence and unified AI-driven decision support |
| Reactive issue handling | Predictive risk identification |
| PM-dependent responses | AI-assisted, evidence-backed insights |
| Static dashboards | Conversational operational intelligence |
| Delayed client updates | Real-time transparency and insights |

BSG's differentiation is **not** annotation automation. The product augments the operational layer *around* human annotation and SME review work — it does not replace it.

### 1.3 Problem statement

Data operations programs consistently surface these pain points that this product must address:

- **Operational complexity** — multiple stakeholders, distributed teams (India, Kosovo), dynamic quality requirements, tight timelines.
- **Fragmented operational data** — intelligence spread across Excel trackers, QA logs, workforce sheets, Power BI dashboards, and SOP repositories.
- **High PM overhead** — substantial effort on status reporting, resource allocation, risk analysis, and client updates.
- **Inconsistent visibility** — different teams use different metrics and versions of operational truth.
- **Rising client expectations** — real-time visibility, predictive insights, quality traceability, and governance maturity (especially in Life Sciences, Finance/Insurance, and Logistics).

## 2. Goals and non-goals

### 2.1 Goals (Phase 1 / MVP)

1. Ship the **Phase‑1 trio** of agents: Delivery Performance, Quality Intelligence, and Client Interaction.
2. Provide a role-scoped **Operations Tower** web experience for internal delivery staff and client users.
3. Unify delivery and quality operational data into a governed, tenant-isolated data foundation.
4. Ensure every AI-generated insight and client-facing narrative is **evidence-backed** and auditable.
5. Enforce **human-in-the-loop** approval for all outbound client communications.
6. Support **admin-configurable client-facing metrics** without code changes.

### 2.2 Non-goals (explicitly out of scope for Phase 1)

| Area | Rationale |
|---|---|
| Automating human annotation / SME review work | Product is an operational intelligence layer around delivery, not a replacement for labelers or reviewers |
| Workforce & Capability Agent (Agent 3) | Full skill matching, SME coverage, utilization — Phase 2; limited workforce-rebalancing *recommendations* remain in MVP |
| Project Governance Agent (Agent 4) | Charter generation, scope/dependency tracking — Phase 2 |
| Operational Knowledge Agent (Agent 6) | SOP retrieval, lessons-learned search — Phase 2 |
| Full Regulatory-Grade Quality Layer | Scorecards, dataset versioning, post-deployment monitoring as a cross-cutting system — Phase 2 |
| Automation engineer tooling as a product feature | Backlog management, live-automation counts — not Phase 1 |
| Team Health Score / weekly voice check-ins | Feeds Phase 2+ agents |
| BSG Leadership cross-client portfolio analytics | Margin, unit economics, automation ROI, churn early-warning — depends on Phase 2+ data |
| Vertical-specific service-pillar workflows | Medical AI evaluation, financial crime validation, logistics exception coding — Phase 3 |
| Predictive/advisory forecasting beyond basic throughput forecast and risk scoring | Phase 4 |
| Autonomous multi-agent orchestration and self-improving recommendation engine | Future per Delivery Performance Agent BRD |
| Multi-client portfolio intelligence, AI relationship sentiment, predictive client-escalation detection | Future per Client Interaction Agent BRD |
| Project readiness / go-live readiness assessment engine | Not in Phase‑1 MVP feature list; confirm before building (see Section 12) |
| Live client production data before governance sign-off | Build/test uses synthetic or sanitized data only |

## 3. Target users and personas

### 3.1 Personas

| Persona | Side | Primary need | Phase‑1 platform role |
|---|---|---|---|
| Project Manager (PM) | Internal | Daily prioritization; know which projects are at risk today | `delivery_manager` |
| Delivery Lead / Delivery Manager | Internal | Portfolio governance; approve AI-drafted client communications | `delivery_manager` |
| Operations Manager | Internal | Balance workforce/throughput across teams | `delivery_manager` |
| QA Lead | Internal | Quality-driven delivery risk before it becomes a client issue | `delivery_manager` |
| Resource Manager | Internal | SME allocation and utilization decisions | `delivery_manager` |
| Program / BSG Leadership (Braunschweig HQ) | Internal | Cross-client portfolio view (limited in Phase 1) | `bsg_leadership` |
| Client Stakeholder / Client Program Manager | External | Own-project delivery health without waiting on PM reporting | `client` |
| Client Leadership | External | Executive confidence with auditable evidence | `client` |
| Super Admin | Internal | System-wide configuration (client-facing metrics, platform settings) | `super_admin` |
| Automation Engineer (2 planned) | Internal | Converts repetitive ops work into tooling — not a Phase‑1 product user | N/A (Phase 2+) |

### 3.2 Access rules (hard requirements)

- **Tenant isolation:** Client-side users must only ever see their own organisation's projects and data. Cross-tenant access must be impossible at the database layer (PostgreSQL RLS), not only at the application layer.
- **Role-based access:** Each role sees and can do only what its permissions allow. AI responses must be permission-aware.
- **Super Admin scope:** Platform-wide configuration only (e.g., which metrics appear on client dashboards). Distinct from Delivery Manager, who operates within client engagements. Super Admin bypasses RLS for config tables only — not client data tables (pending confirmation).
- **Client communication gate:** 100% of AI-drafted outbound client communications require explicit human Delivery Manager review and approval before sending.

## 4. Design principles

1. **Human-centric AI** — AI augments operational teams; it does not replace scientific reviewers, SMEs, or annotation specialists.
2. **Single source of truth** — All operational intelligence is grounded in governed, validated data (unified data foundation / KPI semantic layer).
3. **Evidence-backed AI** — Every AI-generated insight must link to the specific source data it was derived from. No insight may be presented without a citable basis. No hallucinated project status.
4. **Role-based intelligence** — Tailored insights per role and permissions.
5. **Operational explainability** — Insights explain what happened, why, the impact, recommended actions, and confidence level.
6. **Security and governance first** — RBAC, permission-aware AI, audit logging, human-in-the-loop validation, governance approval workflows, and data lineage tracking are foundational.

## 5. Platform foundation requirements

These capabilities are prerequisites for all Phase‑1 agents.

### 5.1 Authentication and authorisation

| ID | Requirement | Priority |
|---|---|---|

| PF-04 | Dedicated end-to-end tests prove a `client` user cannot retrieve another organisation's data. | Must |
| PF-05 | All non-production environments use synthetic or sanitized data only. | Must |

### 5.2 Operations Tower (web application)

| ID | Requirement | Priority |
|---|---|---|
| PF-06 | Role-scoped web UI ("Operations Tower") surfaces agent outputs to Client, Delivery Manager, and BSG Leadership users. | Must |

| PF-08 | Delivery Managers see all teams and projects within their organisation across Phase‑1 agents; can review and approve AI-drafted client communications from this view. | Must |
| PF-09 | Live dashboards support rolling daily throughput, % on-time milestones, and auto-flagged bottlenecks for active projects. | Must |
| PF-10 | Charts for throughput forecast vs. actual and quality drift trends (by team, by week). | Must |

### 5.3 Super Admin configuration

| ID | Requirement | Priority |
|---|---|---|
| PF-11 | Super Admin configures, system-wide, which metrics are surfaced on client-facing summaries/dashboards. | Must |

### 5.4 Evidence and audit trail

| ID | Requirement | Priority |

| PF-16 | Every AI-drafted client communication must have at least one evidence link (`communication_evidence_links`). | Must |

### 5.5 Notifications

| ID | Requirement | Priority |
|---|---|---|

## 6. Agent requirements — Delivery Performance Agent

**Mission:** *"Continuously monitor delivery execution signals, identify operational risks, predict delivery outcomes, recommend mitigation strategies, and generate prioritized operational action plans for delivery leadership."*

### 6.1 User stories

| ID | As a… | I want to… | So that… |
|---|---|---|---|
| DP-01 | PM / Delivery Lead | View a live dashboard of active projects with rolling daily throughput, % on-time milestones, and auto-flagged bottlenecks | I can prioritise daily without manually consolidating spreadsheets |
| DP-02 | PM | See a throughput forecast-vs-actual chart with a schedule-confidence score per project | I can assess whether milestones are likely to be met |
| DP-03 | PM / Delivery Lead | Receive a proactive alert when delay risk crosses a threshold, with slippage probability and contributing causes | I can intervene before issues escalate to the client |
| DP-04 | PM | Ask a natural-language question (e.g., "which projects are at risk?") and get an evidence-backed answer | I get immediate decision support without building reports |
| DP-05 | PM / Operations Manager | Receive a workforce rebalancing recommendation between over- and under-utilised teams on a project | I can redistribute workload before throughput drops |

### 6.2 Functional requirements

| ID | Requirement | Priority |
|---|---|---|
| DP-F01 | Track milestones with planned vs. actual dates and lifecycle status (`milestones`). | Must |
| DP-F02 | Record daily throughput snapshots per project: units completed, forecast, rolling 7-day units (`throughput_snapshots`). | Must |
| DP-F03 | Compute and store schedule-confidence / delivery-confidence scores per milestone as a time-series (`delivery_confidence_scores`). | Must |
| DP-F04 | Auto-detect and flag workflow bottlenecks (`bottlenecks`), distinct from probabilistic risk alerts. | Must |
| DP-F05 | Generate risk alerts with risk tier (low / medium / high / critical), slippage probability, and contributing causes as structured data (`risk_alerts`). | Must |
| DP-F06 | Root-cause intelligence quantifies contributors (e.g., absenteeism, rework, review turnaround) in `contributing_causes` JSONB. | Must |
| DP-F07 | Natural-language query interface for delivery questions, grounded in operational data via RAG. | Must |
| DP-F08 | Delivery forecasting based on throughput trends and milestone plans. | Must |
| DP-F09 | Recommend workforce rebalancing when utilisation imbalance is detected (limited MVP scope — not full Workforce Agent). | Should |
| DP-F10 | Set milestone status to `at_risk` when latest delivery-confidence score falls below the configured threshold (draft: 80%). | Must |

### 6.3 Data inputs

**Structured:** Delivery tracker, milestone plan, throughput logs, resource allocation, QA/rework sheets, risk register, backlog queue.

**Unstructured (Phase 1 via RAG):** SOPs, project charters, escalation notes, PM notes.

### 6.4 Reasoning logic (illustrative)

IF throughput declines AND rework increases AND absenteeism spikes → raise risk score, forecast delay probability, generate mitigations, trigger escalation alert. Output risk tiers: Low / Medium / High / Critical.

### 6.5 Outputs

Action plans, confidence scores, risk summaries, throughput forecasts, workforce recommendations, escalation reports.

### 6.6 Out of scope (this agent)

Autonomous workflow orchestration, multi-agent collaboration, self-improving recommendation engine.

## 7. Agent requirements — Quality Intelligence Agent

**Provides:** Error trend analysis, quality drift monitoring, reviewer consistency insights, rework analysis, root-cause intelligence, and training recommendation triggers.

**Answers:** Is quality impacting delivery velocity? Which error categories are increasing?

### 7.1 User stories

| ID | As a… | I want to… | So that… |
|---|---|---|---|
| QI-01 | QA Lead / PM | View a quality dashboard with gold-set accuracy, inter-annotator agreement, rework rate, and drift alerts by team and by week | I can see quality-driven delivery risk before it becomes a client issue |
| QI-02 | QA Lead / PM | See a root-cause breakdown of quality errors with recommended corrective actions | I can trigger targeted calibration or SOP updates |

### 7.2 Functional requirements

| ID | Requirement | Priority |
|---|---|---|
| QI-F01 | Record weekly quality snapshots per project/team: gold-set accuracy %, IAA (Krippendorff's α), rework rate % (`quality_snapshots`). | Must |
| QI-F02 | Flag quality drift alerts when metrics breach configured thresholds (`has_drift_alert`, `drift_alert_detail`). | Must |
| QI-F03 | Store structured error taxonomy entries per snapshot with category, share %, and recommended action (`quality_error_entries`). | Must |
| QI-F04 | Error categories include at minimum: boundary precision, class confusion, guideline ambiguity (extensible). | Must |
| QI-F05 | Trigger `quality_drift` risk alerts when thresholds are breached. | Must |
| QI-F06 | Surface quality metrics on client-facing dashboards when enabled via `metric_configurations`. | Must |
| QI-F07 | Quality data granularity: weekly per team for Phase 1 (finer granularity is an open question). | Must |

### 7.3 Draft quality thresholds (WIP — not contractual)

| Metric | Draft threshold | Direction |
|---|---|---|
| Gold-set accuracy | ≥ 95% | Below triggers drift alert |
| Inter-annotator agreement (Krippendorff's α) | ≥ 0.85 | Below triggers drift alert |
| Rework rate | ≤ 5% | Above triggers drift alert |
| On-time milestone delivery | ≥ 90% | Portfolio KPI |
| Schedule confidence (per milestone) | ≥ 80% | Below = "at risk" |

Thresholds must be read from `metric_configurations` at runtime, not hardcoded.

### 7.4 Out of scope (this agent)

Full regulatory-grade quality layer (scorecards, dataset versioning, post-deployment monitoring), annotator-level daily tracking (unless confirmed — see Section 12).

## 8. Agent requirements — Client Interaction Agent

**Mission:** *"Continuously translate operational delivery intelligence into transparent, explainable, confidence-based client narratives and actionable engagement insights."*

### 8.1 User stories

| ID | As a… | I want to… | So that… |
|---|---|---|---|
| CI-01 | Delivery Manager | Review and approve an AI-drafted weekly/executive client status summary before it is sent | Clients receive consistent, accurate updates without hallucinated status |
| CI-02 | Client user | See a delivery-confidence score for the current milestone | I have executive-level confidence the program is on track |
| CI-03 | Client user | Ask a question about my own project status and get an answer without waiting on a PM | I get real-time transparency |
| CI-04 | Client user | Log into the Operations Tower and see only my own projects | I trust the data is scoped to my engagement |
| CI-05 | Client user | Submit a CSAT score for a project | BSG can track rolling client satisfaction |

### 8.2 Functional requirements

| ID | Requirement | Priority |
|---|---|---|
| CI-F01 | Auto-draft client status summaries (weekly, executive, ad hoc) from operational data (`client_communications`). | Must |
| CI-F02 | Communication lifecycle: `draft` → `in_review` → `approved` → `sent` (or `rejected`). | Must |
| CI-F03 | Only the backend service account may INSERT communication drafts; Delivery Managers UPDATE status and approval fields only. | Must |
| CI-F04 | A communication may transition to `sent` only if `approved_by` is set and that user has `delivery_manager` role. | Must |
| CI-F05 | Clients see only communications with `status = 'sent'`. | Must |
| CI-F06 | Generate delivery-confidence scores with evidence-backed narrative. | Must |
| CI-F07 | Client conversational interface for natural-language project-status questions, scoped to the client's own org/projects. | Must |
| CI-F08 | Risk transparency: translate operational issues into business language with mitigation progress. | Must |
| CI-F09 | Collect client CSAT scores (1–5 scale) per project (`client_csat_scores`). | Should |
| CI-F10 | All narratives evidence-backed; confidence scoring mandatory; sensitive data protected per RBAC. | Must |

### 8.3 Data inputs

**Structured:** Delivery tracker, milestone plan, throughput logs, QA/rework sheets, resource allocation, risk registers.

**Unstructured:** Client SOPs, training documents, project charters, escalation notes.

### 8.4 Outputs

Project health summaries, executive status reports, delivery-confidence scores, risk narratives.

### 8.5 Out of scope (this agent)

Project readiness / go-live readiness assessment engine, multi-client portfolio intelligence, AI relationship sentiment analysis, predictive client-escalation detection.

## 9. Data model requirements

The authoritative schema is *10. DB Schema.md*. Product-relevant entities and their purpose:

| Entity | Product purpose |
|---|---|
| `organisations` | Tenant root; client isolation boundary |
| `users` | Platform users with role and org membership |
| `projects` | Delivery engagements scoped to one organisation |
| `milestones` | Planned delivery checkpoints |
| `throughput_snapshots` | Daily delivery performance data |
| `teams` / `annotators` | Delivery workforce (site: India / Kosovo) |
| `quality_snapshots` / `quality_error_entries` | Weekly quality metrics and error taxonomy |
| `risk_alerts` | Threshold-crossing delivery and quality events |
| `bottlenecks` | Auto-detected workflow blockers |
| `client_communications` / `communication_evidence_links` | AI-drafted client updates with audit trail |
| `agent_queries` / `agent_query_evidence_links` | NL query log with evidence |
| `delivery_confidence_scores` | Per-milestone confidence time-series |
| `client_csat_scores` | Rolling client satisfaction |
| `metric_configurations` | Super Admin–managed client-facing metric set |
| `notifications` | In-app and email alert delivery |

**Business rules enforced at application layer (not schema alone):**

1. Client communications require Delivery Manager approval before `sent`.
2. Agent queries and communication drafts require ≥ 1 evidence link per transaction.
3. Delivery-confidence threshold drives milestone `at_risk` status.
4. Quality metric breaches trigger drift alerts and `quality_drift` risk alerts.
5. `org_id` for queries must come from JWT claims, never from request body.
6. Soft-deleted rows (`deleted_at IS NULL`) excluded from default queries.

## 11. Platform architecture (three layers)

Aligned with the source proposal and engineering docs:

1. **Layer 1 — Unified data foundation:** Normalise delivery dashboards, quality reports, workforce data, and governance artifacts into a single governed store with KPI semantic layer, data lineage, and auditability.
2. **Layer 2 — Intelligence and analytics layer:** Predictive analytics, trend/anomaly detection, root-cause analysis, forecasting, risk scoring, pattern recognition.
3. **Layer 3 — Conversational AI interaction layer:** Natural-language Q&A, AI-generated summaries, automated reporting, action recommendations, role-based responses, evidence citation.
