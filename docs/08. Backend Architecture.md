## 1. Overview

The backend is a **Python 3.12 FastAPI** application exposing a REST JSON API. It is the sole write path for agent-generated content, operational snapshots, and governance actions (communication approval). It connects to **Supabase-managed PostgreSQL** via **SQLAlchemy 2.x (async)** and the **asyncpg** driver.

Core responsibilities:

1. **Authenticate** requests via Supabase-issued JWTs
2. **Enforce business rules** above RLS (approval workflows, evidence requirements)
3. **Orchestrate Phase‑1 agents** (Delivery Performance, Quality Intelligence, Client Interaction)
4. **Ingest operational data** into append-only snapshot tables
5. **Persist audit trails** (`agent_queries`, evidence links, communications)

Tenant isolation is enforced at **both** the PostgreSQL RLS layer and the application layer — RLS is the backstop, not a substitute for correct handler logic (`05. Tech Stack.md` §10).

## 4. Layered Architecture

```
HTTP Request
     │
     ▼
┌─────────────────────────────────────────┐
│  api/routes/          HTTP handlers      │  ← thin: validate input, call service
├─────────────────────────────────────────┤
│  core/security.py     Auth middleware    │  ← JWT → CurrentUser (id, role, org_id)
├─────────────────────────────────────────┤
│  services/            Business logic     │  ← transactions, orchestration
├─────────────────────────────────────────┤
│  agents/              Agent algorithms   │  ← scoring, NL generation, drift detection
├─────────────────────────────────────────┤
│  db/models/           SQLAlchemy ORM     │  ← mirrors 10. DB Schema.md
├─────────────────────────────────────────┤
│  PostgreSQL + RLS     Supabase           │
└─────────────────────────────────────────┘
```

**Rules:**

- Route handlers do not call LLM APIs directly — they delegate to `services/` or `agents/`
- Agent modules do not import FastAPI request objects
- All multi-table writes use explicit async transaction boundaries

|---|---|---|

### 5.2 Agent modules (`agents/`)

| Agent module | Triggers | Primary outputs |
|---|---|---|
| `delivery_performance.py` | Post-ingestion cron; manual invoke | `risk_alerts`, `bottlenecks`, `delivery_confidence_scores` |

Agents share retrieval logic with the RAG layer (`11. AI & RAG Architecture.md`) but own domain-specific prompt templates and threshold logic.

## 6. Database Interactions

### 6.1 Connection modes

| Mode | Credential | Used for |
|---|---|---|
| **User session** | End-user JWT → Postgres role with RLS | All read APIs; user-initiated mutations allowed by RLS (alert resolve, comm approve) |
| **Service role** | `SUPABASE_SERVICE_ROLE_KEY` | Agent inserts, snapshot ingestion, draft communication creation |

The backend must never expose the service role key to the frontend (`05. Tech Stack.md` §9).

### 6.2 SQLAlchemy models

Models in `backend/app/db/models/` mirror `10. DB Schema.md` exactly:

- One module per table (or grouped by domain)
- UUID primary keys via `gen_random_uuid()`
- Denormalised `org_id` on every tenant-scoped table
- PostgreSQL ENUM types mapped to Python `enum.Enum`
- Append-only tables omit `updated_at` and `deleted_at`

Migrations are authored as Supabase CLI SQL files in `supabase/migrations/`, not Alembic (default per `05. Tech Stack.md` §3 — confirm open question #16).

### 6.3 Transaction patterns

```
async with session.begin():
    # 1. Agent computes result
    # 2. INSERT primary row (e.g. agent_queries)
    # 3. INSERT evidence links
    # 4. INSERT notifications if threshold crossed
    # All succeed or all roll back
```

Evidence links are **mandatory** for agent query answers and communication drafts — a transaction without at least one evidence link must abort for those artefact types.

### 6.4 RLS interaction

RLS policies are defined in migrations (`10. DB Schema.md` per-table notes). Application responsibilities:

| Role | Application must |
|---|---|
| `client` | Never pass another org's id; scope all queries by JWT user's `org_id` |
| `delivery_manager` | Allow approve/reject on own org's communications only |
| `bsg_leadership` | Read-only cross-org; block all mutations except where RLS allows |
| `super_admin` | Full CRUD on `metric_configurations` only; no bypass on client data |

Set PostgreSQL session variable for RLS: `auth.uid()` must match Supabase JWT `sub` claim on user connections.

## 7. Authentication and Authorization

From `05. Tech Stack.md` §4 and `01. Project Summary.md` §6:

| Concern | Mechanism |
|---|---|
| Identity | Supabase Auth (`auth.users`) |
| Application user | `users` table — `id` equals `auth.users.id` |
| Token | JWT (access + refresh) |
| Roles | `app_role` enum: `client`, `delivery_manager`, `bsg_leadership`, `super_admin` |
| Tenant key | `users.org_id` → `organisations.id` |
| Enforcement | PostgreSQL RLS + FastAPI dependency `require_role(...)` |

### 7.1 Middleware flow

```
Authorization: Bearer <jwt>
        │
        ▼
Decode JWT (Supabase JWT secret / JWKS)
        │
        ▼
Load users row → CurrentUser(id, role, org_id, email)
        │
        ▼
Attach to request state
        │
        ▼
Route dependency checks role ∈ allowed_roles
        │
        ▼
DB session uses user JWT for RLS context
```

### 7.2 Authorization matrix (Phase 1)

| Action | client | delivery_manager | bsg_leadership | super_admin |
|---|---|---|---|---|
| Read own org projects | ✓ | ✓ | ✓ (all orgs) | ✓ |
| Read throughput/quality | ✓ (own org) | ✓ | ✓ | ✓ |
| Resolve alerts | — | ✓ | — | ✓ |
| Approve communications | — | ✓ | — | ✓ |
| Submit agent NL query | ✓ (own projects) | ✓ | ✓ (read scope TBD) | ✓ |
| Submit CSAT | ✓ | — | — | — |
| Manage metric_configurations | — | — | — | ✓ |
| Ingest snapshots | — | — | — | service_role only |

## 9. Business Logic Flows

### 9.1 Delivery confidence computation

```
Input:  throughput_snapshots (rolling 7-day), milestones.planned_date,
        projects.daily_target_units
        │
        ▼
Algorithm (delivery_performance agent):
  - Compare actual vs forecast trend
  - Compute score_pct (0–100)
  - Map to milestone_status (on_track | at_risk) using WIP thresholds
        │
        ▼
Output: INSERT delivery_confidence_scores (append-only)
        UPDATE milestones.status where applicable
```

Draft threshold: schedule confidence ≥ 80% = on track (`01. Project Summary.md` §5).

### 9.2 Risk alert generation

```
Input:  latest confidence scores, throughput trend, milestone dates
        │
        ▼
If slippage_probability exceeds tier threshold:
  INSERT risk_alerts (
    alert_type, risk_tier, title, detail,
    slippage_probability, contributing_causes JSONB
  )
  INSERT notifications for assigned DM users
        │
        ▼
Frontend polls GET /alerts
```

`contributing_causes` example structure from schema: `{"absenteeism": 0.60, "rework": 0.25, "review_turnaround": 0.15}`.

### 9.3 Quality drift detection

```
Input:  quality_snapshots for project/team, sequential ISO weeks
        │
        ▼
Compare gold_set_accuracy_pct, iaa_krippendorff_alpha, rework_rate_pct
against draft thresholds and prior-week baseline
        │
        ├── OK → no write
        └── Breach → UPDATE quality_snapshots.has_drift_alert
                     INSERT risk_alerts (alert_type = quality_drift)
                     INSERT notifications
```

### 9.4 Client communication draft

```
Trigger: cron (weekly) or on-demand internal job
        │
        ▼
client_interaction agent:
  1. Retrieve allowed operational rows for project (RAG)
  2. Generate subject + body_draft via LLM
  3. service_role INSERT client_communications (status = draft)
  4. INSERT communication_evidence_links for each cited row
  5. INSERT notifications (communication_pending) for DM
        │
        ▼
DM approves via PATCH endpoints — see §8.5
```

### 9.5 Agent NL query

```
POST /agents/query
        │
        ▼
Validate user role + project scope
        │
        ▼
Retrieve evidence rows (scoped SELECT)
        │
        ▼
LLM call with grounded context only
        │
        ▼
If no evidence → return 422 / structured error (no hallucinated answer)
        │
        ▼
Transaction: INSERT agent_queries + agent_query_evidence_links
        │
        ▼
Return AgentQueryResponse (answer_text, citations[], latency_ms)
```

## 12. Error Handling

### 12.1 HTTP exception mapping

| Condition | Status | Response shape |
|---|---|---|
| Unauthenticated | 401 | `{ "detail": "Not authenticated" }` |
| Wrong role | 403 | `{ "detail": "Insufficient permissions" }` |
| Resource not found or RLS hides row | 404 | `{ "detail": "Not found" }` — no existence leak |
| Pydantic validation failure | 422 | FastAPI default validation errors |
| LLM timeout / provider error | 503 | `{ "detail": "Agent temporarily unavailable" }` |
| Missing evidence for agent answer | 422 | `{ "detail": "Insufficient evidence to answer" }` |
| Governance violation (send without approve) | 409 | `{ "detail": "Communication not approved" }` |

### 12.2 Logging

- Structured JSON logs (request id, user id, org id, route, latency)
- Never log JWTs, service role keys, or draft communication bodies in prod info logs
- Agent queries log `model_used` and `latency_ms` to `agent_queries` table (audit)

### 12.3 Idempotency

Ingestion endpoints should accept idempotency keys or rely on UNIQUE constraints (`throughput_snapshots` project+date) to prevent duplicate snapshots on retry.

## 17. Phase 1 Boundaries

**In scope:**

- REST API for projects, throughput, quality, alerts, communications, agent queries, notifications, admin metrics
- Three agent modules with evidence-backed outputs
- Ingestion services for snapshots
- JWT auth + RLS-scoped data access
- Human-in-the-loop communication approval enforcement

**Out of scope:**

- Workforce & Capability Agent APIs
- Project Governance Agent
- Operational Knowledge Agent / SOP search
- Leadership margin/analytics aggregations requiring Phase 2 data
- Autonomous multi-agent orchestration
